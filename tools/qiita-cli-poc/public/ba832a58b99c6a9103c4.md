> llove (TUI dashboard) ↔ llmesh (オンプレ MCP ハブ) ↔ llive (自己進化型記憶 LLM) を、最小侵襲で繋ぐ設計と実装。

## TL;DR

- 3 つの独立 Python プロジェクト (llove / llmesh / llive) を、**llmesh の既存 `TimelineStore` を経由する** だけの最小侵襲で連携させた。
- llove は llive を **import しない**。単方向依存 (llove → llmesh のみ) と defensive parser で、llive の breaking change を吸収。
- 3 viewer (BWTDashboard / RouteTraceViewer / MemoryLinkVizPanel) と dispatch helper / TimelinePollDriver を **mock 駆動先行** で完成させ、片方の遅延が他リポジトリをブロックしないようにした。
- v0.3.0a1 (llove) / v3.1.0 (llmesh) / v0.5.0 (llive) で **Phase 0-g 完了**。残るは Phase h (E2E 実機検証)。

---

## なぜ 3 つに分けたのか

最初は単一リポジトリでも書けます。けれど次の 3 つは、それぞれ独立した OSS として配布されるべきでした。

| プロダクト | 役割 | 配布形態 |
|---|---|---|
| **llmesh** | オンプレ MCP ハブ + プロトコル統合 | `pip install llmesh-mcp` |
| **llive** | 自己進化型モジュラー記憶 LLM | `pip install llmesh-llive` |
| **llove** | TUI dashboard / Artifact 端末 | `pip install llmesh-llove` |

理由:

- **使う人が違う** — 運用 SRE / データサイエンティスト / 監査担当
- **更新サイクルが違う** — llmesh は安定、llive は研究色、llove は UX 追加が頻繁
- **依存関係を最小にしたい** — llove だけ入れたい現場、llive だけ実験したい研究

## 採用設計: B 案 (既存 router 流用 + ingest 1 endpoint だけ追加)

```
[llive process]  ─POST /timeline/ingest─►  [llmesh MCP server]  ◄─GET /timeline/recent─  [llove TUI]
                                              │ TimelineStore (既存)
                                              └─ event_type で分離
                                                 (bwt_summary / route_trace / concept_update)
```

**鍵は llmesh の既存 `TimelineStore.record(task_id, node_id, event_type, **metadata)` という 5 つ組が、llive 3 種データに完全フィットしていたこと。** 新 router を作らず、ingest endpoint を 1 つ追加するだけ。

`node_id="llive-instance-1"` で複数ソースを分離できる (将来 MQTT / SPC 等も同じ pipeline で扱える)。

## 実装の鍵

### 1. llove は llive を import しない (単方向依存)

依存方向は **llove → llmesh** のみ。llive の dataclass は `from_event(TimelineEvent)` で **llove 側が defensive にパース**します。

```python
# llove/views/llive/route_trace_viewer.py (抜粋)
@classmethod
def from_event(cls, event: TimelineEvent) -> "RouteTrace | None":
    md = event.metadata or {}
    try:
        subblocks = [SubBlock.from_dict(d) for d in md.get("subblocks", [])
                     if isinstance(d, dict)]
    except Exception:
        return None  # 防御的: 不正フィールドは丸ごとスキップ
    ...
```

これで:
- **llive の breaking change は llove のテストを壊さない**
- llmesh だけ更新しても画面は出続ける
- mock 駆動でフロントが先に書ける

### 2. mock 駆動先行 (Phase a-e は接続なしで完結)

llmesh ingest endpoint や llive bridge writer が無いまま、Phase a-e (viewer 全実装 + dispatch + driver) を完成させた。

```python
def make_mock_route_trace_events(n: int = 3) -> list[TimelineEvent]:
    """テスト + デモ用 fixture. llmesh / llive 不在でも viewer が動く."""
    ...
```

これで llove 開発が他 2 リポジトリの遅延でブロックされない。

### 3. TimelinePollDriver は時間軸を持たない

```python
class TimelinePollDriver:
    """fetch_recent → dispatch_events の薄い周回器. polling 周期は持たない."""
    def poll_once(self) -> DispatchResult: ...
    def status_line(self) -> str: ...
```

Textual `Timer` から `poll_once()` を呼ぶ前提。これにより:

- CLI からも sync に動作確認可能
- SSE / WebSocket への切替がローカル変更で済む
- テストで時計を mock する必要がない

### 4. viewer 別のリテンション戦略

| viewer | 戦略 | 理由 |
|---|---|---|
| BWTDashboard / RouteTraceViewer | event_id 単位で全件保持 | 時系列データ |
| MemoryLinkVizPanel | concept_id 単位で latest 保持 | スナップショット意味論 |

「BWT は時間で見るもの」「memory link は最新だけが意味を持つ」という意味論を viewer 側に固定化。同じ pipeline でも viewer ごとに自然な動きになる。

### 5. llive 側の LoveBridge は依存ゼロ

```python
class LoveBridge:
    """3 emitter (bwt_summary / route_trace / concept_update).
    JSONL 出力 + optional MCP push の 2 経路。"""
    def emit_bwt_summary(self, **kw): ...
    def emit_route_trace(self, **kw): ...
    def emit_concept_update(self, **kw): ...
```

依存は **stdlib `urllib.request` のみ**。httpx は llive の extras に追い出せる。MCP push 失敗は **fail-closed** (JSONL は成功扱い、HTTP は warn のみ、UI には伝播しない)。

### 6. task_id は UUID v4 必須

llmesh ingest endpoint の検証に合わせ fail-fast。caller 未指定なら `uuid.uuid4()` で生成。これで「どの llive run の event か」が必ず分かる。

## 設計判断の振り返り

### A 案 (新 router 追加) を捨てた理由

最初は llmesh に新 router (`/llive/...`) を作る案を検討しました。けれど:

- llmesh 側の認証 / mTLS / rate limiter / metadata size 制限を **すべて再実装** する必要があった
- 既存 TimelineStore の 5 つ組がそのまま llive 3 種データにフィットしていた

→ 新 router を作らず、ingest endpoint だけ追加する **B 案** を採用。30 行程度で済んだ。

### 「llove は llive を import しない」を最後まで維持した理由

import すると、

- llive の dataclass 変更が llove の CI を壊す
- llive の依存 (torch / faiss / cryptography) が llove の最小依存を壊す
- llove だけ入れたい現場が、llive を不要に依存することになる

defensive parser で受けることで、**llove は llmesh の TimelineEvent (中立データ) しか知らない** 状態を保てる。

### Phase h (E2E) を最後に置いた理由

実機検証は sandbox で出来ない (Textual TUI / 本物の MCP server / 本物の llive モデル) ので、ここだけは人間の手が要る。けれど Phase a-g がすべて mock 駆動で完成しているので、E2E はほぼ「それぞれを起動して、JSON が流れることを目で確認する」だけになる予定。

---

## まとめ

- 3 リポジトリ連携で大事なのは、**依存方向を単方向に固定** すること、**defensive parser** で受けること、**mock 駆動先行** で他リポジトリの遅延を待たないこと。
- 既存 endpoint がぴったりハマるなら、新しい protocol surface を作らず流用するのが最小侵襲。
- 「viewer ごとに違うリテンション意味論」を持たせると、同じ pipeline でも自然な UI になる。
