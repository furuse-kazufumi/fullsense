## 1 分 version

2026 年の AI 業界では、ただ「うまいプロンプトを書く」だけでは足りなくなってきました。

- **harness engineering**
  AI を囲う外側の器を設計する
- **loop engineering**
  AI が自分で状況を見て、判断しながら 1 周ずつ進む仕組みを設計する

この記事で言いたいのは 1 つです。

> **業界が 2026 年に名前を付けたものを、私はもうローカルで実物として回し始めていた。**

しかも「AI に全部任せる」のではありません。

- 人間が手綱を握る
- AI は器の中で動く
- 危ないときは止まる
- 背景には自前で育てた知識基盤がある

この 4 点をまとめて見せるのが、今回の #43 です。

![prompt → context → harness → loop の階段（2025→2026）](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/paradigm_staircase.svg)

## 1. たとえ話 1: harness は「暴れ馬の手綱」

LLM は賢いです。でも賢いからこそ、雑に放すともっともらしく暴走します。

そこで要るのが **harness** です。日本語なら「手綱」や「馬具」に近いです。

AI に直接なんでもさせるのでなく、

- どの道具を使えるか
- どこまで権限を渡すか
- 結果をどの形で返させるか
- 危険な操作をどう止めるか

を **外側の器の側で先に決める**。これが harness engineering です。

完全版で取り上げた `RAPTOR` は、まさにこの実物です。

- 重い処理や実行は Python 層が持つ
- LLM は判断に集中する
- schema 検証や governance で危険操作を止める

要するに、

> **AI に剣を渡す前に、鞘とルールと見張りを先に作る**

という発想です。

## 2. たとえ話 2: loop は「レシピ」ではなく「毎周回の見回り」

ここが次の段です。

自動化と loop は似て見えますが、同じではありません。

- **自動化**
  決めた手順を順番に流す
- **loop**
  1 周ごとに「今どこまで進んだ?」「まだゴールじゃない?」「危なくない?」を見直す

料理で言えば、

- 自動化は「卵を割る → 混ぜる → 焼く」の固定レシピ
- loop は「焦げてない?」「そもそも卵は使える?」を毎回見ながら進む調理

です。

完全版で触れた `llloop` は、この「見回りつきの輪」をローカルで試すための自作環境です。

- Monitor
- Analyze
- Plan
- Execute
- Knowledge

という **MAPE-K** 型の骨組みで回し、危険な操作は fail-closed で止める。

ここで大事なのは、

> **AI が考えることと、AI が勝手に実行できることは別にしている**

点です。提案はできても、最後のゲートは安全側に倒す。これが loop engineering を「ただの自動化」と分ける肝です。

## 3. たとえ話 3: 知識基盤は「その場検索」より「自前の図書館」

3 つ目の部品が、`RAD` コーパスと `LLM Wiki` です。

これは平たく言えば、AI に毎回その場で Web 検索させるのでなく、

- 先に調べて貯めたノート群を持つ
- それを整理した概念ページを持つ
- 必要に応じてそこから引く

という構えです。

たとえるなら、

- **普通の RAG**
  用があるたびに図書館へ走る
- **LLM Wiki**
  よく使う知識を先に自分の本棚へ整理しておく

の違いです。

この知識基盤があると、一人開発でも

- 制御理論の視点
- セキュリティの視点
- エージェント設計の視点

を重ねやすくなります。つまり「1 人なのに多視点」になりやすい。

### この回でいちばん大事な態度: 使いたい数字を捨てる

完全版は、わざと少し後ろ向きな話から始まります。

私は「これを入れたら記事が一気に映える」という数字を見つけました。でも一次情報を確認すると、その数字は根拠不明でした。だから捨てました。

この態度は地味ですが重要です。

> **AI 時代ほど、派手な数字を盛るより、怪しい数字を捨てるほうが価値になる。**

なぜなら、AI は流暢に補完できてしまうからです。人間の役目は、流れを良くすることより先に、「そこは未確認」と線を引くことです。

## 4. 覚えて帰るポイントは 3 つ

1. **harness engineering** は、AI を安全に使うための「外側の器」の設計
2. **loop engineering** は、AI を固定レシピではなく「毎周回で判断する輪」として設計すること
3. その 2 つを本当に使い物にするには、背景で支える **知識基盤** が要る

そして私の立場は、ここにもう 1 本補助線を引くものです。

> **主役は AI ではなく、手綱を握る人間である。**

AI は便利な部下になれる。でも、器を作るのも、止めるのも、根拠を確かめるのも、人間の仕事です。

## まとめ

2026 年、業界は AI 設計の次の段に名前を付けました。

- prompt engineering
- context engineering
- harness engineering
- loop engineering

私はこのうち後ろ 2 つを、ローカルの実物として組み始めています。

- `RAPTOR`
  手綱つきの実行器

  安全装置つきの輪
- `RAD + LLM Wiki`
  育てる知識基盤
