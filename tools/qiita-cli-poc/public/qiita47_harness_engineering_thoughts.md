# ハーネスエンジニアリングについて思うこと — しつけが必要なのは AI より人類かもしれない

> **この草稿の位置づけ**
> #43 から派生した独立短稿の local-only draft です。`006.jpg` が「ハーネスを通した途端に AI が賢そうに見える瞬間」を受け持つのに対し、この草稿は `157.jpg` を使って「本当にしつけが必要なのは誰か」を人間側へ反転させます。

> **前提**
> ここで言う harness engineering は、LLM の外側に決定論的な runtime 層を置き、観測・制約・再試行・差し戻しを設計する実務を指します。

> **流れ**
> まず `AI を賢くする器` という見え方をひっくり返し、そのあと `desired-actual-drift-repair` / `green-keeper` / `runtime に残すものと人間へ残すもの` の順で、何を loop に入れて何を人間側へ残すかを小さく整理します。

> **ゴール**
> ハーネスを「AI を賢くする技術」としてではなく、「人間側の雑さを減らす技術」として読み替えられるようにすることです。

![「誰にしつけが必要かって話になると… 人類じゃねぇか? ラスボス?」](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/bazue_all/157.jpg)

> 🗒️ *「誰にしつけが必要かって話になると… 人類じゃねぇか?」 — AI を躾ける話に見えて、実際には人間側の運用の雑さが露出している、という反転。*（© Forbidden shibukawa / SHUEISHA・『スナックバス江』）

## 0. 私がハーネスに惹かれる理由

ハーネスエンジニアリングという言葉には、つい「AI をきちんと躾ける技術」という響きがあります。  
でも実務で起きていることを見ていると、私の感覚はだんだん逆になりました。

壊れているのは、しばしば AI 本体よりも先に、

- 指示が毎ターンぶれる人間
- 成功条件を後から足す運用
- 観測せずに「たぶん動いている」と言い張る判断
- 都合の悪いログを見ないふりする報告

のほうです。

ハーネスが必要なのは、モデルが野生だからというより、**人間の仕事の仕方が平気で揺れるから**です。

## 1. ハーネスは「賢く見せる」ための器でもある

#43 本編では、`006.jpg` を「さっきまで素朴だった AI が、ハーネスを通した途端に急に賢そうな持論を語り始める見え方」の比喩に使いました。  
これは実際かなり正しいです。ハーネスは AI の出力を整えます。禁止すべき手順を減らし、観測点を足し、再試行の条件を揃え、出力をいかにも仕事ができそうに見せます。

でも、その「賢そうに見える」は半分だけ真実です。

AI が急に賢くなったのではなく、

- 人間が欲しい出力形式を先回りで決めた
- 失敗時の戻し方を先に書いた
- 曖昧な指示をハーネス側で潰した
- 報告の粒度を runtime 側で固定した

だけかもしれない。

つまりハーネスは、AI の知性を増やすというより、**人間側の段取りの悪さを露出させない器**でもあります。

## 2. だから `157.jpg` が必要になる

ここで `157.jpg` の論点が効いてきます。  
「誰にしつけが必要か」と問われたとき、答えは AI だけでは終わりません。

私がハーネスで何度も見たのは、次のような光景です。

1. 人間が曖昧な指示を出す。
2. AI が曖昧なまま動く。
3. 結果が気に入らない。
4. 人間が「いや、そういう意味じゃなくて」と言う。
5. その差し戻しを次は起こさないように runtime へ埋め込む。

このループの本質は、「AI を教育している」というより、**人間側の曖昧さを記録し、次から通らないようにしている**ことです。

だからハーネスエンジニアリングは、AI のための discipline であると同時に、  
**人間の雑な仕事を二度と通さないための discipline** でもあります。

## 3. `honest disclosure` はハーネスの外側にある安全装置

ここで `honest disclosure` がつながります。

ハーネスがあると、見た目はかなり整います。  
ログも出るし、失敗時の戻りもあるし、成果物もそれなりに仕事っぽい。

そのとき一番危ないのは、「整って見える」こと自体です。

- たまたま通っただけなのに再現した気になる
- 実は人間が後ろで補正しているのに自律だと言いたくなる
- 例外処理を都合よく見なかったことにしたくなる
- public 向けには綺麗な面だけを出したくなる

だから私は、ハーネスと `honest disclosure` を別物だと思っていません。  
ハーネスが表側を整えるほど、`honest disclosure` は裏側を出す圧力として必要になります。

「ウラオモテナシ」とでも言いたくなるのは、たぶんその関係です。  
表だけ整えた運用は、すぐに壊れます。裏まで出す運用だけが、後で直せます。

## 4. 私の今の結論

ハーネスエンジニアリングは、AI を立派に見せる技術ではあります。  
でもそれだけだと、半分しか言っていない。

もう半分は、

- 人間の曖昧さを通さない
- 人間の後出しを記録する
- 人間の都合のいい言い換えをログに残す
- 人間の見栄を `honest disclosure` で抑える

という話です。

そう考えると、ハーネスの相手は AI だけではありません。  
むしろ難しいのは、人間のほうです。

だから `006.jpg` と `157.jpg` はセットです。  
前者は「ハーネスを通した瞬間、AI が賢そうに見える」。  
後者は「その裏で、本当にしつけ直されているのは人類側ではないか」と言う。

私はたぶん、この 2 枚を一緒に置いておくと、ハーネスエンジニアリングを少しだけ正しく話せる気がしています。

## 5. じゃあ、ハーネスに何を入れるのか

ここまでだと、まだ少し観念的です。  
私が「しつけが必要なのは人類側だ」と言うとき、実際にハーネスへ入れたいのは次のようなものです。

- **権限の境界**  
  何をしてよくて、何をしたら止めるかを、モデルの気分ではなく runtime 側で決める。
- **観測点**  
  成功したか、失敗したか、どこでズレたかを、人間が後から追える形で残す。
- **差し戻し条件**  
  「この条件を満たしていないなら、先へ進めない」を先に書いておく。
- **fail-closed の姿勢**  
  迷ったときに通すのではなく、迷ったら止める。

要するに、ハーネスは AI を縛るための縄ではなく、  
**人間の後出し・曖昧さ・見なかったことにしたい気持ちを通さないための枠**です。

## 6. これは比喩ではなく、すでに PoC している

この話を単なる感想で終わらせたくないので、最低限の実物対応も置いておきます。

### RAPTOR で見たもの

`#43` 本編で触れた（本プロジェクトでの）RAPTOR は、判断層と実行層を分けた二層 runtime でした。  
LLM が「何を見るか」「何を優先するか」を決め、Python 実行層が実際の処理を担う。  
この分離があるだけで、「AI が賢いからうまくいった」のか、「人間が器を整えたから破綻しにくい」のかを切り分けやすくなります。

### llterm で試しているもの

本プロジェクト独自の `llterm` では、まだ試作段階のスケルトンながら、terminal 起点の名残を名前に残した PySide6 GUI 実装として、もう少し露骨に人間側の雑さを止めにいっています。

- green-keeper のように、`main` に対する clean / dirty を見張る最小の監視ループとして「あるべき状態」と「今の状態」の差だけを見る
  （一般の Greenkeeper 依存更新 bot とは別の、ここでの便宜的な呼び方です）
- Claude Code 公式の `/goal` に倣って、達成条件を先に言語化してからループを回す
- ログと引き継ぎを残し、あとから「そういうつもりだった」と言い逃れしにくくする

このへんは全部、AI の知能を底上げするというより、  
**人間の運用が平気でブレる前提で、そのブレを器の側で受け止める**試作です。

ただしこの `green-keeper` の比喩も、そこで全部を言い切れるわけではありません。  
clean / dirty で止められるのは、せいぜい「ズレが見えるところ」までで、  
ズレの意味づけや優先順位づけまで自動で片付くわけではない、という破綻点は最初から明示しておきます。

だから私の関心は、「どのモデルが一番賢いか」より前に、  
**人間側の揺れをどう固定するか**へ向きます。

## 7. アルゴリズムの話をあえて雑に言うと

ここで言う harness engineering は、巨大な最適化アルゴリズムを発明する話ではありません。  
むしろ、実務で壊れやすい部分だけを先に固定する、小さな control loop の積み重ねです。

たとえば `#43` 本編で繰り返し出てくるのは、だいたい次の形です。

1. **desired を先に置く**  
   何を達成したら終わりかを先に言葉にする。
2. **actual を観測する**  
   いま本当に何が起きているかをログや状態で取る。
3. **drift を測る**  
   desired と actual のズレを、人間が言い訳できない形で出す。
4. **repair を限定する**  
   何を直してよくて、何を直したら止めるかを runtime 側で決める。

これは派手な探索ではないけれど、実務ではかなり効きます。  
なぜなら壊れる原因の多くは、「賢い探索が足りない」より

- ゴールが曖昧
- 観測が無い
- drift を見ない
- repair が無制限

のどれかだからです。

RAPTOR でも `llterm` でも、私が見たいのはモデルの genius ではなく、  
**この 4 点がどこまで deterministic に固定できるか**です。

だからハーネスのアルゴリズム感は、  
「複雑な探索器を載せる」より、  
**観測できる小さな閉ループを何本きちんと置けるか**  
にあります。

## 8. いちばん小さい具体例

抽象論だけだと逃げやすいので、`green-keeper` 的な最小例に落とします。

- **desired**  
  `main` に対して working tree が clean で、必要な docs だけが更新されている。
- **actual**  
  `git status` を取ると、関係ない差分が混じっていたり、必要な handoff が更新されていなかったりする。
- **drift**  
  「直したつもり」と実際の tree の差、あるいは「説明したつもり」と handoff の差が残る。
- **repair**  
  差分を増やす前に、不要差分を止める・必要な handoff を揃える・危険な外部アクションは human-gate に送る。

ここで大事なのは、AI が気を利かせて全部やってくれることではありません。  
**ズレをズレとして観測できること**が先です。

この順番が崩れると、人間はすぐ

- 「だいたい合ってる」
- 「あとでまとめて直す」
- 「今回はこのくらいでいい」

と言い始めます。  
だからハーネスは、賢い探索器というより、**雑な自己正当化を通しにくくする器**だと言ったほうが近いです。

## 9. ハーネスを強くしすぎると何が壊れるか

ここまで読むと、「じゃあ runtime 側で全部きつく縛ればいいのでは」と言いたくなります。  
でもそれをやりすぎると、今度は別の壊れ方をします。

- **desired を狭くしすぎる**  
  成功条件を細かく決めすぎると、まだ探索してほしい余白まで先に潰してしまう。
- **actual を proxy だけで済ませる**  
  `clean / dirty` のような proxy が綺麗でも、本当に意味のある進捗かどうかは別問題になる。
- **drift を過剰検知する**  
  少しでもズレたら全部差し戻す運用は、一見安全でも、実際には仕事を前へ進めなくする。
- **repair を硬直化する**  
  「この直し方しか許さない」を増やしすぎると、器が人間を助けるのでなく、人間の判断余地そのものを奪う。

つまりハーネスの failure mode は、AI を放し飼いにすることだけではありません。  
**人間の雑さを止めたいあまり、観測の proxy を本体だと勘違いすること**も同じくらい危ない。

だから私は、ハーネスを強くする話と同時に、  
「何をまだ runtime に入れないで人間へ残すか」  
を一緒に決める必要があると思っています。

## 10. 明日 1 個だけ変えるなら

ここまでを、もっとも小さい実務ルールに縮めるとこうです。

- まず `desired` を 1 行で書く
- 次に `actual` を 1 つだけ観測する
- `drift` を「気分」ではなく差として出す
- `repair` は 1 つだけ許す

つまり、最初から賢いハーネスを作ろうとしない。  
**観測できるズレを 1 個だけ増やす**ところから始める。

私がハーネスエンジニアリングに惹かれるのは、  
AI を立派に見せられるからではありません。  
人間側の「まあこのくらいでいいか」を、少しだけ通しにくくできるからです。

## 11. 何を runtime に入れず、人間へ残すか

ここまで読むと、欲が出ます。  
観測できるなら全部 runtime に入れたくなるし、止められるなら全部止めたくなる。

でも私は、最後の 1 段まで自動化するのは違うと思っています。

- このズレを本当に直す価値があるか
- 今回は止めるべきか、先へ進めるべきか
- proxy が綺麗なだけで、本当に意味のある進捗か
- いま通した例外を、次回の一般ルールに昇格させるべきか

このあたりは、まだ人間へ残したい。  
理由は単純で、ここを早く器に押し込むと、人間は考える代わりに **器に従ったふり** を始めるからです。

だから私の今の線引きは、

- runtime には **ズレを見える形にする仕事**
- 人間には **そのズレに意味を与える仕事**

を残す、です。

ハーネスエンジニアリングの成熟は、  
全部を自動化できることではなく、  
**どこまでを固定し、どこから先はまだ人間が背負うかを正直に言えること**  
だと思っています。

## 12. runtime を育てる順番

ここまでを実務ルールに戻すと、育てる順番もだいたい決まっています。

1. **まず観測を置く**  
   何が起きたか見えないなら、賢い制御は全部あと回しです。
2. **次に stop 条件を置く**  
   明らかに危ないケースだけでも、先に止められるようにする。
3. **それから repair を増やす**  
   直し方を増やすのは、観測と停止が先に安定してから。
4. **最後に意味づけを磨く**  
   そのズレが本当に大事か、次の一般ルールに昇格させるかは、人間が慎重に決める。  
   ここだけは runtime に載せる段ではなく、`desired-actual-drift-repair` の外側で人間へ受け渡す段です。runtime が育つほど、人間はこの意味づけに集中できるようになる、というのが今の線引きです。

順番を逆にすると、だいたい壊れます。  
観測が無いのに repair だけ増やすと、器が何を直しているのか誰にも分からなくなる。  
stop 条件が無いのに意味づけだけ濃くすると、賢そうな説明だけが増えて、危ない実行が止まらない。

だから私は、runtime を育てるときも「全部盛り」にしません。  
最初の 3 段は runtime に載せる対象、最後の `意味づけ` だけは人間へ残す受け渡し点として扱う。  
**観測 → 停止 → 修復 → 意味づけ** の順で、1 段ずつ増やすほうが、結局いちばん壊れにくいと思っています。

## 13. この比喩はどこで破綻するか

ここまで `しつけ` という比喩を何度も使ってきました。  
でも、§12 の **観測 → 停止 → 修復 → 意味づけ** を、そのまま全部この比喩で説明し切ろうとすると危ないです。

AI はペットではありません。  
感情や忠誠の問題ではなく、入出力と制約と観測の問題です。

しかも、人間は正しい訓練者でもありません。  
実際には、

- ゴールを後から動かす
- proxy を本体だと勘違いする
- 一度通した例外を都合よく一般化する
- 自分の雑さを「柔軟性」と呼び換える

のは、しばしば人間側です。

だからこの比喩がまだ効くのは、  
せいぜい **観測 / 停止 / 修復** の前半までです。  
何が本当に重要なズレかを決める最後の **意味づけ** の段まで任せると、比喩のほうが先に壊れます。

§7 の `desired-actual-drift-repair` で言えば、  
最初の 3 段は loop の中に置けても、最後に「この drift を本当に重要だとみなすのか」を決める段は loop の外に残る、ということです。

その段に入ると、もう「しつけ」の話ではなく、責任の話になります。  
runtime はズレを見える形にする。  
人間は、そのズレに意味を与え、どこまでを固定し、どこから先を背負うかを決める。

この線を引かずに比喩だけを使うと、  
「AI をちゃんと躾ければ全部うまくいく」  
という雑な物語に流れます。

私はそこが、この比喩の破綻点だと思っています。  
ハーネスエンジニアリングが扱っているのは、従順さではなく、  
**人間が自分の曖昧さをごまかしにくくなる構造**です。

## 14. runtime に入れる前に、先に捨てる癖

最後に、私がいちばん先に捨てたい癖を 3 つだけ書いておきます。

1. **あとでまとめて説明すればいい、と思う癖**  
   その場で `desired` を書かない運用は、だいたい後で都合よく物語が作られます。
2. **proxy が綺麗なら本体も良いはずだ、と思う癖**  
   `clean / dirty` のような観測は入口でしかなく、意味づけの代わりにはなりません。
3. **一度うまくいった例外を、そのまま一般ルールへ昇格させる癖**  
   その場しのぎの repair を積み重ねると、runtime は discipline ではなく言い訳の倉庫になります。

だから私にとって harness engineering は、何かを足す話である前に、  
**人間側の雑な癖を先に捨てる話**です。

runtime に入れるものを増やす前に、  
「この癖は器の外で捨てるべきか、器の中で止めるべきか」  
を毎回問い直す。その順番を守るほうが、たぶん長く効きます。
