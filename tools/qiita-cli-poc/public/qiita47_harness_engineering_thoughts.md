# ハーネスエンジニアリングについて思うこと — しつけが必要なのは AI より人類かもしれない

> **この草稿の位置づけ**
> #43 から派生した独立短稿の local-only draft です。`006.jpg` が「ハーネスを通した途端に AI が賢そうに見える瞬間」を受け持つのに対し、この草稿は `157.jpg` を使って「本当にしつけが必要なのは誰か」を人間側へ反転させます。

> **前提**
> ここで言う harness engineering は、LLM の外側に決定論的な runtime 層を置き、観測・制約・再試行・差し戻しを設計する実務を指します。

> **ゴール**
> ハーネスを「AI を賢くする技術」としてではなく、「人間側の雑さを減らす技術」として読み替えられるようにすることです。

![「誰にしつけが必要かって話になると… 人類じゃねぇか? ラスボス?」](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/bazue_all/157.jpg)

> 🗒️ *「誰にしつけが必要かって話になると… 人類じゃねぇか?」 — AI を躾ける話に見えて、実際には人間側の運用の雑さが露出している、という反転。*（© Forbidden shibukawa / SHUEISHA・『スナックバス江』）

## 0. 私がハーネスに惹かれる理由

ハーネスエンジニアリングという言葉には、つい「AI をきちんと躾ける技術」という響きがあります。  
でも実務で起きていることを見ていると、私の感覚はだんだん逆になりました。

壊れているのは、しばしば AI 本体よりも先に、

- 指示が毎ターンぶれる人間
- 成功条件を後から足す運用
- 観測せずに「たぶん動いている」と言い張る判断
- 都合の悪いログを見ないふりする報告

のほうです。

ハーネスが必要なのは、モデルが野生だからというより、**人間の仕事の仕方が平気で揺れるから**です。

## 1. ハーネスは「賢く見せる」ための器でもある

#43 本編では、`006.jpg` を「さっきまで素朴だった AI が、ハーネスを通した途端に急に賢そうな持論を語り始める見え方」の比喩に使いました。  
これは実際かなり正しいです。ハーネスは AI の出力を整えます。禁止すべき手順を減らし、観測点を足し、再試行の条件を揃え、出力をいかにも仕事ができそうに見せます。

でも、その「賢そうに見える」は半分だけ真実です。

AI が急に賢くなったのではなく、

- 人間が欲しい出力形式を先回りで決めた
- 失敗時の戻し方を先に書いた
- 曖昧な指示をハーネス側で潰した
- 報告の粒度を runtime 側で固定した

だけかもしれない。

つまりハーネスは、AI の知性を増やすというより、**人間側の段取りの悪さを露出させない器**でもあります。

## 2. だから `157.jpg` が必要になる

ここで `157.jpg` の論点が効いてきます。  
「誰にしつけが必要か」と問われたとき、答えは AI だけでは終わりません。

私がハーネスで何度も見たのは、次のような光景です。

1. 人間が曖昧な指示を出す。
2. AI が曖昧なまま動く。
3. 結果が気に入らない。
4. 人間が「いや、そういう意味じゃなくて」と言う。
5. その差し戻しを次は起こさないように runtime へ埋め込む。

このループの本質は、「AI を教育している」というより、**人間側の曖昧さを記録し、次から通らないようにしている**ことです。

だからハーネスエンジニアリングは、AI のための discipline であると同時に、  
**人間の雑な仕事を二度と通さないための discipline** でもあります。

## 3. `honest disclosure` はハーネスの外側にある安全装置

ここで `honest disclosure` がつながります。

ハーネスがあると、見た目はかなり整います。  
ログも出るし、失敗時の戻りもあるし、成果物もそれなりに仕事っぽい。

そのとき一番危ないのは、「整って見える」こと自体です。

- たまたま通っただけなのに再現した気になる
- 実は人間が後ろで補正しているのに自律だと言いたくなる
- 例外処理を都合よく見なかったことにしたくなる
- public 向けには綺麗な面だけを出したくなる

だから私は、ハーネスと `honest disclosure` を別物だと思っていません。  
ハーネスが表側を整えるほど、`honest disclosure` は裏側を出す圧力として必要になります。

「ウラオモテナシ」とでも言いたくなるのは、たぶんその関係です。  
表だけ整えた運用は、すぐに壊れます。裏まで出す運用だけが、後で直せます。

## 4. 私の今の結論

ハーネスエンジニアリングは、AI を立派に見せる技術ではあります。  
でもそれだけだと、半分しか言っていない。

もう半分は、

- 人間の曖昧さを通さない
- 人間の後出しを記録する
- 人間の都合のいい言い換えをログに残す
- 人間の見栄を `honest disclosure` で抑える

という話です。

そう考えると、ハーネスの相手は AI だけではありません。  
むしろ難しいのは、人間のほうです。

だから `006.jpg` と `157.jpg` はセットです。  
前者は「ハーネスを通した瞬間、AI が賢そうに見える」。  
後者は「その裏で、本当にしつけ直されているのは人類側ではないか」と言う。

私はたぶん、この 2 枚を一緒に置いておくと、ハーネスエンジニアリングを少しだけ正しく話せる気がしています。

## 5. じゃあ、ハーネスに何を入れるのか

ここまでだと、まだ少し観念的です。  
私が「しつけが必要なのは人類側だ」と言うとき、実際にハーネスへ入れたいのは次のようなものです。

- **権限の境界**  
  何をしてよくて、何をしたら止めるかを、モデルの気分ではなく runtime 側で決める。
- **観測点**  
  成功したか、失敗したか、どこでズレたかを、人間が後から追える形で残す。
- **差し戻し条件**  
  「この条件を満たしていないなら、先へ進めない」を先に書いておく。
- **fail-closed の姿勢**  
  迷ったときに通すのではなく、迷ったら止める。

要するに、ハーネスは AI を縛るための縄ではなく、  
**人間の後出し・曖昧さ・見なかったことにしたい気持ちを通さないための枠**です。

## 6. これは比喩ではなく、すでに PoC している

この話を単なる感想で終わらせたくないので、最低限の実物対応も置いておきます。

### RAPTOR で見たもの

`#43` 本編で触れた RAPTOR は、判断と実行を分けた二層構造でした。  
LLM が「何を見るか」「何を優先するか」を決め、Python 実行層が実際の処理を担う。  
この分離があるだけで、「AI が賢いからうまくいった」のか、「人間が器を整えたから破綻しにくい」のかを切り分けやすくなります。

### llterm で試しているもの

自作の `llterm` では、まだ試作段階のスケルトンながら、もう少し露骨に人間側の雑さを止めにいっています。

- green-keeper のように「あるべき状態」と「今の状態」の差だけを見る
- Claude Code 公式の `/goal` に倣って、達成条件を先に言語化してからループを回す
- ログと引き継ぎを残し、あとから「そういうつもりだった」と言い逃れしにくくする

このへんは全部、AI の知能を底上げするというより、  
**人間の運用が平気でブレる前提で、そのブレを器の側で受け止める**試作です。

だから私の関心は、「どのモデルが一番賢いか」より前に、  
**人間側の揺れをどう固定するか**へ向きます。

## 7. アルゴリズムの話をあえて雑に言うと

ここで言う harness engineering は、巨大な最適化アルゴリズムを発明する話ではありません。  
むしろ、実務で壊れやすい部分だけを先に固定する、小さな control loop の積み重ねです。

たとえば `#43` 本編で繰り返し出てくるのは、だいたい次の形です。

1. **desired を先に置く**  
   何を達成したら終わりかを先に言葉にする。
2. **actual を観測する**  
   いま本当に何が起きているかをログや状態で取る。
3. **drift を測る**  
   desired と actual のズレを、人間が言い訳できない形で出す。
4. **repair を限定する**  
   何を直してよくて、何を直したら止めるかを runtime 側で決める。

これは派手な探索ではないけれど、実務ではかなり効きます。  
なぜなら壊れる原因の多くは、「賢い探索が足りない」より

- ゴールが曖昧
- 観測が無い
- drift を見ない
- repair が無制限

のどれかだからです。

RAPTOR でも `llterm` でも、私が見たいのはモデルの genius ではなく、  
**この 4 点がどこまで deterministic に固定できるか**です。

だからハーネスのアルゴリズム感は、  
「複雑な探索器を載せる」より、  
**観測できる小さな閉ループを何本きちんと置けるか**  
にあります。
