### ollama llava:7b (on-prem) — 140 s, 591 chars

> The image shows a screenshot of a webpage with a colorful banner at the
> top featuring text that reads "FullSense" and includes a subtitle
> **"Unlimited Brand Splash on Premium LM Systems."**
> Below this, there is a description stating
> **"FullSense is an AI-powered branding platform that helps you create
> stunning visuals for your website or social media in seconds."**
> The products listed are **"FullSense," "Live," and "Love."**
> These products appear to be related as they all pertain to the
> FullSense branding platform, with "Live" and "Love" being specific
> features or services offered by FullSense.

Problems (in increasing severity):

1. **Subtitle misread** — "Unlimited Brand Splash on Premium LM Systems"
   is not what the card shows. The real subtitle is "Umbrella brand &
   spec for an open-source family targeting self-evolving, on-prem,
   audit-friendly LLM systems."
2. **Hallucinated body text** — There is no "AI-powered branding platform
   that helps you create stunning visuals" anywhere on the card.
3. **`ll*` tokenisation hostility resurfaces** — `llive` becomes "Live"
   and `llove` becomes "Love". Same family of error as the text-Brief
   `lllive` typo from llama3.2:3b — the `ll` prefix is consistently
   dropped or recombined by small Ollama models.
4. **llmesh and llmesh-suite missed entirely.** Out of 4 product chips,
   only 2 were named (and those mangled).

The model didn't fabricate the FullSense name, but it filled in plausible-
sounding branding-platform copy that doesn't exist in the image, which
on an audit-focused product family is a worse failure mode than missing
detail.

### ollama llama3.2-vision:11b (on-prem) — ERR cold-start timeout

`ollama pull llama3.2-vision` completed (7.9 GB) but the first inference
on the og-card did not return within 600 s. Likely cold-start cost of an
11B vision model on this machine. Repro after `ollama run llama3.2-vision`
has been left running once to warm the cache, or after restoring cloud
credentials so the comparison no longer hinges on a single on-prem path.

Tracked as a follow-up benchmark in the next session.

## Implications

- **VLM column is structurally unblocked today only on-prem.** That
  matches the FullSense story (on-prem inference as a primary
  differentiator), but `llava:7b` quality on a 1200×630 product card is
  not good enough for serious lldesign use (3-of-5 products missed +
  hallucinated body copy). A 11B vision model or restored cloud
  credentials are needed before VLM can ship in a v0.x feature.
- **Tokenisation hostility to `ll*` prefixes is now confirmed across two
  Ollama model families** (llama3.2:3b text and llava:7b vision —
  llava's text backbone is Vicuna/LLaMA-2 family). When llive
  v0.2.0 lands its VLM/coding-LLM RAD integration (see
  [`project_llive_v02_rad_integration`](https://github.com/furuse-kazufumi/raptor/blob/main/.claude/projects/C--Users-puruy-raptor/memory/project_llive_v02_rad_integration.md))
  the model selection criteria should include a `ll*` glossary smoke
  test, not just a perplexity score.
- **bench_vlm.py race-condition fix in flight** — initial run had both
  llava and llama3.2-vision invocations resolving to the same model name
  because of process-wide env var sharing across ThreadPoolExecutor
  workers. Fixed in the same session; the panel now correctly
  distinguishes the two ollama models.
