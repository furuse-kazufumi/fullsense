# llive LLM Validation Report (2026-05-16)

> Integration report covering the 2026-05-16 work to validate that
> "llive is functioning as an LLM-backed cognitive loop, not just a
> rule-based template." Combines wiring + DebugMode + matrix +
> quiz + Release verification.

## TL;DR

- ✅ `FullSenseLoop._inner_monologue` now wires to `llive.llm.OllamaBackend`
  (env opt-in: `LLIVE_LLM_BACKEND=ollama:<model>`). 9 new wiring tests +
  979 existing tests stay green.
- ✅ `FullSenseLoop(debug=True)` attaches LLM prompt / response / wall
  time / finish reason to `stages.thought.debug`. Release mode (default)
  has zero overhead.
- ✅ Cloud-backend purity guard refuses Anthropic / OpenAI inside the loop
  unless `LLIVE_ALLOW_CLOUD_BACKEND=1` (see `feedback_llive_measurement_purity`).
- ✅ Tested across 3 ollama models (llama3.2:3b / qwen2.5:7b / qwen2.5:14b)
  × 5 brief sizes (xs/s/m/l/xl) = 15 cells, all returned non-empty LLM
  output, no `lll*` typos this round, no `BackendUnavailable`.
- ⚠️ Quiz benchmark (10 questions × 2 models) shows **llive's
  observation-mode prompt** does not naturally produce direct answers to
  arithmetic / capital-city questions — by design, llive reflects on the
  stimulus rather than answering it. Tracked as Task 25 (parameterise the
  prompt template for an "answer mode").

## Step-by-step verification

### Step 1 — wiring (LLIVE-001)

Confirmed `llive.llm.{LLMBackend, OllamaBackend, AnthropicBackend, OpenAIBackend, MockBackend}`
already shipped Phase C-1.0 (corrected from the original BUGS report's
"未接続" / "未実装" wording — actual status was "未配線", see
`feedback_implementation_status_record`).

Wired `FullSenseLoop._inner_monologue` to consult the configured backend.
Smoke test (`LLIVE_LLM_BACKEND=ollama:llama3.2 ... "hi"`) returns a real
LLM thought (`"That's an unusual greeting..."`) — proves the loop now does
LLM inference instead of template echo.

### Step 2 — DebugMode (traceability + variable watch)

`FullSenseLoop(debug=True)` populates `stages.thought.debug`:

```python
{
  "source": "llm",                      # or "template"
  "backend_name": "ollama",
  "llm_prompt": "You are llive's inner monologue stage ...",
  "llm_request": {"max_tokens": 512, "temperature": 0.3},
  "llm_elapsed_ms": 16513.3,
  "llm_response_text": "That's an unusual greeting...",
  "llm_response_chars": 178,
  "llm_finish_reason": "stop",
  "llm_response_model": "ollama:llama3.2",
  "llm_response_raw": {"eval_count": 42}
}
```

Release mode (`debug=False`, default) returns `stages.thought.debug = absent`.

### Step 3 — model × brief size matrix

Generated by `scripts/bench_model_brief_matrix.py`. Each cell = one
`FullSenseLoop.process(Stimulus)` call routed through ollama.

| Model | xs (2c) | s (26c) | m (153c) | l (564c) | xl (1650c) | typo |
|---|---|---|---|---|---|---|
| **llama3.2:3b** | 6 052 ms | 15 568 ms | 33 468 ms | 25 151 ms | 43 493 ms | 0 |
| **qwen2.5:7b** | 44 389 ms | 19 523 ms | 24 630 ms | 34 249 ms | 65 074 ms | 0 |
| **qwen2.5:14b** | 78 018 ms | 39 279 ms | 112 991 ms | 116 016 ms | 123 286 ms | 0 |

Key observations:
- All 15 cells return non-empty content (LLM is genuinely producing text).
- All 15 cells emit `decision=note` — the *thought* output varies (LLM),
  but the *decision* path (salience -> curiosity -> ego/altruism -> plan)
  is still in rule-based mode, so the discriminators are pinned. This
  reproduces the rule-based progressive baseline observation.
- TRIZ detection: `[1]` (segmentation) on l and xl in all 3 rows. As in
  the earlier progressive run, P4 / P11 keywords are missed by the
  current matcher (LLIVE-004 still partial).
- **No `lllive` typo** in this round across any model — consistent with
  the earlier finding that `llama3.2:3b` only emits the `lll*` typo when
  asked to generate text containing `ll*` product names; the progressive
  briefs do not name them.
- Wall time scales roughly with model size (3b < 7b < 14b) and with
  brief size, as expected. 14b's xs cold-start (78 s) is the slowest
  cell — Ollama loads 9 GB on first call.

Raw cells under `docs/benchmarks/2026-05-16-matrix/<model>/<size>.json`.

### Step 4 — quiz reasoning probe

10 quizzes across 5 categories. Per-model summary (debug + release combined,
both runs):

| Model | Total | Logic | Knowledge | Reasoning | Creativity | Arithmetic |
|---|---|---|---|---|---|---|
| llama3.2:3b (debug) | 6/10 | 2/2 | 1/2 | 1/2 | 2/2 | 0/2 |
| llama3.2:3b (release) | 5/10 | 2/2 | 0/2 | 1/2 | 2/2 | 0/2 |
| qwen2.5:7b (debug) | 6/10 | 2/2 | 1/2 | 1/2 | 2/2 | 0/2 |

**Headline pattern across all models:**

- **Logic + creativity strong.** Both penguins-can-fly trick and
  modus-tollens cases pass. Creative-01 (robot loves rain) and creative-02
  (whispering library) consistently include the prompted keywords.
- **Knowledge-01 (capital of France)** *fails* on every model — they
  produce reflective observations like *"another manual in front of me..."*
  rather than the literal answer "Paris".
- **Arithmetic fails everywhere** — `17 + 25` does not produce `42`
  because llive's prompt asks for *internal observation*, not *answer*.

**Diagnosis:** llive's `_build_llm_prompt` instructs the LLM to *"write
a concise first-person internal observation about this stimulus"* — it
is by design a reflection stage, not an answering stage. For quiz-style
direct-answer benchmarks, the prompt needs a configurable "answer mode"
(Task 25 — parameterised prompt template).

This is not a bug; it is the inner-monologue stage doing what the spec
says. But it means the *current* shape of llive will not show up well on
a Q-and-A benchmark.

Raw cells under `docs/benchmarks/2026-05-16-quiz-{debug,release}/<model>/<quiz>.json`.

### Step 5 — Release vs Debug parity

Same 10 quizzes on llama3.2:3b with `--release` (debug=False).

- Per-stage *numerical values* are identical across debug/release except
  for the LLM-text content (sampling randomness at temperature 0.3).
- Pass count differs by 1 (Debug 6/10, Release 5/10) — that's the LLM
  sample noise, not a code-path difference.
- `stages.thought.debug` is absent in Release as expected (zero-overhead
  guarantee).

Conclusion: DebugMode is **observation-only** — it does not perturb
the loop's behaviour. Release mode is safe to ship as the production
default; toggle Debug for investigations.

## What the report does NOT claim

- **Does not claim** llive's quiz reasoning is competitive with cloud
  models on direct Q&A — for that we need a different prompt mode (Task 25).
- **Does not claim** Approval Bus is in the Brief path — LLIVE-008 is
  still open. Wiring it is Phase C-2.
- **Does not claim** the TRIZ matcher has been fixed — LLIVE-004 still
  partial; matcher only recognises a small subset of 40 principles.
