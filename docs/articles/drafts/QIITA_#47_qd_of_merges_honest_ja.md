# #47 歩行ロボットの進化エンジンで LLM をマージしたら、正直つらい話になった — QD-of-merges 実験の全記録

> 前回(#46)は「異なる LLM は重みでは混ざらない。混ぜられるのは蒸留だけ」という**理論**の話でした。今回はその実証編です。**歩行ロボットを進化させる自作エンジン(MAP-Elites)を、そのまま LLM の「マージレシピ探索」に転用**して、同じ土台の兄弟モデルを実際にマージし、held-out perplexity で測りました。
>
> 結論を先に言うと、**私の「賢い探索」は、正直に測ると、素朴なベースラインに負けました**。最初は「勝った!」と思ったのですが、ベースラインを公平にした瞬間に優位が消えました。この記事は、その偽陽性の発見から、別の手法での再挑戦、そして最後に効いた「別の場所」までを、負けも含めて時系列で記録したものです。
>
> 立場の開示: 筆者は自宅の CPU 環境で省メモリな小型 LLM を組む個人開発(FullSense)を進めています。実験はすべて CPU・小型モデル(1.35 億パラメータ)・小コーパスで、単一 seed です。**「うまくいった」より「うまくいかなかった内訳」に価値がある**という方針(honest disclosure)で書きます。数値には一次情報(arXiv ID)を添えます。

---

## 0. この記事で持ち帰れるもの(要旨)

- **ロボットの進化エンジン(MAP-Elites)は、一字も変えずに LLM のマージレシピ探索へ転用できました。** 枠組みの転移そのものは成立します。
- **でも「進化探索が普通のマージに勝った」という最初の結果は、偽陽性でした。** 対抗馬(ベースライン)のグリッドが**粗すぎた**だけで、細かくしたら普通のマージが勝ちました。
- **「ちゃんとした」多エキスパート・マージ手法(TIES)を試しても、素朴な足し算(task arithmetic)に負けました。** TIES が想定する「衝突の解消」が、実際には**相補的な更新を捨てる**方向に働いたためです。
- **一方、重みではなく「活性(activation)」を操作する別手法は、綺麗に効きました。** 可逆・再訓練不要で、感情を出力に注入できました。
- **いちばんの教訓は技術ではなく態度です:** 手法の勝ちを主張する前に、**ベースラインを最適化し切る**こと。粗いベースラインは偽陽性を生みます。

専門の方は §4(偽陽性の解剖)と §7(教訓)だけでも要点が取れます。はじめての方は §1(用語)→ §2(比喩)から。

## 1. 用語(さきに地図を配ります)

- **モデルマージ(model merging)**: 複数モデルの**重みを直接混ぜて** 1 つにする手法。同一の土台(base)から派生した兄弟同士なら効きます。
- **task vector(タスクベクトル)**: 「微調整後の重み − 土台の重み」。そのモデルが**学んだ差分**を表すベクトル(arXiv:2212.04089)。
- **SLERP / TIES / DARE**: マージの代表的な数式。SLERP=球面上の補間、TIES=符号の多数決で衝突を解消(arXiv:2306.01708)、DARE=一部をランダムに落として薄める(arXiv:2311.03099)。
- **perplexity(パープレキシティ、PPL)**: 言語モデルが次の単語をどれだけ「驚かず」当てられるかの指標。**低いほど良い**。
- **MAP-Elites / QD(Quality-Diversity)**: 「性能」だけでなく「振る舞いの多様性」も同時に保つ進化的探索。もともとロボットの歩き方を多様に進化させるために使いました(arXiv:1504.04909)。
- **ベースライン(baseline)**: 手法の良し悪しを測る**比較の基準**。ここでは「素朴なマージ」がそれにあたります。
- **活性ステアリング(activation steering)**: 重みを変えず、推論中の**内部信号(活性)に方向ベクトルを足して**振る舞いを操舵する手法(ActAdd 2308.10248 / RepE 2310.01405)。
- **honest disclosure(正直な開示)**: 良い結果ほど内訳を疑い、失敗を消さずに教訓として残す態度。この記事の背骨です。

## 2. かみくだき(料理のブレンド、という比喩)

2 人の料理人がいます。一人は**だし**が得意(和食)、もう一人は**スパイス**が得意(カレー)。この 2 人の「腕」を 1 つのレシピに混ぜて、和食もカレーも上手い万能コックを作りたい。

- **素朴な足し算(naive linear)**: 両方の隠し味を半分ずつ入れる。だしもスパイスも「そこそこ」残る。
- **TIES(符号の多数決)**: 「だしを足す」派と「スパイスを足す」派で意見が割れた工程では、**多数決で片方だけ採用**する。衝突を「解消」する発想です。
- **進化探索(QD)**: いろんな配合比を大量に試して、いちばんバランスの良いレシピを探す。

直感的には「賢く衝突を解消する TIES」や「大量に探す QD」が勝ちそうです。ところが今回の 2 人の料理人は**相補的**でした。だしとスパイスは**両方あってこそ**万能になる。多数決で片方を捨てる TIES は、かえって味を壊した ── これが今回いちばん意外だった発見です。

## 3. 舞台設定:ロボットの進化エンジンを LLM に向ける

私は別プロジェクトで、**四足・二足ロボットの歩き方を進化させる**エンジンを書いていました。中核が MAP-Elites という QD アルゴリズムです。「速く歩く個体」だけでなく「跳ねる」「すり足」など**多様な歩容**を格子(archive)に保存し続けます。

ある日ふと思いました。**「歩容を進化させる代わりに、モデルの『マージレシピ』を進化させたら?」**

マージレシピを遺伝子(genome)として符号化すれば、歩容進化のコードは**一字も変えずに**流用できます。遺伝子 = [どれだけ疎にするか(density)、どれだけ落とすか(drop)、各エキスパートの混合係数 λ]。評価関数だけを「歩いた距離」から「マージ後モデルの perplexity」に差し替える。これが **QD-of-merges** の発想です。

土台には、同じ base から派生した兄弟モデルを選びました:

- **SmolLM2-135M**(base、素の言語モデル)
- **SmolLM2-135M-Instruct**(指示追従を微調整した兄弟)

この 2 つは**重みの形が完全に一致**し(マージ可能)、task vector の大きさ(‖τ‖)も十分にありました。そして held-out perplexity には、きれいな**トレードオフ**が実在しました:

| モデル | 素の文章 PPL | 指示応答 PPL |
|---|---|---|
| base | **6.40**(得意) | 5.30 |
| Instruct | 7.25 | **3.58**(得意) |

素の文章は base が、指示応答は Instruct が得意。**片方を立てれば片方が落ちる**。マージで両方を回収できるか? ここから実験が始まります。

## 4. 第一幕:「勝った!」と思ったら、偽陽性だった

まず、task vector を**層ごとに違う強さで**接ぎ木する探索をしました。「早い層は base のまま(素の言語能力を守る)、遅い層は Instruct を強く入れる(指示能力を得る)」といった配分を、QD に自動発見させたのです。

評価は 2 つの perplexity のバランス(調和平均)。対抗馬として、全層を**同じ係数 λ で混ぜる**均一マージを**7 点**のグリッドで測りました。結果:

- 均一マージ(粗い 7 点)の最良: バランス指標 **0.727**
- 層別 QD の最良: **0.797**

しかも QD が見つけた配分は、まさに直感どおりの非一様(早い層 λ≈0、遅い層 λ≈1)。**「層ごとに配分する自由度が効いた!」**と、この時は本気で思いました。図にすると QD の点が均一マージの曲線の「外側」に出ていて、いかにも勝っている。

### ここで踏みとどまる(honest disclosure)

でも、待てよ、と。**対抗馬の 7 点は、粗すぎないか?** 均一マージの perplexity 曲線をよく見ると、λ を上げると素の文章 PPL が **6.40 → 7.77 に一度悪化**してから、λ≈0.65 付近で急に良くなる、**非単調で鋭い谷**を持っていました。7 点では、この谷の底を**跨いで見落とす**可能性がある。

そこで均一マージを **51 点**に細分化して測り直しました。すると:

- 均一マージ(細かい 51 点)の最良: λ≈0.68 で **0.838**

**層別 QD(0.797)は、細かくした均一マージ(0.838)に負けました。** 最初の「勝ち」は、対抗馬のグリッドが粗くて最適な λ を跨いでいた**だけ**の偽陽性だったのです。単一の task vector の接ぎ木は、結局のところ**ほぼ 1 次元**の問題で、たった 1 つの均一な λ をちゃんと調整すれば、層別の自由度で出し抜けるものではなかった。

これは苦い結果ですが、**メタのレベルでは勝ち**です。もし 7 点のまま「QD が勝った」と発表していたら、それは嘘でした。ベースラインを公平にする、という一手間が、偽陽性を潰しました。

## 5. 第二幕:「ちゃんとした」多エキスパート手法でも、素朴な足し算に負けた

「単一の τ が 1 次元なら、**2 つの τ を混ぜれば**話が変わるはず」── これが次の仮説でした。ここで登場するのが TIES です。TIES は複数モデルの task vector を混ぜるとき、**符号が食い違う座標を多数決で解消**します。衝突がある時こそ真価を発揮する、というのが売りです。

問題は、同じ base の「2 人目のエキスパート」が必要なこと。ダウンロードで済ませてもよかったのですが、**再現性と制御のため自作**しました:

- SmolLM2-135M(base)を、**標準ライブラリの Python コード** 123KB で 120 ステップだけ CPU 微調整。
- 結果、そのコードの held-out perplexity が **10.18 → 5.16 に半減**。**code-expert** の完成です。

これで 3 者が揃いました: base / chat-expert(Instruct)/ code-expert。指標も、指示応答 PPL(chat が勝つ)と コード PPL(code が勝つ)で、**きれいに分離**しました。いよいよ TIES と QD の出番です。結果:

| マージ手法 | バランス指標 |
|---|---|
| **素朴な足し算(0.5, 0.5)** | **0.746** |
| TIES 均一(0.5, 0.5) | 0.000 |
| TIES グリッド探索(4×4)の最良 | 0.000(全点が 0) |
| QD で調整した TIES の最良 | 0.662 |

**素朴な足し算(0.746)が、あらゆる TIES(0.000〜0.662)を上回りました。** TIES 均一に至ってはコード PPL が 13.7 まで**悪化**しています。

### なぜ TIES が負けたか(観測からの解釈)

TIES は「符号が食い違う = 片方はノイズだから捨てる」という前提の手法です。ところが chat-expert と code-expert は**相補的**でした。指示能力とコード能力は**両方あってこそ**万能になる。符号が食い違う座標で TIES が(振れ幅の大きい)chat 側を採り、code の有用な更新を捨てた結果、コード能力が崩れた ── §2 の料理のたとえのとおりです。

**TIES は衝突の解消が本当に要る時にしか効かない。** 相補的なエキスパートには、両方を残す素朴な足し算が勝つ。QD で疎化(density 0.89 / drop 0.82)を強めて足し算に近づけると 0.662 まで回復しましたが、それでも素朴な足し算には届きませんでした。

こうして **#22(モデルマージ)の結末が確定しました: 単一 τ でも複数 τ でも、適切に調整した簡単なベースラインが、進化探索や TIES の追加機構を上回る。** この小さいモデル・同じ base の兄弟という設定では、凝った探索は割に合わなかった。

## 6. 第三幕:場所を変えたら、あっさり効いた(活性ステアリング)

重みを混ぜる話はここまで全部 null(有意差なし/負け)でした。では**別の場所**── 重みではなく、推論中の**活性(activation)**── を触ったらどうか。

やり方はシンプルです(ActAdd / RepE):

1. 対照的なプロンプト対を用意する。ポジ「I love this, wonderful and…」/ ネガ「I hate this, terrible and…」。
2. ある層でのそれぞれの内部活性を取り、**差ベクトル v = 平均(ポジ) − 平均(ネガ)** を作る。
3. 推論中、その層の出力に **α·v を足す**フック(hook)を張る。フックを外せば**完全に元通り**。

SmolLM2-135M-Instruct の 8 層目で試すと:

- **感情が α に沿って動く**: 中立プロンプトでの「ポジ語 − ネガ語」次トークン log-prob が、α=0 で +2.85 → α=+0.5/+1/+2 で **+3.95 / +5.14 / +6.81**(正方向は単調)。
- **完全に可逆**: フックを外すと α=0 の値が**厳密に一致**(差 = 0.0)。重みは一切変えていません。
- **生成も一貫**: 中立プロンプト "My day was" → α=+3 で "filled with peace, and a beautiful celebration… a simple joy"(明確にポジティブで文章として自然)。

正直な限界もあります。**強い負の α(−3)は「効かせすぎ(over-steer)」で文が壊れます**("it' is it' is…" の退化反復)。効果の幅は 135M ゆえ限定的です。それでも、**可逆・再訓練不要・base 非依存**という性質は、重みマージ(同一 base 必須)とは対照的に扱いやすい。これは対話ランタイムに**フックとして直挿し**でき、「性格ダイヤル」を CPU で提供できる芽になります。

**融合の基質(substrate)を変えたら効いた** ── ここが第三幕の要点です。

## 7. いちばんの持ち帰り:手法ではなく、態度

この一連の実験でいちばん価値があったのは、新しいマージ手法ではありません。**次の 3 つの態度**でした。

1. **ベースラインを最適化し切ってから勝ちを主張する。** 第一幕の偽陽性は、対抗馬が粗かっただけでした。手法が良く見える最大の理由は、しばしば「比較相手が弱い」ことです。分解能を上げる、という一手間が偽陽性を潰します。
2. **「ちゃんとした手法」の前提を疑う。** TIES は「衝突を解消する」名手ですが、そもそも**解消すべき衝突がない(相補的)**なら、その機構は害になります。手法の想定と、目の前のデータの構造が合っているかを確認する。
3. **null を消さない。** うまくいかなかった 2 幕を捨てて、効いた第三幕だけ書けば「成功譚」になります。でもそれは嘘です。**どこで効かなかったかの地図**こそ、次に同じ轍を踏まないための資産です。

これは私の個人開発の一貫した方針(honest disclosure)ですが、研究一般に効く態度だと思います。異常に良い結果が出たら、**勝った気になる前に必ず内訳を疑う**。

## 8. 残ったもの・次にやること

- **確かに残った成果**: ロボット進化エンジン → LLM マージの harness 転移が動くこと、端点(base/Instruct)を厳密に再現できること、マージ数式の実装が数値的に正しいこと(numpy 版と一致)、自作エキスパートのパイプライン、そして活性ステアリングが可逆に効くこと。
- **未検証(この結論を一般化しない)**: もっと大きいモデル、**本当に衝突している**エキスパート集合、healing(接合後の再学習)付きの深さ方向マージ(frankenmerge)では、別の結末があり得ます。今回の null は「135M・同一 base 兄弟・小コーパス・単一 seed」の範囲の話です。
- **次の一手**: 活性ステアリングを対話ランタイムに載せる(性格ダイヤル)。進化的マージは、より大きく・真に衝突するエキスパート集合で価値を再検証する。
