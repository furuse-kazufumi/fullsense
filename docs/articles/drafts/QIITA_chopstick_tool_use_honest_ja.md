# AIに「箸」で物を掴ませたら、また手作りに負けた話 — 「滑る」と「潰す」を分けて正直に測る

> 対象: ロボット操作・強化学習・進化計算・MuJoCo に興味がある人 / 「学習は手作りに勝つ」を検証したい人。
> 前提知識: Python が読めれば十分。ロボット・制御用語は都度かみくだきます。
> 数値はすべてローカル実測。この実験はまだ**途中**で、結論ではなく「正直な途中経過と、その測り方」を書きます（続編で「滑る/潰れる直前を読む予測器」を試します）。

---

## 0. 3行まとめ（先に結論）

- 動機は **「手を失った人の AI 義手（prosthetic hand）が、箸で食事を掴めるか」**。MuJoCo で **AI 義手が 2 本の箸（chopsticks）** で物をつまむ最小リグを組み、「手で書いた制御」vs「進化で学習した方策」を、いつもの正直な物差しで比べ始めた。箸は**本物の持ち方**（下は固定・上だけ動かす）に寄せた。※まだ動く義手ではなくシミュレーションのトイ。
- まだ易しい入り口（箱をつまむ）だが、早くも**同じ結末の匂い**がする: 手書きのスクリプト制御は箱を自明に掴むのに、ゼロから進化させた素朴な線形方策は**接触までしか届かない**。ロケット着陸の記事と同じ「手作りが素朴な学習に勝つ」が、また顔を出した。
- でも本題はそこではない。物をつまむ研究は世の中に山ほどある。**私が価値を置くのは「失敗を段階ごとに帰属できる正直な測り方」**——「掴めた」を最終姿勢で判定し、「滑って落とした」と「握り潰した」を**別々に**測る。この記事は、その物差しをどう作ったかの話だ。

---

## 1. 用語（先に地図）

| 用語 | かみくだき |
|---|---|
| 把持（grasp / grasping） | ロボットが物を「掴む」こと。ここでは箸 2 本で挟む |
| 道具使用操作（tool-use manipulation） | 手そのものでなく、「箸」という**道具を介して**物を操ること。素手の平行グリッパより接触が複雑 |
| 分割評価（split evaluation） | 成功率を 1 本の数字にせず、「整列／接触／滑り／持ち上げ」を**別々に**測る。どの段階でコケたかを帰属できる |
| 滑り（slip） | 掴んだ（つもりの）物が、握りから逃げること。丸い物で起きやすい |
| 潰し（crush） | 柔らかい物を握りすぎて壊すこと。ここでは「握力の上限（force cap）を超えたら潰した」と見なす代理指標 |
| condim（contact dimensionality） | MuJoCo の接触が持つ自由度。3=滑り摩擦だけ、4/6=**ねじれ・転がり摩擦も**。丸い物が握りから回って逃げるのを止めるのに要る |
| 窓積分（windowed impulse） | 一瞬の接触力ではなく、一定時間の力を**均して**測ること。ソルバのノイズを「予測できた」と偽らないための衛生 |
| MAP-Elites / ES（進化戦略, evolution strategy） | 解の集団を変異・選抜して良くする最適化。ここでは箸の制御方策を進化させる |
| held-out | 学習に一切使っていない新しい設定での成績。汎化の唯一の証拠 |
| budget-matched baseline | 学習と**同じ計算予算**で調整した手作りの対抗馬。「弱い相手に勝った」を封じる公平な審判 |

## 2. なぜ「箸」で、なぜ先に先行研究を調べたか

新しい機能を作る前に、まず調べる。これは私の習慣だ。今回もそうして、そして**正直に計画を縮めた**。

ロボットに箸を使わせる研究は、すでに厚い。ワシントン大学の一連の仕事（*Grasping with Chopsticks*, ICRA 2021 / *CherryBot*, 2023）は、実機の箸で模倣学習・強化学習を回し、空中で揺れる小物を掴むところまで行っている。さらに決定的なのが **Yang, Yin, Liu, *Learning to Use Chopsticks in Diverse Gripping Styles*（SIGGRAPH 2022, arXiv:2205.14313）**——物理シミュレーションの**人型の手が箸を保持し、多様な持ち方を自動発見して物を運ぶ**。私が「野心的な最終形」として思い描いていた「人型ハンドで箸を持つ」段は、**この論文がすでに高い完成度でやっていた**。

だから正直に、その段は**切った**。「多様な持ち方を進化で見つける」も、Yang らが別手法（ベイズ最適化）で達成済みだから、それだけでは差別化にならない。

では、この題材で私が正直に主張できる新規性はどこにあるか。タスクではない。**「滑る・潰れる直前を予測する低次元の世界モデルを、進化の選択圧（ゲート）として使えるか」**——これ一点だ。そしてそこに賭ける前に、まず**正直に測れる土台**が要る。この記事はその土台の話で終わる（賭けの結果は次回）。

> 教訓の先取り: 調べると、しばしば「自分の面白い案」は誰かがやっている。落胆ではなく、**どこが本当に空いているか**を教えてくれる地図だと思えばいい。

## 3. 舞台: 2 本の箸でつまむ、を最小構成で組む

MuJoCo に、次の最小のリグ（rig = 実験台）を建てた。イメージは **「手を失った人の義手（prosthetic hand）に AI を組み込んだら、箸で食事を掴めるか」**——介助技術（assistive technology）としての動機だ。だから機構を**本物の箸の持ち方**に寄せた。正しい持ち方を調べると、箸は**非対称**に持つ：**下の箸は固定**（親指の付け根と薬指で支え、まったく動かさない）、**上の箸だけ**を親指・人差し指・中指で動かして「一の字」を書くように開閉する（出典: [はし和文化研究会](https://hashiwabunka.jp/hashi/howto) / [はりま屋](https://www.rakuten.co.jp/dohichushop/contents/ohashinomochikata/) / [千葉市 幼児教育資料](https://www.city.chiba.jp/kodomomirai/yojikyoiku/shido/documents/hashinomochikata.pdf)）。私のリグもそれに従い、AI 義手が **下の箸を固定した梃子（anvil）** として保持し、**上の箸 1 本だけ**を支点まわりに回して先端を閉じる。駆動は「義手＝指の作用 → 上の箸（梃子）→ 物体」で、物体は下の固定箸に押し当てて挟む。義手ごと上下して持ち上げる（lift）。挟まれる物（block）は自由に動ける剛体だ。

> 正直な設計変更（初版 → 現在、3 段の訂正）: **(1)** 初版は各箸を直動関節（prismatic slide）で直接動かす「箸の形をした 2 本指グリッパ」で、手も指も無かった。**(2)** 次に手が箸を握る梃子に作り替えたが、**両方の箸を対称に動かしていた**——これは箸の持ち方として間違い。**(3)** 調べ直すと、本物の箸は**下を固定し上だけ動かす**非対称だと分かったので、下の箸を関節ロックで固定・上だけ可動にし、見た目も **AI 義手**（ガンメタルの筐体＋分節した指＋手の甲で青く光る AI モジュール）にした。数値は毎回取り直したが、結論（後述）は変わっていない。**正直に言えば、これはまだシミュレーションのトイであって、動く義手ではない**——「AI 義手が箸を使う」という動機を、制御の問い（手作り vs 学習）に結びつけるための最小の舞台だ。

```
world
 └─ hand（AI 義手）      （z スライド＝腕ごと持ち上げ / 指・AI モジュールは可視）
     ├─ stickL（上の箸） （支点まわりの hinge 回転＝この 1 本だけ動かして閉じる）
     └─ stickR（下の箸） （関節ロック＝固定。動かさない anvil）
 block                   （自由関節つき・箱/球/円柱）
```

![AI 義手が正しい持ち方で箸を握る——下の箸は固定、上の箸 1 本だけを動かして先端を閉じ、箱を挟んで持ち上げる（scripted 制御・held=成功）。手の甲の青く光る部分が組み込みの AI モジュール](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/chopstick/box_grasp.gif?v=4)

XML はロケット記事と同じく生の f-string で書いた（小さな手作りシーンは、手続き API より読みやすい）。ここで**最初のロボティクスの罠**にはまる。

**罠①: 丸い物は握りから「回って」逃げる。** 箸 2 本のつまみは、接触点が 2 つでほぼ一直線。この軸のまわりに物が回る自由が残る。既定の接触（condim=3, 滑り摩擦だけ）だと、球はくるっと回って握りをすり抜ける。止めるには**ねじれ・転がり摩擦（condim=6）**が要る。逆に言えば、ここの摩擦値は「掴める/掴めない」を二値で決める強力なノブで、**答えが出るまで摩擦をいじると、それは実験者の自由度（experimenter's degrees of freedom）＝ズル**になる。だから摩擦は「stage-2 で事前登録するノブ」と決め、stage-1 では固定した。

```xml
<default>
  <geom condim="6" friction="1.2 0.05 0.02" solref="0.01 1" solimp="0.9 0.95 0.001"/>
</default>
```

**罠②（後述の潰し判定）** も同じ匂いがする——接触力は timestep とソルバ設定に敏感で、一瞬だけスパイクする。これは §4 で回収する。

## 4. 正直な物差し: 「掴めた」を最終姿勢で、「滑る」と「潰す」を分ける

ここが記事の芯だ。物をつまむ実験は「成功率 80%」みたいな一本の数字で報告されがちだが、それでは**どの段階でコケたか**が消える。私は次の 3 つを**排他的に**帰属する。

- **held（掴めた）** = 最後の保持窓（trailing hold window）で、両側が接触していて・持ち上がっていて・（cap があれば）潰していない。ここが正直さの肝: 途中で一瞬触れただけは掴んだと見なさない。**最終の静止状態で判定する**（過渡の xy や瞬間の持ち上がりで判定すると偽陽性が出る——これは過去の運動学把持で痛い目を見た教訓だ）。
- **slipped（滑った）** = held でなく・潰してもいない・かつ「掴んだのに逃げた」か「握らないまま転がり去った」。丸い物の失敗モードだ。
- **crushed（潰した）** = 握力の上限（force cap）を、保持窓の**窓積分（最大）力**が超えた。ここで**一瞬のピーク力でなく窓で均す**のが罠②への対処: solref/solimp/timestep 由来のスパイクを「予測できた」と誤って成果にしないため。

コードにすると、失敗が段階に落ちる:

```python
capped   = spec.force_cap > 0.0
crushed  = capped and grip_force_max > spec.force_cap      # 握りすぎ
held     = final_two_sided and lifted and not crushed       # 潰さずに持ち上げた
slipped  = (not held) and (not crushed) and (contacted or xy_escape > slip_threshold)
```

そして物の形を変えるだけで、**本物の難易度の勾配**が出る（すべて実測）:

| 物体 | 手書きスクリプト制御の結果 |
|---|---|
| 箱（box） | **held**（掴めた） |
| 球（sphere） | **slipped**（滑った——接触する前に転がり逃げることさえある） |
| 箱＋低い握力上限（cap 0.3N） | **crushed**（同じ握りが上限を超える） |
| 一度も閉じない制御 | どれでもない＝**clean miss**（誤って「滑った」に数えない） |

![同じ scripted 制御で球をつまもうとすると、転がって握りから逃げる（slipped=失敗）](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/chopstick/sphere_slip.gif?v=4)

この 4 行が意味するのは、「箱は自明・球は難しい・締めれば潰す」を**別々の指標として観測できている**ということだ。実験として、ここまでが土台。

## 5. 学習を足す: 反応する制御を進化させる

手書きのスクリプトは**開ループ**（決め打ちの時間スケジュール）で、球が転がり出しても対応できない。だから**閉ループ**の方策を進化させる。観測（箱の位置・箸の位置・持ち上げ量・両側接触フラグ）から、対称な挟み量と持ち上げ量を出す小さな線形方策だ。最適化は `SimpleGaussianES`（cma があれば CMA-ES）を、ロケットや歩行進化で使ったのと**同じコア**を無改変で流用した。

```python
def linear_controller(genome, spec):
    w = genome.reshape(2, 6)          # [pinch, lift] × [bias, bx, by, blift, handlift, two_sided]
    def command(t, obs):
        out = w @ features(obs)
        return np.array([pinch, pinch, lift])  # クリップは省略
    return command
```

## 6. 早くも出た「正直な予備結果」——易しい箱ですら、手作りに負ける

ここで**過大に語らない**ことがすべてだ。これは stage-2 の本実験ではない。ごく小さな予算での、易しい箱の上での機構チェックにすぎない。だがその予備結果が、正直で、そして見覚えがある。

- 何もしないゼロ方策の適応度 = **0.00**（当然、何も掴まない）。
- 進化させると **0.61 → 0.50 程度**まで上がる。中身を見ると **contacted=True（接触は latch した）だが held=False**。つまり**掴みかけるが、持ち上げまで至らない**。
- 一方、手書きのスクリプト制御は同じ箱を**自明に held**（適応度 ~6）。

素朴な線形方策をゼロから小予算で進化させると、**接触までは行くが、手書きには遠く及ばない**。これは、ロケット着陸の記事で見た **robust NULL（学習は公平な PID に勝てず引き分け）** と同じ地層だ。手作りの制御が、素朴な学習にまた勝っている。

正直な但し書きを厚めに置く: 予算は小さいし、初期化はゼロだし、線形方策は表現力が低い。**「学習は箸で勝てない」と主張しているのではない**。「易しい入り口の、素朴な設定では、手作りが強い」という**予備観察**だ。だがこの再現性こそが、次の賭けの動機になる。

## 7. なぜ「負け」を喜ぶのか（gift-to-reader）

- **手作りに負けたら、まず喜べ。** それは「ベースラインが強い」という証拠だ。学習が弱い審判にだけ勝つのは、成果ではなく計測の失敗。
- **「学習が勝った」は、反証してから言え。** 分割評価・budget-matched baseline・最終姿勢での判定・窓積分での潰し判定——これらは全部、自分の楽観を殺すための仕掛けだ。
- **正直な null は、再現すると"法則"になる。** ロケットで一度、箸でもう一度。「素朴な学習は手作り制御に勝てない」が二度出たなら、問うべきは「では、いつ勝つのか」だ。
- **タスクの新規性を追わない。** 箸把持は既出だ。価値は「同じ正直な物差しで、最後まで測り切る方法」の側に置く。

## 8. 次回に続く

土台はできた: 箸のリグ、滑る/潰す/掴めたを分ける物差し、そして進化の機構。難易度の勾配（箱→球→潰し）も観測できている。残っているのは、いちばん面白い問いだ——

> **「滑る直前・潰れる直前」を予測する低次元の世界モデルを作り、それを進化の選択圧（ゲート）として与えたら、球や柔らかい物で、学習は手作りを超えられるか?**

もし超えられたら、それは「反応の速さ」でも「記憶」でもなく、**"予測"** が効いた最初の証拠になる。もし超えられなくても——それはそれで、正直な null がまた一つ、地図に増えるだけだ。どちらでも私は書く。

次回は、その予測器を、公平な審判（budget-matched baseline）と held-out の上で、事前登録つきで殴り合わせる。良い数字が出る前に、まずコケ方が見えるはずだ。
