# 作って分かった LLM の中身 ― 自作言語モデルで覗く構造【シリーズ総目次】

著者: 古瀬 和文（ぷるやん）

> このページは、大規模言語モデル(LLM: Large Language Model)の構造を「自分で組み直して確かめる」視点で解説する
> 連載シリーズの入口です。**一般版**（絵と比喩で腑に落とす）と**技術版**（数式・コード・実測で納得する）の2本立てで、
> 同じ地図を二つの高度から通ります。どちらから読んでも、途中から読んでも大丈夫なように書いています。

---

## このシリーズが他と少し違うところ

LLM の解説はすでに世の中にたくさんあります。このシリーズが付け加えたいのは、たった一つ――
**「フレームワークのブラックボックスを開けて、推論エンジンを自分で組み直し、公式実装と実測して誤差ゼロで再現した」**
という一次体験です。

図を眺めるのと、部品を自分で削り出して組み立て、動かして測るのは、理解の質が違います。私は後者をやりました。
自作した推論エンジンの出力が、公式のリファレンス実装と**浮動小数点の丸め誤差の域(logits 2e-4)で一致**し、
メモリ最適化版に至っては**1ビットの狂いもなく完全一致(max|Δ|=0.0)**しました。

「組み直したら、寸分違わず同じ答えが出た」――これが、私が中身を一つずつ説明できる根拠です。
うまく再現できたということは、各部品が何をしているかを取り違えていない、ということだからです。

> この「測って確かめる」姿勢は、私の職業病かもしれません。詳しくは下の自己紹介で。

## 書いている人（なぜ私がこれを書くのか）

私は 25 年以上、「カメラで見て、機械を動かす」計測・制御の現場でシステムを作ってきたエンジニアです。
外観検査、実装部品の位置決め、X線検査、三次元計測、レーザ干渉計測――「現場で不良を取りこぼさず、ラインを止めない」ことを
目標に、画像処理と制御の両面から装置を組んできました。

- **ソフトウェア**: C/C++ による画像処理ライブラリ開発を中心に、多項式近似・スプライン補間・フーリエ変換・主成分分析(PCA)
  などの数値・統計アルゴリズムを、必要に応じて自分で実装。Python で周辺ツールやアプリ、そして RPA(定型業務の自動化)も自作。
  計測業界にいたので、数理解析(数値・統計解析)は得意分野です。
- **ハード・計測**: 結像光学系、近赤外・短波赤外カメラ、レーザ干渉計、導波路・ファイバー芯合わせ、モアレによる位置決めなど、
  光学とセンサの設計・評価。DIO/AIO、ステージ制御、EtherCAT で装置全体を動かすところまで。
- **少し変わった経験**: 公式 SDK の無い双腕ロボットを、付属 Windows アプリを Windows メッセージ経由で操作する自作プログラムで動かし、
  両目と腕先カメラで校正プレートの位置・姿勢を推定しながら、頭・腰・両腕・手首の角度を自動で合わせる制御ループを組みました。
  **「きれいな API が無くても、現物をどうにかして動かす」** タイプの仕事は、かなり場数を踏んでいます。

このシリーズは、その延長線上にあります。**「測って・動かす」を 25 年やってきた人間が、その同じ規律で LLM を分解したら
何が見えたか。** フーリエ変換は位置エンコーディング(RoPE)として、相関・テンプレートマッチングは注意機構(attention)のスコアとして、
PCA は埋め込み空間の直観として、キャリブレーションは学習ループとして――現場の道具が、そのまま LLM の中で顔を出します。
その驚きも含めて、お裾分けできればと思います。

## 読み方ガイド

- **AI をよく使うけれど中身は気にしたことがない方** → まず**一般版**を #0 から順に。比喩と絵だけで通れます。
- **実際に AI 開発をする／したい方（レポートとして読みたい方）** → **技術版**を。数式・擬似コード・実測値・
  「実務で効く落とし穴」まで踏み込みます。モデル選定・評価の罠・メモリ最適化・責任ある設計は技術版 #5・#6 が中心です。
- **両方** → 各回、一般版で全体像を掴んでから技術版で深掘りすると、いちばん楽に入れます。

各回は「①用語ミニ辞典 → ②かみくだき → ③詳細」の順で、途中で詰まらないように作っています。

## 全体地図

```
あなたの文章
   │
   ▼
① 言葉を細かく刻む(トークン化) ─┐
② 意味の座標に変える(埋め込み) ─┴─ 第1回
   │
   ▼
③ どこに注目するか決める(注意機構) ── 第2回 ★心臓
   │
   ▼
④ 知識を引き出して考える(順伝播層) ── 第3回
   │  （③④を何十層も重ねる）
   ▼
⑤ 次の一語に点数をつけて選ぶ(出力・学習) ── 第4回
   │
   ▼
次の一語 →（①へ戻って繰り返す）

そして現実の壁：
   ⑥ 長い会話でメモリと速度が壁になる(KVキャッシュ・量子化・線形化) ── 第5回
   ⑦ 実際に開発で使うには(モデル選定・評価・責任) ── 第6回
```

## このシリーズを貫く約束（honest disclosure）

- **数字は実測だけ**を載せます。うまくいかなかったこと（自作の文字言語モデルが会話にならなかった、
  極端な圧縮でモデルが壊れた、素朴な進化探索が単純な貪欲法に負けた）も消さずに、教訓として残します。
- **「異常に良い結果は、まず内訳を疑う」**。これは計測の現場で、うますぎる測定値をまず校正から疑うのと同じ規律です。
- 自作の推論ランタイムがもたらしたのは「中身を検査・改造できること」であって、**会話の賢さそのものは学習済みの重み由来**です。
  その継ぎ目をぼかしません。
