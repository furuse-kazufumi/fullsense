# 筋骨格モデルに水を飲ませる — 「口に入っていない」の一言から、ボトルが床をすり抜けていたバグを追った話

> 対象: MuJoCo / ロボット / 生体力学シミュレーションを触る人 / 良い結果より**正直な内訳と、詰まって直すまでの過程**を読みたい人。
> 前提知識: Python が読めれば十分。MuJoCo とバランス制御の用語は都度かみくだきます。
> この記事は成功談ではありません。**「飲めた」と思っていたものが飲めておらず、直そうとしたら別のバグが顔を出し、その真犯人が私の最初の推測と違った**——その一部始終を、測った数字ごと残す話です。

---

## 0. 3行まとめ(先に結論)

- 前回、**700本系の筋骨格モデル(MS-Human-700)**の右手——このうち操作用変種の**81本の筋肉**——に水ボトルを掴ませ、「口元へ運んで飲む動作ができた」と書いた。だが**動画を見た人間の一言「口に入っていない・口も開いていない」で、それが過剰主張だと分かった**。
- 直そうとして分かった二段構え: (1) 目標点が**喉の高さ**にあって唇より低かった(私の置き間違い)。(2) そもそもこの操作用モデルには**顎関節も顔面筋も無く、口が物理的に開けない**。→ 顎を開ける**派生モデル**を作った。
- ところが派生モデルに学習済みの制御を載せると、**腕が空を掴み、ボトルは床をすり抜けて消えた**。犯人を「腕関節のズレ」と見当をつけたが——**測ったら外れ**。真犯人は「関節を1本足したせいで、**位置で決まる初期姿勢データ(keyframe)の番号が全部1つずつずれ**、ボトルの高さ欄に別の値が入り、**1.57m 落下して床下(z=−0.63m)に沈んでいた**」こと。**0 を一つ、正しい場所に挿す**だけで直った。直後、筋肉の手はボトルを 0.62m 持ち上げ、開いた口元へ運んだ。

この記事の主役は「飲めた」ではなく、**その手前の、地味で、しかし MuJoCo を触る誰もが一度は踏むバグ**だ。

## 2. 発端: 「口に入っていない」の一言

これは連作の続きだ。前回([700本の筋肉で人体骨格を立たせようとして、正直に転んだ話](https://qiita.com/furuse-kazufumi/items/0ecb640dc6e86452a23c))で、私はこの筋骨格モデルの手に「掴んだボトルを命令した場所へ運ぶ」制御を進化させた。その最難の目標が「体側・口元へ運ぶ」——つまり**飲む/食べる動作**だった。

そして私は、目標点を「実際の口」に置き直したら**同じ制御の容量のまま口元まで届いた**、と判定した。数字上は届いていた(ボトル中心と目標点の距離が数センチ)。「飲む動作ができた」と書きかけた。

ところが、レンダリングした動画を見た人間(私に方向性を指示する側)から、**3つの棄却**が返ってきた。

1. **「口に入っていない」**——ボトルは喉・鎖骨のあたりで止まっていて、実際の顎(あご)には届いていない。
2. **「口も開いていない」**——そもそも口が閉じたまま。飲んでいるように見えない。
3. **「何をしているのか分からない」**——正面カメラ1つでは、腕の動きが読み取れない。

これは、このシリーズがずっと守っている作法——**「異常に良い結果が出たら、勝った気になる前に内訳を疑う」**——が、他人の目によって作動した瞬間だった。私は「距離が数センチ」という**数字を信じて、絵を疑わなかった**。人間は絵を見て、数字の嘘を見抜いた。正直に、主張を取り下げるところから始めることにした。

---

## 3. 一つ目の訂正: 目標が「喉」の高さだった

まず (1) から。私が「口」として設定していた目標点は `(0.07, 0, 1.46)` だった。だが実際の顎のジオメトリを測ると、下顎の底は高さ **1.50m**、唇のあたりは **1.55m** 付近にある。私の目標 `1.46` は**顎の底より下、つまり喉・鎖骨の高さ**だった。

ボトルは「口元」でなく「喉元」に運ばれて、そこで止まっていた。距離の数字が小さかったのは、間違った目標にはちゃんと届いていたからだ——**目標の置き場所が間違っていれば、達成率が高いほど質が悪い**。目標を実際の唇の高さ `(0.05, 0, 1.55)` に直した。

これは前回の記事で私が書いた「単一の線形シナジーには、腕を体側へ引き戻す容量が足りない(だから届かない)」という説明への、自分自身への訂正でもある。**届かなかった真因は容量ではなく、目標の置き間違いだった**。容量のせいにするのは、いつでも一番ラクな逃げ道だ。

> 教訓の先取り: 「アルゴリズムの限界だ」と結論する前に、**採点表(目標・報酬)が正しいかを疑え**。停滞の原因を能力に帰属するのは、たいてい早すぎる。

## 4. でも、このモデルには口が無い

次に (2)「口が開いていない」。これは私の設定ミスではなく、**モデルそのものの限界**だった。

MS-Human-700 には全身・歩行用・操作用の3変種がある。今回使っている**操作用**変種は、腕と手をとことん精密に作る代わりに、**頭部を単一の剛体**として扱っている。顎関節も、顔面の筋肉も無い。81本の筋肉は全て腕・手・体幹のもので、**口を開ける手段が一つも無い**。つまり、どれだけ上手にボトルを運んでも、この体は原理的に口を開けられない。

指示は明快だった——「**顎を開くモデルに改造する**」。そこで、元モデル(251MBのアセット群)をまるごとコピーして不変のまま置き、その操作用の胴体定義ファイルだけを編集して、頭部の顎ジオメトリを**独立した子ボディ `mandible`(下顎)+ヒンジ関節 `jaw_open`** として切り出した。この関節に角度を与えれば下顎が降りて口が開く。

ただし、ここには最初から**正直な線引き**がある。**この顎には筋肉が付いていない**。だから顎の開閉は、筋肉で解く動きではなく、**位置を直接書き込む台本(kinematic)**だ。ボトルが口に近づいたら、こちらが顎を開く。**腕の把持と運搬は 81 本の筋肉が本当に解いているが、口が開くのは演出**——この区別は、この記事の最後まで持ち歩く。

改造したモデルは無事に読み込めた。関節が1本増えて自由度は 48 → 49。筋肉の数(81)は変わらないので、**学習済みの制御はそのまま載るはず**だった。顎を閉じた絵・開いた絵も個別に確認して、開閉そのものは動いた。

——ところが、そこに学習済みの飲む制御を載せた瞬間、話がおかしくなる。

## 5. バグ: 顎を足したら、ボトルが床をすり抜けた

改造した顎モデルで、把持・運搬を学習済みの制御(前回の champion)を再生した。期待は「元と同じように掴んで口へ運ぶ」。実際に起きたのはこれだ。

![バグ: 顎を足しただけのモデルで学習済みの制御を再生すると、腕は飲むように口へ上がるのに、手には何も無い。ボトルは床をすり抜けて消えている](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/musculo_drink/drink_jaw_broken.gif?v=1)

筋骨格は飲むように腕を口元へ上げる。**だが手は空っぽで、台の上のボトルも消えている**。制御はボトルがあるはずの場所を掴もうとして、そこに何も無いので、腕は空を切って発散していく。

顎を1本足しただけで、把持が完全に壊れた。筋肉の数も、腕の関節も、何一つ触っていないのに——だ。

## 6. 犯人探し: 推測を捨てて、まず測る

こういうとき、原因を**推測で決めてパッチを当てる**のが一番危ない。私は最初、こう見当をつけていた——「関節を1本足したから、腕の関節の初期角度がずれて、変な姿勢から始まっているに違いない」。もっともらしい。だがこのシリーズの規律は「**『たぶん X が原因』より『X を確認する、確認方法は Y』**」だ。

そこで、元モデルと顎モデルの**開始直後の状態を並べて測る**診断を書いた。腕の主要7関節の角度・手のひらの世界座標・ボトルの世界座標を、両モデルで比べる。結果はこうだ。

| 測った量 | 元モデル | 顎モデル | 差 |
|---|---|---|---|
| 腕の reach 関節 7本の角度 | (基準) | (同じ) | **最大 0.0** |
| 手のひらの世界座標 | (基準) | (同じ) | **0.0** |
| **ボトルの高さ z** | **0.94 m** | **−0.63 m** | **−1.57 m** |
| 手〜ボトル距離 | 0.108 m | 1.667 m | — |

私の推測は**外れ**だった。腕の関節も手の位置も、元と**完全に一致**している(初期姿勢は壊れていない)。壊れていたのは**ボトルだけ**——本来 z=0.94m の机の上にあるべきボトルが、**z=−0.63m、つまり床を突き抜けた地中**から始まっていた。

手は正しい初期姿勢にいる。掴む相手(ボトル)だけが、地面の下に落ちている。だから腕は正しく「机の上のボトルがあるはずの位置」へ動くのに、そこには何も無い。**空振りの正体はこれだ**。

> ここで「腕がずれたに違いない」と思い込んでコードを弄っていたら、直らないどころか、正しかった腕の制御を壊していただろう。**測ることは、間違った修理を防ぐ**。

## 7. 真犯人: keyframe は「番号」で決まる。関節を足せば、全部ずれる

なぜボトルだけ、しかも高さ 1.57m ぶんも落ちたのか。ここが、MuJoCo(に限らず物理エンジン)を触る人全員に効く教訓だ。

MuJoCo は、モデルの**全関節の位置を一列の配列 `qpos` に詰める**。各関節は、体の木構造の順に `qpos` の「何番目から何番目」という**場所(スロット)**を割り当てられる。そして「実験開始時の姿勢」を定義する **keyframe** は、その `qpos` を**値の並びそのまま**で書いた、ただの数の列だ——`qpos="-0.07 0.03 ... 0.94 ..."` のように。**i 番目の値は i 番目のスロットに入る**。位置で決まる。

さて、私は顎ヒンジ `jaw_open` を頭部に足した。MuJoCo はそれに **42番目のスロット**を割り当てた。すると——**42番以降の関節は、全部1つずつ後ろにずれる**。ボトルの位置を保持するスロットも、42→43 へ動いた。

ところが、**keyframe の数の列は 48 個のまま**、元モデルの並びを引きずっていた(私は顎を足したのに、開始姿勢の数列を更新していなかった)。モデルの `qpos` は 49 個に増えたのに、keyframe は 48 個。MuJoCo は足りない1個を末尾に 0 で埋める。結果、**42番目から先が、全部1つずつズレて読まれた**。

図にするとこうだ。

```
スロット番号:  ... 41    42(新)   43       44       45      46      47      48
本来の中身:    ... 腕末  [顎=0]   ボトルx   ボトルy   ボトルz   回転…    回転…    回転…
keyframe実際:  ... 腕末   0        0        0        -1.57    0.32    0       0(padding)
                          ↑ここから全部1つ手前の値を読んでいる
```

腕(スロット41以前)は無傷だった——**ずれは42番目から始まったから**。だがボトルは違う。ここで一つ、大事な前提を挟む: **このボトルは「机の高さ(0.94m)を原点にした上下スライド関節」に乗っていて、keyframe に入る値はその原点からの相対変位(メートル)を表す**。本来ボトルの上下スライドは 0(=机の上にそのまま)であるべきだった。

ところが1つずれた結果、その高さ欄に、**本来は隣の関節の回転角だった `-1.57` という値**が流れ込んだ。−1.57 は**ラジアンで測った回転角(約 −90 度)**のつもりの数字だ。それが**そのまま垂直スライドの変位(−1.57 メートル)として読まれた**——角度の数字が、距離の数字に化けたのだ。だからボトルは机の高さから 1.57m ぶん沈み、**0.94 − 1.57 ≒ −0.63m、床を突き抜けた地中**に現れた。

「0.94 − 1.57 = −0.63」が引き算ではなく足し算(相対変位の加算)になるのは、ボトルの高さが**絶対座標ではなく机からのスライド量**だからだ——ここを飛ばすと「z が丸ごと −1.57 に上書きされるはずでは?この一致は出来すぎ(こじつけ)では?」と疑われる。実際、この記事の内訳を検証してくれた別プロセスも、モデル定義(ボトルが上下スライド関節に乗っていること)を一次ソースで確かめるまで、一度は「破綻している」と判定しかけた。**キレイに合う数字ほど、機構まで遡って確かめる**——それはこのシリーズの信条そのものだ。

**関節を1本足すことは、keyframe の全スロットを1つずらすこと**なのに、私は数列を古いまま放置していた。緑のテストは通っていた(モデルは読み込めるし、関節数も筋肉数も正しい)。**テストが検出できない、位置ずれという静かなバグ**だ。そしてそれを暴いたのは、テストではなく——**動画を見た人間の「口に入っていない」の一言**だった。

## 8. 直す: 0 を一つ、正しい場所に

原因が分かれば、直し方は拍子抜けするほど簡単だ。**元の 48 個の keyframe に、顎のスロット(42番目)へ `0`(顎は閉じた状態=角度0)を1個挿し込んで、49 個にする**。そうすれば、42番以降は本来のスロットへ正しくずれ、ボトルは机の上に戻る。

```
変更前(48個): ... 0.3141  0 0 0  -1.57 0.32 0
変更後(49個): ... 0.3141  0 0 0 0  -1.57 0.32 0
                          ↑ 顎の 0 を1個挿入
```

適用して測り直すと、顎モデルは**元モデルと数値まで一致**した。

| 測った量 | 元モデル | 顎モデル(修正後) |
|---|---|---|
| ボトル開始 z | 0.94 m | **0.94 m** |
| 持ち上げ量 | +0.62m(z=1.56へ) | **+0.62m(z=1.56へ)** |
| 掴み続けた?(手〜ボトル) | ✓ (0.043m) | **✓ (0.043m)** |
| 口(唇)への最終距離 | 0.050m | **0.048m** |

手〜ボトルの相対位置ベクトルの差は **0.0(測定精度 1e-6 未満で一致)**——つまり顎モデルは、把持と運搬に関して元モデルと**区別がつかない**動きを再現した。学習し直しは一切要らなかった。壊れていたのは制御ではなく、初期化データの並びだったのだから。

## 9. 飲んだ

## 10. 正直な限界

気持ちよく終わりたいところだが、内訳を伏せると嘘になる。

- **顎の開閉は演出だ**。このモデルには顔面筋も顎の筋肉も無いので、口は物理で開いていない。ボトルが近づいたら、こちらが**下顎の角度を書き込んで開けている**(kinematic)。81本の筋肉が本当に解いているのは**腕の把持と運搬**であって、口の開閉ではない。しかも正直に言えば、台本はこの顎関節の**設計上の可動域(正側は +0.05 ラジアンまで)を大きく超える +0.6 ラジアンまで**、位置を直書きして開けている——qpos を直接上書きすると関節リミットが効かないからだ。つまり口の開きは、筋肉が無いだけでなく**可動域の面でも「物理を無視した演出」**だと明記しておく。GIF を見て「筋肉で口まで開けた」と受け取られると誇張になる。
- **「飲む」そのものはしていない**。液体の摂取をシミュレートしているのではなく、**ボトルを開いた口元へ運ぶ所作**だ。飲む「ジェスチャ」であって、飲む「生理」ではない。
- **命令追従はまだ弱い**。前回の記事の通り、この制御は「口へ運ぶ」という単一目標には届くが、任意の一点へ厳密に置く精度は3分の1に留まる。単一の線形シナジーの容量の話は、こちらでは今も生きている(§3で否定したのは「口に届かない原因が容量」という**誤帰属**であって、容量限界そのものではない)。
- **顎モデルは派生物**で、元の公開リポジトリには入れていない(元アセットは不変のままコピーし、胴体定義だけ改変)。再現するには同じ改変が要る。

これらを差し引いても、**「筋肉の手が水ボトルを掴んで口元へ運び、口が開く」**という所作は本物として残る。誇張は削り、実体は残す——それだけだ。

## 11. 何を持ち帰るか

この一件から、私が持ち帰る教訓は3つある。どれも、AI やロボットの派手な話ではなく、地味で、繰り返し効くものだ。

1. **停滞や失敗の原因を「能力の限界」に帰属する前に、採点表(目標・データ)を疑え**。「口に届かないのは制御の容量が足りないから」は、実際には「目標を喉の高さに置いていたから」だった。アルゴリズムのせいにするのは、いつも一番ラクで、たいてい早すぎる。
2. **推測でコードを直す前に、測って犯人を特定せよ**。「腕がずれたに違いない」は測ったら外れで、真犯人はボトルの初期位置だった。もし推測のまま腕の制御を弄っていたら、正しかったものを壊していた。**測ることは、間違った修理を防ぐ保険**だ。
3. **keyframe や初期化データは「番号(位置)」で決まる。構造(関節)を足したら、必ず作り直せ**。MuJoCo で関節を1本挿すことは、`qpos` の全スロットを1つずらすこと。位置ベースの初期化データを更新し忘れると、**テストは緑のまま、物体だけが静かに床へ落ちる**。そしてその静かなバグを暴いたのは、テストではなく、**絵を見た人間の違和感**だった。

最後の点が、このシリーズの背骨に一番近い。自動化された検証は「壊れていないこと」を保証するが、「**間違ったことを、正しく実行していないか**」は保証しない。ボトルは、地面の下で、正しく掴まれるのを待っていた。それに気づいたのは、数字ではなく、動画を見て「これは飲んでいない」と言った人間だった。**人間をループに残す(Human-in-the-Loop)**とは、こういう瞬間のためにある。

## 付録: 方法(再現用メモ)

数字を条件抜きで出すのは不誠実なので、前提を並べておく。

- **モデル**: MS-Human-700 操作用変種(Apache-2.0)/ MuJoCo。頭部の顎ジオメトリを子ボディ `mandible` + ヒンジ `jaw_open`(軸 z、設計可動域 −0.7〜0.05 rad)に分離した派生モデル。自由度 48→49、筋肉 81 は不変。ただし描画時の顎開けは設計可動域を超える +0.6 rad まで qpos 直書き(リミット非適用)で、これは演出。
- **制御**: 筋シナジー方策(K=8、観測22次元=腕7関節の角度・速度14 + 手→ボトル3 + 位相2 + 命令3、パラメータ913個)。前回の把持 champion から warm-start して唇目標で CMA-ES 進化。
- **バグと修正**: 顎ヒンジは `qpos` の42番目に入り、以降のスロットを1つずらす。keyframe が48個のまま(位置ベース)だったため、MuJoCo が末尾を0埋め→42番以降が全て1つずれ、ボトルの上下スライド欄(机の高さ 0.94m からの相対変位)に回転角 −1.57 が入り z=0.94−1.57=−0.63m に落下。修正=元の48個 keyframe の42番目に `0.0` を挿入し49個に。末尾の実値で見ると:

```python
import numpy as np
# orig_key48 = 元モデルの keyframe(48 値)
broken = np.concatenate([orig_key48, [0.0]])   # 末尾に 0 を足しただけ = MuJoCo の自動 padding 相当(壊れる)
fixed  = np.insert(orig_key48, 42, 0.0)         # 42 番目(0-index)に顎の 0 を挿入 = 正しい(49 値)

# 末尾(ボトル由来のスロット)の並び:
#   壊れた: ... 0.3141  0 0 0  -1.57 0.32 0        ← -1.57 が高さ欄に入る
#   直した: ... 0.3141  0 0 0 0  -1.57 0.32 0      ← 0 が1個ずれて -1.57 は本来の回転欄へ

# reset 直後にボトルの世界 z を測ると:
#   broken -> z = -0.63(床下)   /   fixed -> z = 0.94(机の上)= 元モデルと一致
```

  修正後、顎モデルは元モデルと手→ボトルベクトルまで一致(差 0.0、測定精度 1e-6 未満)。
- **数値の来歴**: 元/顎モデルの reset 直後の `qpos`・世界座標の直接比較、および学習済み方策の決定的な再実行(4秒ロールアウト、134フレーム、口への最接近 0.031m、最終の顎開度 0.6 rad)から取得。すべてノート PC の CPU。この記事の全定量主張は、公開前に**独立の別プロセス(3系列)で一次データ・実コードへ再照合**し、全 CONFIRMED を確認した。
- **正直な区別**: 腕の把持・運搬は 81 筋の閉ループ制御。顎の開閉は kinematic(台本)で、顔面筋非搭載ゆえの演出。

## 12. 次回に続く

飲むところまで来たので、次は**箸で食べる**——食物に見立てた物を摘んで口へ運ぶ動作を、この解剖学的な手に載せたい(以前の記事で「手が無い」と宿題に残した箸の話を、本物の手で回収する)。そしてもう一つ、飲む動作の**命令追従を強める**多目標訓練——「口へ」だけでなく「任意の一点へ厳密に」を、単一シナジーの容量の壁ごと測りに行く。

> この記事の数値は、乱数種を固定した再現可能な CPU 実行から取った。良い結果ほど内訳を疑い、詰まったら推測でなく測る——それが、私がこのシリーズで守っている一点だ。今回は「飲めた」の裏に、床下 1.57m のボトルが隠れていた。
