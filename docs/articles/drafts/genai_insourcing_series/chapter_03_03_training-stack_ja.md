# 第3回 GPUで学習を回す — フレームワークと分散の実務

> 連載「**生成AIを内製する 〜自宅から世界モデルへ〜**」の第3回です。第1回では「何を学習させるか(データ)」、第2回では「何を学習するか(アーキテクチャ)」を扱いました。今回はいよいよ、その2つを **実際に GPU の上で動かして損失を下げていく** 段階 — **学習スタック(training stack)** の実務に踏み込みます。

## この回で得られること(なぜ読むと得か)

前回まで、私たちは机の上で設計図を眺めていました。データを集め、Transformer の数式を確認し、「これで学習できるはずだ」というところまで来ました。ところが、いざ `python train.py` と叩いた瞬間に、多くの人が同じ壁にぶつかります。

- **`CUDA out of memory`** — メモリが足りずに 1 ステップも進まない
- **`loss: nan`** — 数時間回した損失が突然跳ね上がり、二度と戻ってこない

この2つは、モデル定義そのもののバグではありません。**モデルを「どう回すか」の作法** に属する問題です。そして生成AIの内製という営みでは、実のところ **この作法の巧拙が成否を大きく左右します**。少なくとも私の実感では、設計図(モデル)の美しさよりも、燃料系や冷却系(学習スタック)を組み切れるかどうかで、車が走るか一メートルも動かないかが決まってきました。

この回を読み終えると、次のことが「桁感」で分かるようになります。

- なぜ 70億パラメータのモデルが 80GB の GPU に載らないのか、その内訳
- 足りないメモリを削る **標準工具の一覧と、効く順番・副作用**
- 複数 GPU をどの軸で割るか(データ/テンソル/パイプライン/文脈の4並列)
- 何日も発散させずに回すための安定化の作法
- そして最も大事な **「自宅で作れる領域」と「作れない領域」の正直な分水嶺**

抽象論は最小限にします。すべて `git clone` して手元で回せる実在のツール名と、フェルミ推定の桁で語ります。それでは配管工事を始めましょう。

## 用語ミニ辞書

先に言葉の地図を渡します。初めて見る語もここに戻れば迷いません。

- **混合精度学習(mixed precision training)**: 順伝播・逆伝播の計算を 16bit で行い、重みの累積(master copy)だけは 32bit で保つ手法。速度とメモリを稼ぎつつ、数値誤差の蓄積を抑えます。
- **bf16 / fp16**: どちらも 16bit 浮動小数点です。**bf16(Brain Floating Point 16)** は指数部が 8bit で、fp32 と同じ広さの表現範囲(ダイナミックレンジ)を持ちますが、仮数部が 7bit と粗いのが特徴です。**fp16(half precision, 半精度)** は指数部 5bit・仮数部 10bit で、精度は高いものの表現範囲が狭く、桁あふれ対策(loss scaling)が必要になります。
- **勾配チェックポイント(gradient checkpointing / activation recomputation, 活性再計算)**: 逆伝播のために取っておく中間活性(activation)を一度捨て、必要になったら再計算する技法。メモリを「時間」と交換します。
- **オプティマイザ状態(optimizer state)**: Adam 系の最適化手法が、パラメータごとに保持する 1次・2次モーメント。これがメモリ消費の主犯です。
- **ZeRO(Zero Redundancy Optimizer, ゼロ冗長オプティマイザ)**: DeepSpeed の分散手法。同じデータを全 GPU が重複して持つのをやめ、オプティマイザ状態・勾配・パラメータを段階的に分割(shard, シャード)します。
- **FSDP(Fully Sharded Data Parallel, 完全シャード化データ並列)**: PyTorch 純正の同種手法。ZeRO の Stage-3(パラメータまで割る)に相当する分割を標準化したものです。
- **LoRA(Low-Rank Adaptation, 低ランク適応)/ QLoRA(Quantized LoRA, 量子化LoRA)**: 巨大な重み行列の「更新分」を、小さな低ランク行列の積で近似する省メモリ微調整。QLoRA はさらに土台モデルを 4bit に量子化して削ります。
- **データ並列(DP)/ テンソル並列(TP)/ パイプライン並列(PP)/ 文脈並列(CP)**: モデルと計算を GPU 群にどの軸で割るか、の4通り。後の節で詳述します。
- **MFU(Model FLOPs Utilization, モデルFLOPs利用率)**: 理論ピーク FLOPS のうち、モデルの有効な計算に実際に使えた割合。実務における「燃費」指標です。
- **μP(muP, Maximal Update Parametrization, 最大更新パラメータ化)**: 小さいモデルで調整したハイパーパラメータ(HP)を、幅を広げた大モデルへそのまま転移させるための初期化・スケーリングの流儀です。

## 直感 — 学習スタックは「配管」である

まず、大づかみの感覚を体に入れましょう。

意外に思われるかもしれませんが、**モデル本体(Transformer)の数式は 1 ファイルに収まります**。Karpathy 氏の教育用実装 nanoGPT の `model.py` は、たった 300 行ほどです。ではモデルを学習させるコードの残りの膨大な部分は何をしているのか。**その大半は「配管」** です。

巨大な重みと、その勾配と、オプティマイザ状態という3つの大荷物を、限られた VRAM(GPU メモリ)にどう詰め込むか。それを複数の GPU にどう流すか。そして途中で破綻(発散)させずに、何日も回し続けるか。学習スタックとは、この配管工事の総称です。

比喩を続けます。

> モデルは「エンジン」。学習スタックは「燃料系・冷却系・変速機」です。エンジン単体がどれほど美しくても、燃料が溢れれば(OOM: Out Of Memory)止まりますし、回しすぎればオーバーヒート(loss spike で発散)します。

内製で最初にぶつかる壁は、ほぼ必ず **「VRAM が足りない」** と **「損失が突然跳ねて戻らない」** の2つです。本節以降は、この2つの壁に対する標準工具のカタログだと思ってください。

### メモリの内訳を「桁」で握る

以後すべての見通しをよくするために、まず **1 パラメータあたり何バイト必要なのか** を体に入れます。混合精度 + AdamW という最も標準的な構成で、状態は次のように積み上がります(出所: ZeRO, Rajbhandari+ 2019 の内訳)。

| 保持するもの | バイト数 |
|---|---|
| bf16/fp16 の重み | 2 byte |
| bf16/fp16 の勾配 | 2 byte |
| fp32 マスター重み | 4 byte |
| Adam 1次モーメント $m$ | 4 byte |
| Adam 2次モーメント $v$ | 4 byte |
| **合計** | **16 byte/param** |

ここに、さらに活性(activation)のメモリが別途乗ります。

この「16 byte/param」を、7B(70億)モデルに当てはめてみましょう。

$$7\times10^9 \;\text{param} \times 16\ \text{byte} = 112\ \mathrm{GB}$$

**活性を除いた「静的な状態」だけで 112GB** です。単体 GPU の VRAM は 24〜80GB ですから、最初からまったく載りません。

> 図: 16 byte/param の内訳を積み木で示し、7B で 112GB に届く様子。単体 GPU(24/80GB)のラインを重ねて「そもそも載らない」を一目で。

この一行の不等式こそが、これから紹介するすべての省メモリ・分散手法が存在する理由です。逆に言えば、**「16 byte をどう削るか」** さえ押さえれば、無数に見える技法はきれいに整理できます。

## フレームワークの選び分け — 「回すコード」から逆算する

道具の話に入ります。抽象的なカタログではなく、実際に `git clone` して手元で回せる代表を、**目的別の梯子** として並べます。上ほど「学習・理解」向き、下ほど「本番・大規模」向きです。

| フレームワーク | 作者/主体 | 立ち位置 | 向く用途 |
|---|---|---|---|
| **nanoGPT** | Karpathy (2022–23) | 教育用最小実装(PyTorch, ~300行) | 仕組みを読む / 小モデル事前学習の起点 |
| **llm.c** | Karpathy (2024) | 純 C/CUDA の GPT-2/3 学習(依存ゼロ級) | 効率の底を知る / 第一原理の理解 |
| **litgpt** | Lightning AI | 実用寄りの単一/少数ノード学習・微調整 | 20〜30 種のモデルを素直に回す |
| **torchtitan** | PyTorch 公式 (2024) | PyTorch ネイティブの production 事前学習 | FSDP2/TP/PP を純正 API で組む |
| **Megatron-LM** | NVIDIA | TP+PP の元祖・大規模の基準 | 数十〜数千 GPU の事前学習 |
| **GPT-NeoX** | EleutherAI | Megatron+DeepSpeed 融合の OSS | 大規模 OSS 事前学習(Pythia 等) |
| **HF Trainer + Accelerate** | Hugging Face | 汎用の学習ループ抽象化 | 微調整全般 / エコシステム連携 |
| **Axolotl** | OSS コミュニティ | 微調整の設定ファイル駆動ラッパ | LoRA/QLoRA/SFT を YAML で量産 |

### 選び分けの第一原理は「規模 × 目的」の2軸

道具が多くて迷ったら、この2軸だけで切り分けられます。

**理解したい / 小さく事前学習したい(〜数百 M param、1〜8 GPU)** なら、**nanoGPT か llm.c** です。特筆すべきは、Karpathy 氏が llm.c で **GPT-2(124M)を 8×A100 80GB・約 90 分・およそ \$20 で再現** したことです(出所: karpathy/llm.c Discussion #481, 2024。FineWeb データ、MFU は約 60% 前後。金額は当時の A100 クラウド単価を前提とした桁感で、要検証)。「事前学習など雲の上の話」という思い込みを、体で壊してくれる最良の教材です。

**素直に色々な既存モデルを事前学習/微調整したい(1〜数十 GPU)** なら、**litgpt か torchtitan**。torchtitan は **FSDP2 を既定の 1D 並列** とし、そこに TP+SP や PP を純正 API で積んで 3D/4D 並列を組みます(出所: TorchTitan, arXiv:2410.06511, 2024)。「PyTorch の正史に沿って」大規模の作法を学べるのが強みです。

**とにかく大きく事前学習したい(数十 GPU 以上)** なら、**Megatron-LM / GPT-NeoX**。TP と PP の実装が枯れていて、MFU を最後まで絞り出す最適化が詰まっています。

**微調整が主目的** なら、汎用の **HF Trainer + Accelerate**、あるいは設定駆動で LoRA/QLoRA を量産できる **Axolotl**。研究の試行錯誤なら前者、決まったレシピを運用で回すなら後者、というのが実務感覚です。

> **内製の現実解**: 自宅〜小研究室(GPU 1〜8 枚)であれば、事前学習は **nanoGPT / litgpt / torchtitan**、微調整は **Axolotl / HF+PEFT(Parameter-Efficient Fine-Tuning, パラメータ効率的微調整)** の組み合わせが、ほぼ回答になります。Megatron/NeoX は「本当に必要になってから」で遅くありません。最初から重装備を選ぶのは、たいてい遠回りです。

## メモリ削減 — 16 byte/param をどう削るか

いよいよ最初の壁「VRAM 不足」への対処です。先ほどの 16 byte/param を、**副作用を理解したうえで、効く順・副作用の軽い順に** 削っていきます。ここは順番が命です。

### (1) 混合精度(bf16 が第一選択)

順伝播・逆伝播を bf16 で行い、重みの累積だけは fp32 に残します(出所: Mixed Precision Training, Micikevicius+ 2017)。行列積が Tensor Core 上で 2〜8 倍速くなり、活性メモリも半分になります。

ここで **bf16 と fp16 の使い分けが、実務の分かれ目** になります。bf16 は fp32 と同じ指数レンジを持つので、**loss scaling(勾配のアンダーフローを避けるためのスケール操作)が不要** です。この「不要」が、安定性の面で決定的に楽なのです。一方の fp16 は仮数部が広く精度こそ上ですが、表現範囲が狭くて発散しやすい。

結論として、**A100/H100 世代以降は原則 bf16**。V100 など bf16 非対応の旧世代でのみ、fp16 + loss scaling を使う、と覚えてください。

### (2) 勾配チェックポイント(活性再計算)

中間活性を保存せず、逆伝播のときに再計算します(出所: Training Deep Nets with Sublinear Memory Cost, Chen+ 2016)。活性メモリは層数 $L$ に対して $O(L) \to O(\sqrt{L})$ に落ちます。

代償は、順伝播をもう一度やる分の計算増です。実測でおおむね **学習時間 +20〜30%(桁感、要検証)**。長い系列長や大きなバッチで OOM するときの、定番の逃げ道です。

### (3) 8bit オプティマイザ

Adam のモーメント $m, v$ を、ブロック単位の量子化で 8bit に圧縮します(出所: 8-bit Optimizers via Block-wise Quantization, Dettmers+ 2021、実装は `bitsandbytes`)。オプティマイザ状態(モーメント2種)が $8\ \mathrm{byte} \to 2\ \mathrm{byte/param}$ に落ち、16 byte/param のうち **6 byte を削れます**。精度の劣化はほぼ無視できるという報告が多く、単体 GPU 微調整の常備品です。

### (4) ZeRO / FSDP のシャーディング(分散の話と地続き)

「重複保持をやめる」段階的な分割です。ここは次節の分散並列と地続きになります。原則は **「Stage を上げるほどメモリは減るが、通信は増える」** というトレードオフ。これを理解して選びます。

なお ZeRO の数え方では、「オプティマイザ状態」= fp32 マスター重み 4 + モーメント 4+4 = **12 byte/param**、fp16 の重み+勾配が 4 byte、合わせて 16 byte/param、と数えます。

- **ZeRO-1**: オプティマイザ状態(12 byte)だけを $N$ 台に分割 → $12 \to 12/N\ \mathrm{byte/param}$。追加通信は軽微です。
- **ZeRO-2**: 上に加えて勾配も分割。逆伝播で reduce-scatter が入ります。
- **ZeRO-3(= FSDP フル分割)**: パラメータ本体まで分割。前向き・後向きの各層で「all-gather → 使う → 破棄」を繰り返します。$N$ 台なら理論上 **1 台あたり $16/N\ \mathrm{byte/param}$** まで落ちますが、層ごとの all-gather 通信が乗るため、**相互接続(NVLink / InfiniBand)の帯域が律速** になります。
- **ZeRO-Offload / -Infinity**: 状態を CPU RAM や NVMe へ退避します。VRAM は劇的に空きますが、PCIe 帯域が首を絞めて **著しく遅くなります**。「動かないよりはマシ」の最後の手段です。

### (5) QLoRA(微調整の最終兵器)

土台モデルを **4bit(NF4: 4-bit NormalFloat)量子化** して凍結し、学習は低ランクアダプタ(LoRA)だけに限定します(出所: QLoRA, Dettmers+ 2023)。土台 7B が $7\times10^9 \times 0.5\ \mathrm{byte} \approx 3.5\ \mathrm{GB}$、勾配やオプティマイザ状態は極小のアダプタ分だけで済みます。

原論文の看板は **「65B の微調整を単一 48GB GPU に載せた」** こと。二重量子化(double quantization)と paged optimizer(OOM 時に状態を CPU へ退避する仕組み)を併用します。

### メモリ削減の一望

7B モデルを例に、桁感でまとめます(各行は「静的状態のみ」で、活性は別途乗ります)。

| 手法 | 1 param あたり | 7B での状態メモリ | 主な代償 |
|---|---|---|---|
| fp32 素朴 Adam | ~16 byte* | ~112 GB | — |
| bf16 混合精度 + Adam | 16 byte | 112 GB | (基準) |
| + 8bit optimizer | ~10 byte | ~70 GB | ほぼ無し |
| + FSDP/ZeRO-3(8台) | ~2 byte | ~14 GB | 通信増 |
| LoRA(全微調整比) | 土台 14GB + 微小 | ~15 GB | 表現力に上限 |
| QLoRA(4bit 土台) | 土台 3.5GB + 微小 | ~5–10 GB† | 量子化誤差 / やや遅い |

<small>\* fp32 素朴実装は数え方で 12〜16 byte と揺れます。ZeRO-3 の行は基準 16 byte を 8 分割した桁感であり、8bit optimizer の行と単純加算した値ではありません。† 活性込みの実効値で、系列長・バッチに依存します(要検証)。</small>

この表を眺めると、**112GB → 5〜10GB** という2桁近い圧縮が、いくつかの標準工具の重ね掛けだけで到達できることが見て取れます。「魔法」ではなく「配管の積層」なのです。

## 分散並列 — モデルと計算をどの軸で割るか

GPU が2枚以上になった瞬間、「何を」「どの軸で」割るかという新しい問題が生まれます。ここには **4つの直交する軸** があり、大規模学習ではこれらを **積層(3D/4D 並列)** して使います。ひとつずつ直感を掴みましょう。

> 図: 1つの Transformer を、DP(横に複製)/ TP(1層を輪切り)/ PP(層をブロックで縦割り)/ CP(系列を分割)の4方向に割る様子を並べ、それぞれの通信が走る場所を矢印で。

### (a) データ並列(Data Parallelism, DP)

モデル全体を各 GPU が複製し、バッチだけを分割します。各 GPU が自分の分の勾配を計算し、all-reduce で平均を取る。**最も単純で、最初に使う軸** です。

素朴な DP は「モデルが 1 GPU に載る」前提です。載らないなら、ZeRO/FSDP でパラメータを割る「シャード付き DP」に切り替えます。通信は勾配の all-reduce($O(\text{param})$/step)です。

### (b) テンソル並列(Tensor Parallelism, TP)

1つの層の重み行列を、GPU 間で横に割ります(出所: Megatron-LM, Shoeybi+ 2019)。行列積 $Y = XW$ の $W$ を列方向に分割し、各 GPU が部分積を計算して all-reduce で合成する。

ポイントは、**各層の内部で all-reduce が走る** こと。通信が非常に重いので、**基本的にノード内(NVLink で繋がった 2〜8 GPU)に閉じ込めます**。単体モデルが 1 GPU に載らないときの第一手です。

対になる技法として **Sequence Parallelism(SP, 系列並列)** があります。TP が扱わない正規化・dropout などの部分の活性を系列方向に分割し、活性メモリをさらに削ります(出所: Reducing Activation Recomputation…, Korthikanti+ 2022)。

### (c) パイプライン並列(Pipeline Parallelism, PP)

モデルを層の塊(stage)に縦割りし、各 GPU が数層ずつ担当します。マイクロバッチを次々に流し込んでパイプラインを埋める方式です。

**通信は stage の境界だけ** なので軽く、**ノード間(InfiniBand)を跨ぐのに向いています**。代償は **パイプラインバブル(pipeline bubble)** — 立ち上がり/立ち下がりで一部の GPU が遊んでしまう空白時間です。GPipe や 1F1B(one-forward-one-backward)というスケジューリングで、このバブルを圧縮します。

### (d) 文脈並列 / 系列並列(Context Parallelism, CP)

長い系列そのものを GPU 間で分割します。Attention は全トークン間の相互作用なので単純には割れませんが、**Ring Attention**(出所: Liu+ 2023)は Key/Value ブロックを GPU のリング上で回しながら部分 attention を積算し、超長文脈(数十万〜100万トークン)を可能にします。長文脈が要件でなければ、通常は不要な軸です。

### 積層の設計指針(第一原理)

4軸をどう組むか。原則は明快です。

> **通信コストの重い軸ほど、帯域の太いリンクに割り当てる。**

実務の並べ方は、**TP はノード内(NVLink)、DP/PP はノード間(InfiniBand)、CP は長文脈が要るときだけ**。torchtitan や Megatron は、この割り当てを意識した mesh(デバイス格子)の設定を持っています。

### 1〜数 GPU の現実的構成

自宅・小研究室のスケールに引き寄せて、具体的な処方箋を書きます。

- **1 GPU**: 並列はありません。混合精度 + 勾配チェックポイント + 8bit optimizer(+ 必要なら QLoRA)で、とにかく「載せる」ことに集中します。
- **2〜8 GPU(1ノード、NVLink あり)**: **FSDP(ZeRO-3)を基本線** に。モデルが 1 枚に載るなら、素朴 DP + ZeRO-1/2 で十分速いです。載らない大モデルの微調整なら FSDP、事前学習で層が大きいなら TP をノード内で併用します。
- **注意(自宅マルチGPUの落とし穴)**: **コンシューマ GPU(RTX 4090 など)は NVLink 非対応** です(3090 は対応していました)。GPU 間通信が PCIe 経由になり、TP/FSDP の効率が大きく落ちます。自宅でマルチ GPU を組むなら、**「NVLink の有無」がスタック選択を左右する現実的な制約** になります(要検証: 世代ごとの対応状況)。

## 安定化 — 何日も発散させずに回す

もう1つの壁、「損失が突然跳ねて戻らない」への対処です。大規模学習の失敗の大半は、OOM か **発散(divergence)** のどちらかです。損失が突然跳ね上がって戻らない **loss spike** を、どう防ぎ、起きたらどう復旧するか。ここが安定化の核心です。

### (1) 学習率スケジュール: warmup + cosine decay

学習の初期は、重みが乱雑で勾配も荒れています。ここでいきなり本命の学習率をかけると発散します。そこで **warmup**(学習率を 0 から線形に立ち上げる)→ **cosine decay**(徐々に減衰させる)が、事実上の標準です(GPT-3, PaLM 等で採用)。

桁感として、warmup は総ステップの数百〜数千 step、あるいは **総トークンの 0.1〜1% 程度** が目安です(モデル規模で変わります、要検証)。

### (2) 勾配クリッピング(gradient clipping)

全パラメータの勾配ノルムを、閾値で頭打ちにします。**global norm 1.0** が広く使われる定番値です。単発の異常な勾配が重みを吹き飛ばすのを防ぐ、**最も費用対効果の高い安全弁** です。1行入れるだけで効くので、外す理由がありません。

### (3) loss spike への対処

正直に言えば、**発散は完全には予防できません**。だから大規模学習の実運用は、次のような多層防御を敷きます(出所: PaLM, Chowdhery+ 2022 が「spike 直前のチェックポイントに戻し、問題バッチをスキップして再開する」運用を報告しています)。

- **こまめなチェックポイント**(数百〜数千 step ごと)。spike 発生時に **直前の健全な checkpoint へ巻き戻す**。
- **問題バッチのスキップ**: spike を誘発したデータバッチを飛ばして再開する。
- **予防的設計**: bf16 を採用する(fp16 の loss scaling 起因の不安定を避ける)、埋め込みの学習率を下げる、QK-LayerNorm など正規化を追加する。

### (4) 初期化と μP

初期化を間違えると、学習が始まらないか、すぐに発散します。標準は、各層の分散を保つスケール初期化です(Transformer では、残差経路を層数でスケールする流儀が一般的)。

その一歩先が **μP(Maximal Update Parametrization)** です。これは **「小モデルで探した最適なハイパーパラメータ(特に学習率)を、幅を広げた大モデルへそのまま転移」** できるよう、初期化・スケーリングを設計する流儀です(出所: Tensor Programs V / μTransfer, Yang+ 2022)。

μP の実務価値は、**コストの構造そのものを変える** 点にあります。素朴にやると、大モデルで HP 探索するには大モデルを何度も回す必要があり、探索コストが学習コストに比例して膨らみます。μP なら **「安い小モデルで探索 → 高い大モデルへ一発転移」** と、探索を計算の安い側に押し込めます。内製で「一発勝負の事前学習を絶対に外したくない」ときに効く道具です(ただし後述の限界があります)。

## 事前学習 vs ファインチューニング — スタックはどう変わるか

同じ「学習」という言葉でも、事前学習と微調整では、メモリ配分・データ規模・使う工具がまるで別物になります。ここを混同すると見積もりを大きく誤ります。

| 観点 | 事前学習(pretraining) | ファインチューニング(fine-tuning) |
|---|---|---|
| データ規模 | 数百 B〜数 T tokens | 数千〜数百万サンプル(桁が全く違う) |
| 学習パラメータ | 全パラメータ | 全 or 一部(LoRA なら 0.1〜1%) |
| メモリの主犯 | オプティマイザ状態 + 活性 | (全微調整)同左 /(LoRA)ほぼ土台のみ |
| 律速 | **計算量(FLOPs)** | しばしば **メモリ**(状態は小さいが土台が大きい) |
| 主フレームワーク | nanoGPT/torchtitan/Megatron | HF Trainer/Axolotl + PEFT |
| 安定化の重み | warmup/clip/spike 対策が生命線 | 過学習・破滅的忘却(catastrophic forgetting)対策 |

### 計算量のフェルミ推定(6ND 則)

事前学習のコストを「桁」で掴む、最も大事な経験則がこれです。事前学習の計算量は、

$$C \approx 6ND\ \text{FLOPs}$$

で近似できます(**$N$=パラメータ数、$D$=学習トークン数**。係数 6 は「順伝播 2 + 逆伝播 4」の、per-token・per-param の乗加算に由来。出所: Scaling Laws, Kaplan+ 2020)。トークン量の目安は、**Chinchilla 最適の $D \approx 20N$** です(出所: Hoffmann+ 2022)。

> 図: 6ND 則でコストの桁を掴む。$N$ と $D$ を軸に、GPT-2 124M と GPT-3 175B を対数目盛でプロットし、その間に横たわる「10 桁の谷」を可視化。

この式に、両極端を代入してみます。

**GPT-3 級(175B, 300B tokens)— 自宅では作れない領域**

$$C = 6 \times 175\times10^9 \times 300\times10^9 \approx 3.15\times10^{23}\ \text{FLOPs}$$

A100 の bf16 ピーク 312 TFLOPS に MFU 40% を噛ませると、有効速度は $1.25\times10^{14}$ FLOP/s。所要は $3.15\times10^{23} / 1.25\times10^{14} \approx 2.5\times10^9$ GPU 秒、すなわち **約 70 万 GPU 時間** です。

仮に 8×A100 を 24 時間フル稼働させても、月あたり $8\times24\times30 = 5{,}760$ GPU 時間。$700{,}000 / 5{,}760 \approx$ **約 120 か月(10 年級)** です。これは個人の到達域の外にあります。

**自宅で「作れる」領域(GPT-2 124M 級)**

Chinchilla 最適で $D \approx 20 \times 124\times10^6 \approx 2.5\times10^9$ tokens。

$$C = 6 \times 124\times10^6 \times 2.5\times10^9 \approx 1.9\times10^{18}\ \text{FLOPs}$$

RTX 4090(bf16 実効 165 TFLOPS × MFU 35% ≈ $5.8\times10^{13}$ FLOP/s)なら、$1.9\times10^{18}/5.8\times10^{13} \approx 3.3\times10^4$ 秒、つまり **約 9 時間** です(桁感で、MFU の仮定に強く依存するため要検証)。これは先ほどの llm.c の実測(124M を 8×A100・90 分)とも桁が整合します(8 枚で割れば 1 時間強)。

**分解と仮定の明示**(上のフェルミ推定):

- **分解**: 所要 GPU 時間 = 総 FLOPs ÷(GPU ピーク TFLOPS × MFU × 3600)。
- **仮定**: A100 bf16 = 312 TFLOPS dense(A100 80GB, HBM2e 2039 GB/s)/ H100 SXM ≈ 990 TFLOPS(A100 の約 3 倍・HBM3 3350 GB/s)/ MFU 35〜50%(よく調整して 40〜50%、家庭環境は下振れ)。RTX 4090 の 165 TFLOPS は fp32 累積時の実効値(fp16 累積なら約 2 倍だが、訓練は fp32 累積が普通)。
- **出所**: TFLOPS は NVIDIA データシート系、MFU は PaLM の 46% 報告等、6ND/20N は上記 Kaplan/Hoffmann。

### 微調整のメモリ帰結 — 分水嶺はここ

7B を全微調整すると、事前学習と同様に 112GB 級のメモリが要ります。ところが **QLoRA なら 5〜10GB 級**(前節の表)まで落ち、単体 24GB GPU に載ります。

**「事前学習は無理でも、微調整は自宅で回る」** — この分水嶺が、まさにここです。内製の戦略としては、**「土台は公開重みを使い、自分のドメインへ QLoRA/LoRA で寄せる」** が、費用対効果で圧倒的に現実的だと言えます。

## 誤解されやすい点・正直な限界

ここまでの道具を、誇張せずに使いこなすために、正直な注意書きを列挙します。ここを飛ばすと、見積もりも設計も楽観に振れます。

- **「並列にすれば比例して速くなる」は嘘です**。TP は各層で all-reduce、FSDP は層ごとに all-gather が走り、**GPU を増やすほど通信が増えます**。NVLink/InfiniBand の帯域が律速で、線形スケールはあくまで理想値です。特に **PCIe 接続のコンシューマ GPU 多枚差し** は、スケール効率が想像よりずっと悪くなります。

- **MFU は簡単には上がりません**。理論 312 TFLOPS でも、実効は MFU 40〜50% で **約半分** が現実です。「ピーク TFLOPS × 台数 × 時間」で見積もると、必ず楽観に振れます。見積もりには必ず MFU を噛ませてください。

- **bf16 は万能ではありません**。仮数 7bit の粗さが効いてしまう場面(小さな数の累積、特定の正規化)では、かえって不安定化しえます。実務では、**正規化・softmax・累積を部分的に fp32 に上げる混在** が普通です。「全部 bf16 で軽くしたら発散した」は典型的な罠です。

- **LoRA/QLoRA は全微調整の下位互換ではありませんが、上位でもありません**。低ランク近似ゆえに、**土台から大きく離れる学習(新しい言語、大幅なドメイン移動)には表現力が足りない** ことがあります。「知識の注入」より「振る舞いの調整・様式の適応」に強い、と割り切るのが正直な線です。QLoRA の 4bit 量子化誤差も、タスクによっては効いてきます(要検証)。

- **μP は銀の弾丸ではありません**。HP 転移は主に **幅(width)方向** で成り立つ理論で、深さ・バッチ・データ量を同時に変えるには追加の注意が要ります。実装も繊細で、正しく組めていないと転移が崩れます。効果は本物ですが、導入コストと検証が必要です。

- **loss spike は「予防」でなく「復旧」前提で設計します**。大規模ほど、完全な予防は不可能に近いです。チェックポイント頻度とロールバック手順を **最初から** 組み込むこと。spike が出てから設計を始めると、数日分の計算が溶けます。

- **自宅規模の切り分け(誇張しない)**:
  - **作れる** = 小モデル(〜数億 param)の事前学習、公開土台への LoRA/QLoRA 微調整、そして学習スタックの完全な理解。
  - **作れない** = 数十 B 以上のフルスクラッチ事前学習(70 万 GPU 時間級は個人の到達域外)。

内製の勝ち筋は「巨大事前学習の再現」ではありません。**小さく事前学習して原理を握り、微調整でドメイン価値を出す** 方向にあります。ここを見誤らないことが、限られた計算資源を活かす唯一の道だと考えています。

## 本節の要点

学習スタックは配管であり、実務は結局この3つに収斂します。

1. **16 byte/param をどう削るか**(混合精度 → 勾配チェックポイント → 8bit optimizer → FSDP/ZeRO → QLoRA)
2. **どの軸で割るか**(DP / TP / PP / CP を帯域の太さに合わせて積層する)
3. **どう発散させずに回すか**(warmup+cosine、grad clip、チェックポイント巻き戻し、μP)

そして、事前学習と微調整では律速が **計算 ⇄ メモリ** で入れ替わり、自宅規模では **微調整とスモール事前学習が現実的な主戦場** になります。
