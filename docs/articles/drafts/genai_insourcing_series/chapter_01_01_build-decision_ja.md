# 第1回 どこまで自分で作るか — from-scratch から RAG までの地図

> 連載「**生成AIを内製する 〜自宅から世界モデルへ〜**」第1回

## はじめに — 「自分で作る」の前に、地図を広げましょう

「生成AIを内製したい」——そう言葉にした瞬間、多くの方の頭には *ゼロからモデルを訓練する自分* の姿が浮かぶのではないでしょうか。GPU をうならせ、乱数から立ち上がった重みが少しずつ言葉を覚えていく。ロマンのある絵です。ただ、そのロマンにいきなり飛び込むと、たいていは数千 GPU 時間と数十万ドルの請求書という現実に頭を打ちます。

この連載「**生成AIを内製する 〜自宅から世界モデルへ〜**」は、その名のとおり「自宅の机の上にある PC 一台から、少しずつ世界モデルへ手を伸ばしていく」道のりを、正直な桁感つきで一緒に歩く企画です。派手な成功譚ではなく、**どこにお金と時間が消えるのか**、**どこは自宅で本当に作れて、どこは借りても割に合わないのか**を、フェルミ推定(概算による桁の見積もり)で毎回はっきりさせていきます。

その第一歩が、この地図です。

内製と一口に言っても、実は **from-scratch(ゼロからの事前学習)/ 継続事前学習 / SFT / 蒸留 / RAG** という、握る深さの違う複数の選択肢が連続的に並んでいます。ここを混同したまま突き進むと、「SFT で新しい事実を覚えさせようとして幻覚が悪化する」「無関係な人気モデル同士を平均して壊す」といった、原理的に避けられた失敗に何日も溶かすことになります。

この回を最後まで読むと、次の3つが手に入ります。

- **握力のダイヤル**という一本の軸で、5つの選択肢を頭の中で並べ替えられるようになる
- **6ND 則**というたった一つの式で、どの選択肢がいくらかかるのかを桁で暗算できるようになる
- 「独自 LLM が欲しい」という願望の**大半は from-scratch を必要としない**、という費用対効果の急所を、正直な数字とともに腹落ちさせられる

コードはまだ一行も書きません。けれど、この地図を持たずに走り出すと、あとで一番高くつきます。まずは全体像から握っていきましょう。

## この回で使う言葉(用語ミニ辞書)

本文に入る前に、繰り返し出てくる言葉を先に並べておきます。読み進めて迷ったら、ここに戻ってきてください。

- **内製(in-house / from-scratch)**: モデルの重みや振る舞いを「自分の手に握る」度合い。本稿では *0か1* の二択ではなく、**連続体(グラデーション)**として扱います。
- **事前学習(pre-training)**: ラベルなしの大規模テキストで「次のトークン」をひたすら予測させて学ぶ工程。ここでモデルの**世界知識と言語能力の大半**が形づくられます。
- **from-scratch(ゼロからの事前学習)**: 乱数で初期化した重みから事前学習を回すこと。最も重い「握り方」です。
- **継続事前学習(continued pretraining)** / **ドメイン適応事前学習(Domain-Adaptive PreTraining, DAPT)**: 既存の base モデルの重みを起点に、対象ドメインの生テキストで事前学習を追加する手法(Gururangan et al. 2020「Don't Stop Pretraining」)。
- **教師ありファインチューニング(Supervised Fine-Tuning, SFT)**: 「入力 → 望ましい出力」の対を見せて、**振る舞い**(指示追従・出力形式・文体)を教える工程。
- **低ランク適応(Low-Rank Adaptation, LoRA)**: 元の重み $W$ を凍結したまま、低ランクの差分 $\Delta W = BA$($B\in\mathbb{R}^{d\times r},\ A\in\mathbb{R}^{r\times k},\ r\ll d$)だけを学習する手法(Hu et al. 2021)。
- **量子化 LoRA(Quantized LoRA, QLoRA)**: base を 4bit に量子化した上に LoRA を載せ、**単一 GPU で大型 base を微調整**できるようにした手法(Dettmers et al. 2023)。
- **知識蒸留(Knowledge Distillation)**: 大きな教師モデルの出力(確率分布=ソフトターゲット、または生成テキスト)を、小さな生徒モデルに写し取ること(Hinton et al. 2015)。
- **検索拡張生成(Retrieval-Augmented Generation, RAG)**: モデルの重みには一切触れず、**推論時**に外部知識を検索して文脈へ差し込む手法(Lewis et al. 2020)。
- **スケーリング則(scaling laws)**: 損失がパラメータ数 $N$・データ量 $D$・計算量 $C$ の冪乗で決まる、という経験則(Kaplan et al. 2020 / Hoffmann et al. 2022、通称 **Chinchilla**)。
- **モデルマージ(model merging)**: 複数モデルの重みを合成して 1 つにする操作(model soups, task arithmetic, TIES など)。**同一 base 前提**という強い制約があります(後述)。
- **base モデル**: 事前学習だけを済ませた素の重み。SFT・蒸留・マージの**共通祖先**になります。
- **浮動小数点演算(floating point operations, FLOPs)**: 学習・推論の計算量を測る単位。掛け算や足し算 1 回を 1 FLOP と数えます。後述の 6ND 則は、この FLOPs で学習コストの桁を掴む式です。

略語を一つの語呂で覚えるなら——**「RAG は棚、SFT は作法、蒸留は写し、継続 PT は増改築、from-scratch は更地」**。この後の家の比喩がそのまま対応します。

## 直感 — 「握力」のダイヤル

まず一番大事な感覚から入ります。「内製する」は on/off のスイッチではありません。**どこまで自分の手で握るかを回す、無段階のダイヤル**だと考えてください。

家に例えると、驚くほどきれいに並びます。

- **RAG = 本棚を置く。** 家(モデル)は借り物のまま。必要な資料を、そのつど棚から抜いて机に広げるイメージです。家の構造には一切触れません。
- **SFT + LoRA/QLoRA = 内装リフォーム。** 骨組み(base が持つ知識)はそのまま。応答の作法・口調・出力形式という「暮らし方」だけを上書きします。
- **蒸留 = 大邸宅の間取りを小さな家に写す。** 教師モデルの判断パターンを、生徒サイズの家に移築する作業です。
- **継続事前学習 = 増改築。** 基礎(base)を活かしつつ、専門ドメインの部屋を新しく建て増しします。**世界知識そのものに手を入れられる最初の段**です。
- **from-scratch = 更地から建てる。** すべての柱と配線を自分で敷く自由と引き換えに、莫大な資材(データ)と工期(GPU 時間)を丸ごと背負います。

ここで腑に落ちてほしい単調なトレードオフが一つあります。**右に行くほど「握れる自由度」は増えますが、握った瞬間に責任(データ品質・計算・評価・保守)も丸ごと自分に移ってくる**、ということです。自由と責任がセットで動く。ここを直感として持っておくだけで、判断がぶれなくなります。

そして先に結論の肌感を言ってしまうと、**多くの実務では RAG と SFT の二枚で要件の 8〜9 割が埋まります**(筆者の感覚値であり、案件によります)。図式化すると、次のような勾配になります。

```
知識だけ外付け ────────────────────────────► 重みを全部握る
  RAG  <  SFT+LoRA  <  蒸留  <  継続事前学習  <  from-scratch
 触る重み: 0      0.1〜1%     全体        全体          全体(乱数初期化)
 律速:  検索精度   データ品質   教師推論   GPU×トークン   GPU×トークン(桁違い)
```

> 図: 握力のダイヤル。左端(RAG)は重みに触れず、右端(from-scratch)は乱数から全部握る。右に行くほど「自由度」と「責任・コスト」が同時に増える一本道であることを示す。

一番下の「律速(ボトルネック)」の行に注目してください。左に行くほど勝負どころは**検索の精度やデータの品質**で、右に行くほど**GPU × トークン数**、つまり生の計算量になります。この「右側は計算がボトルネック」という事実が、次に登場するたった一つの式ですべて桁で読めてしまう——そこが今回の山場です。

## 原理と実務 — 地図に数字を入れていく

### 連続体を「何を制御できるか」で分解する

比喩は直感を与えてくれますが、設計判断には「で、具体的に何が変わって、何が変わらないのか」の表が要ります。5つのレベルを、**学習する重み / 変えられるもの / 変えにくいもの**で分解します。

| レベル | 学習する重み | 変えられるもの | 変えにくいもの |
|---|---|---|---|
| RAG | なし | 参照する事実・鮮度・出典 | 推論様式・文体・言語能力 |
| SFT + LoRA/QLoRA | 0.1〜1%(アダプタ) | 出力形式・指示追従・口調・軽いスキル | 素の世界知識(ほぼ増えない) |
| 蒸留(応答蒸留型) | 全体 or アダプタ | 教師の判断・推論パターンの模倣 | 教師を超える能力(通常は漸近。例外あり) |
| 継続事前学習 / DAPT | 全体 | ドメイン語彙・専門知識・分布シフト | アーキテクチャ・トークナイザ |
| from-scratch | 全体(乱数初期化) | 全て(語彙・構造・データ選択) | —(代わりに全責任を負う) |

この表から取り出してほしい**設計上の核心**は、たった一行です。

> **SFT は「振る舞い」を、事前学習(継続を含む)は「知識」を担う。**

役割分担がきれいに分かれています。この分業を取り違えて、**SFT で新しい事実を教え込もうとすると失敗します**。後半の「誤解されやすい点」で、なぜそうなるのかを詳しく見ます。まずは「振る舞いの担当」と「知識の担当」は別部署なのだ、と覚えておいてください。

### コストの第一原理 — 6ND 則

さて、今回の主役の式です。地図の右側(重みを握る側)のコストは、驚くほど単純な一本の近似で桁が読めます。

Transformer の学習計算量(単位は浮動小数点演算数、FLOPs)は、順伝播(forward)がおよそ $2ND$、逆伝播(backward)がその 2 倍で、合計すると次のように近似できます。

$$C \approx 6 N D \quad [\text{FLOPs}]$$

ここで $N$ はパラメータ数、$D$ は学習トークン数、係数の **6** は経験則として広く使われる値です(Kaplan et al. 2020 / Hoffmann et al. 2022)。パラメータ 1 個・トークン 1 個あたり、学習には約 6 回の浮動小数点演算がかかる——たったそれだけの話です。

これを **GPU 時間**に換算するには、GPU の実効速度で割ります。

$$T_{\text{GPU-hour}} = \frac{6ND}{P_{\text{peak}}\cdot u \cdot 3600}$$

フェルミ推定なので、**仮定と出所を明示**します。

- $P_{\text{peak}}$(A100, BF16 密行列)$= 312\ \text{TFLOPS} = 3.12\times10^{14}$ FLOPs/s(出所: NVIDIA A100 データシート)
- $u$ = モデル FLOPs 利用率(Model FLOPs Utilization, MFU)$\approx 0.4$。実測では 30〜55% あたりに散らばり、PaLM の報告値 46% 前後を中央にとって丸めた値です(要検証: ハード・実装・並列化構成で大きく振れます)。
- したがって**実効速度 $\approx 1.25\times10^{14}$ FLOPs/s**。

数字を一つ暗記するなら、この **「A100 一枚 ≒ 毎秒 $1.25\times10^{14}$ 回」** です。ここから、あらゆる学習の桁がはじけます。

**検算(手法の妥当性確認)** をしておきましょう。式が信用できるかは、既知の実績に当てて一致するかで決まります。Llama-2-7B($N=7\times10^9$、$D=2\times10^{12}$ トークン)を当てはめると、

$$C = 6\times 7\times10^9 \times 2\times10^{12} = 8.4\times10^{22}\ \text{FLOPs} \Rightarrow \approx 1.87\times10^{5}\ \text{A100-hour}$$

これは Llama 2 論文の報告値 **184,320 A100-hour**(Touvron et al. 2023)とほぼ一致します。桁がずれていないどころか、有効数字レベルで合ってしまう。**式が実データで校正できた**ので、以降はこの式を信用して各レベルの桁感を出していきます。異常なほどよく合ったときこそ疑うべきですが、ここは係数 6 と MFU の見積もりが素直に効いた、健全な一致だと考えています(裏を返せば、この一致は「Llama 2 の実効 MFU がたまたま 0.4 前後だった」ことにも支えられています。仮定が効いている点は忘れないでおきましょう)。

### レベル別コスト早見(桁感、A100 換算)

校正済みの 6ND 則を、5つのレベルに順に当てていきます。**あくまで桁の見積もり**であり、断定ではありません。

| レベル | 典型データ量 | 概算計算 | 概算 GPU 時間 | クラウド概算費用 | 自宅可否 |
|---|---|---|---|---|---|
| RAG | —(学習なし) | 埋め込み推論のみ | ~0(索引構築のみ) | 数百〜数千円 | ◎ |
| SFT + QLoRA(7B) | 1万〜10万例(~$10^8$ tok) | $6ND\approx 6\times10^{18}$ | 十数 GPU 時 / 単GPUで数時間 | ほぼ電気代 | ◎ |
| 蒸留(応答蒸留=SFT型) | 教師生成 10万〜100万例 | 教師**推論**が律速 | 教師規模次第 | 数千〜数万円 | ○ |
| 継続事前学習(7B, 50B tok) | $10^{10}$〜$10^{11}$ tok | $2.1\times10^{21}$ | ~4,700 GPU 時 | ~0.7万〜1.9万ドル | △ |
| from-scratch(7B, Chinchilla最適 140B tok) | $D\!\approx\!20N$ | $\approx 5.9\times10^{21}$ | ~13,000 GPU 時 | ~2万〜5万ドル | ✕(競争力なら✕) |
| from-scratch(7B, 実運用級 2T tok) | $10^{12}$+ tok | $8.4\times10^{22}$ | ~187,000 GPU 時 | **~30万〜70万ドル** | ✕ |

費用帯は、A100-80GB を **1 時間あたり 1.5〜4 ドル**(spot 〜 on-demand、出所: 主要クラウドの公表単価の幅)として掛けた桁感です。

ここで **正直な限界(honest disclosure)** を強調させてください。上表の費用は **「成功した 1 回分の純計算費」だけ** です。データ整備・失敗ラン・人件費・評価コストは含みません。実プロジェクトでは、失敗ランと試行錯誤で **2〜3 倍** に膨らむのが通例です(経験則。案件で大きく振れます)。ですからこの表は、発注書の金額ではなく、**「発注前に桁を間違えないための下敷き」** として読んでください。

念のため、Chinchilla 最適の 140B トークン行を検算しておきます。

$$6\times(7\times10^9)\times(1.4\times10^{11}) = 5.88\times10^{21}\approx 5.9\times10^{21}\ \text{FLOPs}$$

これを実効速度 $1.25\times10^{14}$ FLOPs/s で割ると、

$$\frac{5.9\times10^{21}}{1.25\times10^{14}\times3600}\approx 1.3\times10^4 = \text{約 13,000 GPU 時}$$

表の値ときちんと整合します。同じ式が全行を貫いていることが、桁を信じてよい根拠です。

最後に、from-scratch の最下段がなぜ 30 万ドル級まで跳ね上がるのかに触れておきます。理屈は「推論効率」です。**Chinchilla 最適($D\approx20N$)は、与えた計算を損失最小に配分する点** にすぎません。ところが実運用のモデルは、**あえて最適を超えてトークンを注ぎます**(Llama 系の 2T トークンがまさにこれ)。理由は、**「小さく作って長く訓練し、安く推論する」** という設計思想です。学習時に余計に払っておけば、その後何百万回と回す推論が軽くなる。この「学習でコストを前払いして推論を安くする」判断こそが、from-scratch の総コストを表の下段まで押し上げる主因です。

> 図: 6ND 則でコストの桁を掴む。横軸=学習トークン $D$、縦軸=GPU 時間(対数)。SFT/QLoRA は左下の点、from-scratch(実運用級)は右上で 5〜6 桁上に離れることを示す。

### いつ何を選ぶか — 判断軸マトリクス

桁が読めたので、次は「で、自分のケースはどれか」を決める番です。手持ちの制約から逆引きできるよう、6つの判断軸で並べます。

| 判断軸 | RAG | SFT/LoRA | 蒸留 | 継続事前学習 | from-scratch |
|---|---|---|---|---|---|
| **手持ちデータ量** | 文書があれば可 | 1万〜10万の入出力対 | 教師出力を大量生成可 | ドメイン生テキスト $10^{10}$+ tok | 高品質 $10^{12}$+ tok |
| **計算予算** | ほぼ不要 | 単GPU〜 | 中(教師推論) | 数GPU〜ノード | 数百〜数千 GPU |
| **欲しい差別化** | 最新事実・出典明示 | 形式・作法・軽スキル | 大モデル能力の小型化 | ドメイン知識・語彙 | 独自構造・独占能力 |
| **ライセンス制約** | base に依存 | base の商用可否に依存 | **教師出力の利用規約**に依存(重大) | base ライセンスに依存 | 自由(自作) |
| **オンプレ/機密要件** | 索引もローカルなら◎ | 学習・推論を閉域で完結◎ | 教師が API 依存だと機密流出リスク | 完全ローカル可 | 完全ローカル可 |
| **鮮度要件** | ◎(索引更新のみ) | ✕(再学習必要) | ✕ | ✕ | ✕ |

この表を「上から順に試す」チェックリストに畳むと、自宅〜中小規模ではだいたい次の順序になります。

1. **まず RAG。** 知識の鮮度・出典・機密隔離が主目的なら、重みを一切触らずに要件の大半が満たせます。最初にここを試さない理由はほぼありません。
2. **形式・口調・指示追従が欲しければ SFT + QLoRA。** ここで一つ、よくある誤解を正しておきます。単一の 24GB 級 GPU(RTX 4090 など)で 7B を微調整できるのは、**QLoRA の 4bit 量子化**(Dettmers et al. 2023)の功績であって、Alpaca ではありません。Alpaca 自体は LLaMA-7B の**フル微調整**で 8×A100 を使っています。Alpaca(Taori et al. 2023)が確立したのは、self-instruct による**命令チューニング用データと SFT レシピ**の定石のほうです。**技術(単一 GPU 化=QLoRA)とレシピ(命令データ=Alpaca)を分けて捉える**と、この界隈の話がすっきり通ります。
3. **RAG × SFT は競合ではなく併用。** 知識は RAG、振る舞いは SFT、と役割分担するのが最短経路です。「どちらか」で悩むのは、たいてい問いの立て方を間違えています。
4. **ドメイン語彙・分布が根本的にズレて初めて継続事前学習。** 医療・法務・特定言語・コードなど、base の事前学習分布から遠い領域に限られます。
5. **大モデルの能力を小型・オンプレに落としたいとき蒸留。** 具体例として、DeepSeek-R1(DeepSeek-AI 2025)は、R1 の推論能力を **SFT のみ(RL なし)で約 80 万サンプル=推論 60 万 + 非推論 20 万** で蒸留しました。ここに、括ると落ちてしまう非対称があります。蒸留元の base は Qwen2.5(1.5/7/14/32B)・Llama-3.1-8B-Base・Llama-3.3-70B-Instruct の混在で、**Llama 側は 8B が base 起点・70B が Instruct 起点** という食い違いがあるのです。「Qwen/Llama 系から蒸留」と一言で括ると、この差分が見えなくなります。細部に、設計者の意図が宿っています。

### 「無関係な別モデルの平均で新モデルは作れない」— マージの原理的制約

ここは、SNS で最もよく誤解される論点なので、原理から潰しておきます。

「人気モデルを何個か平均すれば最強が作れるのでは?」——直感的には魅力的ですが、**重み空間のマージが機能するのは、被マージ体がすべて同一 base(同一初期化)から派生している場合に限られます**。単純平均も、task arithmetic($\theta_{\text{new}}=\theta_{\text{base}}+\sum_i \lambda_i(\theta_{\text{ft}_i}-\theta_{\text{base}})$)も、TIES-Merging も DARE も、この前提の上で初めて成立します。理由は損失地形(loss landscape)の構造にあります。

- ニューラルネットには **置換対称性(permutation symmetry)** があります。独立に初期化・学習した 2 つのモデルは、機能的に等価でも、重みは別々の「谷(basin)」に落ちています。両者を素朴に平均すると、谷と谷の間の **壁(高損失領域)** を横切ってしまい、性能が崩壊します。
- 一方、**同一 base から微調整した重み同士**は、**線形モード接続性(linear mode connectivity)**(Frankle et al. 2020)を満たしやすく、平均しても同じ谷に留まります。だからこそ **model soups**(Wortsman et al. 2022)や **task arithmetic**(Ilharco et al. 2022)が成立するのです。
- 例外として **Git Re-Basin**(Ainsworth et al. 2022)は、片方をニューロン置換で整列させれば別 init でも接続しうると示しました。ただし、実務的な汎用手段とは言えません。

したがって、**Llama と Mistral の重みを平均しても意味を成しません**(初期化も、しばしばトークナイザすら違います)。ここから、「内製で得られる自由度」の正直な範囲が引けます。

- **重みマージで得られる自由度** = *同じ祖先から生えた* 能力の再結合に限られます。例えば「日本語 SFT 版 + コード SFT 版(どちらも Llama-3-8B 派生)」の合成なら成立します。**異系統は不可**です。
- **系統をまたいで混ぜたい**なら、重み空間ではなく **知識空間** での融合が必要です。出力分布を介した蒸留や、**FuseLLM**(Wan et al. 2024)のような確率分布の融合がこれにあたります。ただしこれは「マージ」ではなく**蒸留の一種**であり、生徒を新規に学習させるコストがかかります。

この 2 分法——**同一 base の重み合成 vs. 異系統の知識融合**——は、実は本連載の後続回で扱う「モデル融合」の分類軸そのものです。今回は「安易な平均は壊れる」という原理だけ、しっかり握っておいてください。

### from-scratch が正当化される稀なケース

ここまで散々「from-scratch は割に合わない」と言ってきましたが、フェアに、**正当化される条件**も挙げておきます。ほぼ常に adapt/distill が正解だとしても、次のいずれかに当てはまるなら from-scratch を検討する価値があります。

- **既存 base のライセンスが要件と両立しない。** 商用配布や独占が必須で、寛容ライセンスの base ですら許容できない場合。自作なら権利は完全に自分のものです。
- **アーキテクチャ/トークナイザ自体が研究対象。** 新しい注意機構・状態空間モデル・独自語彙(非ラテン言語や専門記号)を base から検証したいとき。**継続事前学習では構造とトークナイザは変えられません**——ここが継続 PT の天井です。
- **base の事前学習分布から致命的に遠い。** 既存モデルがほとんど見ていないモダリティ・言語・記号体系で、継続事前学習の「増改築」では基礎が保たない場合。
- **極小モデルを特定タスクに最適配分。** $10^7$〜$10^8$ パラメータ級を専用データで最適トークン配分するなら、from-scratch でも現実的な GPU 予算に収まります(まさに 6ND 則で桁を見積もれます)。

裏を返せば、**「独自の LLM が欲しい」という動機の大半は、継続事前学習 + SFT で満たせます**。競争力のある汎用 7B を from-scratch する 30 万〜70 万ドル(純計算のみ)は、ほぼ常に adapt に回したほうが費用対効果で勝ります。

## 誤解されやすい点・正直な限界

ここからは、実際に手を動かすと踏み抜きやすい落とし穴を、正直に並べます。どれも筆者が(あるいは周囲が)一度は引っかかった類のものです。

1. **「from-scratch こそ真の内製」は誤りです。** 重みを乱数から回すことが、内製度の高さではありません。**継続事前学習も SFT も、対象を自分の手に握る立派な内製**であり、費用対効果は桁で優れます。本質は、「握る」対象を **知識 / 振る舞い / 事実** のどこに置くかを設計することです。

2. **SFT は知識を増やしません。** SFT/LoRA が教えるのは主に **出力の作法**(形式・指示追従・文体)であって、新しい世界知識の注入は不得手です。少数の SFT 例で事実を覚え込ませようとすると、覚えないか、**幻覚(hallucination)を悪化させる**かのどちらかになりがちです。新しい事実は、**RAG(鮮度・出典が要るなら)か継続事前学習(分布ごと入れたいなら)**の担当です。表の「変えにくいもの」列を思い出してください。

3. **RAG と fine-tuning は目的が直交します。** 「RAG か FT か」は、そもそも誤った二者択一です。**RAG = 知識の外付け、FT = 振る舞いの内面化**。両者は別の軸にあります。要件を「知識が欲しいのか、作法が欲しいのか」で切り分ければ、選択はほぼ自動的に決まります。悩みが解けないときは、たいてい問いが「知識と作法」を混ぜています。

4. **蒸留は通常は教師に漸近しますが、例外があり、ライセンスが汚染しうる点に注意です。** 応答蒸留の生徒は、多くの場合、教師の能力上限に漸近します(教師が実効的な天井に近い)。ただし *原理上超えない* は言い過ぎです。**born-again networks**(Furlanello et al. 2018, ICML)は同容量の生徒が教師を上回ることを実証し、**weak-to-strong generalization**(Burns et al. 2023)は弱い教師の監督で強い生徒が教師を超えうることを示しました。dark-knowledge の正則化効果や、教師アンサンブルの蒸留でも、生徒が上回る例があります。ですから正確には「超えない」ではなく **「通常は漸近、条件次第で超過もある」** です。加えてライセンス面が重要で、**商用 API の出力を学習データに使う行為は、多くの提供元の利用規約で禁止**されています(競合モデル育成の禁止条項。要検証: 提供元・時期で条項は変わるため、使う前に必ず現行規約を確認してください)。オンプレの機密案件で外部 API を教師にすると、機密が API 側へ流出するリスクも併発します。安全策は、**自作または寛容ライセンスの教師**を使うことです。

5. **マージの過大評価に注意です。** 「良いモデルを何個か平均すれば最強が出来る」は、**同一 base かつ相補的な微調整**という条件下でのみ成立する、狭い技法です。無関係な系統の平均は破綻します(前節の原理)。マージは *安上がりな能力の再結合* であって、**新能力の創出装置ではありません**。ここは期待値を正しく持つだけで、無駄な時間を大幅に節約できます。

6. **自宅規模の、正直な切り分け**(RTX 4090 単基〜数基、クラウドの一時借り前提):

   **作れるもの**
   - RAG パイプライン全体
   - 7B〜13B の QLoRA-SFT
   - **数十億トークン級の継続事前学習**(単基 4090 なら日〜週、数基でさらに短縮)
   - 自作・寛容ライセンス教師からの蒸留
   - 同一 base 派生のマージ

   ここで、GPU 可否の内訳も 6ND で正直に見ておきます。7B の継続 PT を実効速度 $\approx 6.6\times10^{13}$ FLOPs/s(4090, FP16 累積 ~165 TFLOPS × MFU 0.4 と仮定。要検証: 実装効率で上下します)として計算すると、
   - **数十億($10^{10}$)トークン**: 単基で **約 74 日相当**、数基に分ければ **日〜週**。
   - **数百億($10^{11}$)トークン**: $6ND\approx 4.2\times10^{21}$ で、単基だと **約 740 日 = 週〜月級**、複数 GPU 前提。

   つまり「日〜週で回る」のは **数十億トークン以下 × 数基** のときだけで、数百億トークンの端は自宅単基では現実的ではありません。「継続 PT は自宅でいける」は正しいですが、**トークン量で線を引く**のを忘れないでください。

   **作れないもの(現実的でない)**
   - 競争力のある汎用 7B+ の from-scratch(数十万 GPU 時 = クラウドでも $10^5$〜$10^6$ ドルのオーダー)
   - 数千億〜兆パラメータ級の事前学習

   ここは「借りても割に合わない」領域です。正しい判断は、見栄を張らずに **既存 base に adapt すること**——それに尽きます。
