# 700本の筋肉で人体骨格を立たせようとして、正直に転んだ話 — でも手はボトルを掴んだ

> 対象: ロボット・バイオメカニクス・進化計算・MuJoCo に興味がある人 / 「筋肉で動く人体モデル」を触ってみたい人 / 良い結果より**正直な内訳**を読みたい人。
> 前提知識: Python が読めれば十分。制御・生体力学の用語は都度かみくだきます。
> 数値はすべてノート PC の CPU で実測。この記事は成功談ではなく、**「一方は正直に負け、一方は勝った」その両方を同じ物差しで測った話**です。

---

## 0. 3行まとめ(先に結論)

- **筋骨格モデル(musculoskeletal model)**——骨・腱・**700本の筋肉**でできた全身の人体を、自宅の CPU 上の MuJoCo に載せ、**筋肉を1本ずつ進化で動かして**「立つ」「掴む」をやらせてみた。トルクのモーターではなく、**縮む力しか出せない筋肉**で動かすのが肝。
- **立位(standing)は正直に負けた**: 何もしないと約1秒で崩れ落ちる骨格を、閉ループの制御で**崩落を3.7〜4.5秒まで遅らせた**(ベースラインの3.7〜4.5倍)。でも6秒立ち続けることはできず、最後は必ず倒れた。**倒立振子(inverted pendulum)を素朴な方策で恒常安定化するのは、この計算予算では届かない**——誇張せず負けを残す。
- **把持(grasp)は勝った**: 肩を固定した腕モデルで、**81本の筋肉の右手が机の上の水ボトルを掴み、頭上0.82mまで持ち上げて保持した**。しかも「放り投げ」ではなく「掴んで保持」であることを、手のひらとボトルの距離を測って**検証**した。**なぜ立位は負けて把持は勝てたのか**——その違いが今日の主役だ。

## 1. 用語(先に地図)

| 用語 | かみくだき |
|---|---|
| 筋骨格モデル(musculoskeletal model) | 骨・関節だけでなく**筋肉と腱**まで持つ人体シミュレーション。今回は700本の筋肉つき全身 |
| 筋アクチュエータ(muscle actuator) | 「筋肉」を動力源にした駆動。モーターと違い**縮む(引く)力しか出せず**、0〜1の活性で力が決まる非線形な代物 |
| 自由度(DoF / degrees of freedom) | 独立に動ける関節軸の数。今回の歩行用モデルは36、全身は85 |
| 冗長(redundant) | 動かしたい自由度より筋肉の数がずっと多い状態(36自由度を100筋で動かす)。「正解の筋肉の組合せ」が無数にあり、探索が難しい |
| 筋シナジー(muscle synergy) | 100本の筋肉を、少数(K本)の「まとめて動くパターン」に圧縮した表現。人間の脳も筋肉をこう束ねて使うとされる |
| 閉ループ制御(closed-loop) | 体の状態(傾き・速度・接地)を**見ながら**筋肉を調整する制御。見ずに決め打ちするのが開ループ(open-loop) |
| 倒立振子(inverted pendulum) | 逆さに立てた棒。放っておけば倒れる。二足立ちの本質はこれを絶えず立て直すこと |
| CMA-ES(進化戦略) | 解の集団を変異・選抜して良くする最適化。今回は制御方策の~800〜1000個の数値を進化させる |
| ベースライン(baseline) | 学習前・無制御の対照。「学習でどれだけ良くなったか」を測る出発点 |
| RTF(real-time factor) | シミュがどれだけ実時間より速く回るか。13×なら1秒の物理を約0.08秒で計算(値は CPU・並列度に強く依存) |

## 2. なぜ「全身骨格」で、なぜ先に調べたか

新しいことを始める前に、まず調べる。これは私の習慣だ。今回の題材は **MS-Human-700**——中国 LNS Group が作り、**ICRA 2024** で発表した全身の筋骨格モデル(Zuo, He, Shao, Sui, *Self model for embodied intelligence*)。Apache-2.0 で公開され、MuJoCo の公式モデル集 **Menagerie** にも収録されている。骨・腱・**700本の筋肉**が入った、文字通りの「全身骨格」だ。

そして論文を読んで、正直に期待値を調整した。この論文の主眼は「700筋という高次元を**階層的な低次元表現(≒筋シナジー)**で制御し、歩行を学習する」ことにある。つまり**まともに動かすには、素朴な方法では足りない**と、作者自身が言っている。

> 教訓の先取り: 「筋肉で人体を立たせる」は、GPU クラスタと階層強化学習を投じても難しい、れっきとした研究課題だ。それをノート PC の CPU と進化戦略でやろうというのだから、**負ける可能性を最初から勘定に入れる**。負けたら負けたと書く——それがこのシリーズの背骨だ。

私が価値を置くのは「立たせられた」という結果そのものではない。**同じ物差しで、どこまで行けてどこで止まったかを正直に帰属すること**——それが今日書きたいことだ。

---

## 3. 舞台: 700筋の人体を、自宅の CPU に載せる

まず MS-Human-700 を、自作の研究基盤 onocollo(MuJoCo の上に被せた型付きランタイム)にロードした。3つの変種がある。

| 変種 | 自由度 | 筋肉 | 用途 |
|---|---|---|---|
| 全身(full) | 85 | **700** | 全身すべての筋 |
| 歩行用(locomotion) | 36 | 100 | 上半身を簡略化。立位・歩行の実験に使う |
| 操作用(manipulation) | 48 | 81 | 右腕と**詳細な手**+机上の水ボトル |

大事な事実をいくつか。

- **アクチュエータは全部「筋肉」**。ロボットのモーターのように「+3 N·mから−3 N·m」と両方向に回せるのではなく、筋肉は**縮む(引く)方向にしか力を出せない**。活性値は0〜1で、Hill 型という力‐長さ‐速度依存の非線形モデルで力が決まる。だから「関節を右に回したければ右の筋、左は左の筋」と**対で用意**し、しかも数が自由度より多い(=冗長)。
- **膝が生体力学的に精密**。単純な蝶番ではなく、大腿骨と脛骨が転がり滑る動きを、複数の補助関節と拘束式で再現している。
- 骨盤は「浮遊基底(floating base)」——地面に固定されず、3方向のスライドと3方向の回転で自由に動く。初期姿勢では高さ0.95mで直立している。
- **速度**: このモデルはノート PC(Intel Core, Ice Lake 世代のモバイル CPU)の**1スレッド・アイドル時**で **RTF 約13倍**(6秒の物理を約0.46秒、~6500ステップ/秒。dt=0.002s)。ただしこれは占有時の値で、**進化のために7コアを並列で回すと1プロセスあたり ~3〜5倍まで落ちる**(コアの奪い合い)。RTF は CPU 型番・並列度・積分器の設定でこれだけ変わるので、数字は必ず条件とセットで読むこと。それでも「何千回もロールアウトする」進化が現実的に回る速さではある。

![MS-Human-700 歩行用変種の初期直立姿勢。脚に赤く見えるのが筋肉と腱。骨盤の高さ0.95mで立っている](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/musculo/locomotion_init.png?v=1)

## 4. まず正直なベースライン——そして既存コントローラが「壊れる」

新しい制御を作る前に、必ず**無学習のベースライン**を測る。異常に良い数字を後で疑えるように、出発点を先に釘で打っておく。歩行用モデル(床あり)で、6秒間ロールアウトして「骨盤が0.6mを割ったら転倒」とした。

| 制御 | 転倒までの時間 | 結末 |
|---|---|---|
| 無制御(全筋弛緩) | **0.50秒** | 崩落 |
| 全筋最大収縮 | 0.99秒 | 崩落 |
| 毎ステップ乱数 | 1.00秒 | 崩落 |

**何もしなければ、人体骨格は約1秒で崩れ落ちる**。全部の筋肉を最大で固めると、剛性が上がってほんの少し延命するが、それでも立ってはいられない。これが立位課題のスタート地点だ。

そしてここで、**既存の道具が壊れることを実測した**——これが今回いちばん技術的に面白い発見だった。onocollo には既に「Menagerie のロボットの制御を進化させる」コントローラがある。だがそれは各アクチュエータに**担当関節の角度**を読ませて動かす設計だった。ところが筋肉には「担当する単一の関節」が無い(腱を介して複数関節をまたぐ)。結果、**100本の筋肉すべてが同じ(ゼロの)入力を受け取り、しかも重みを共有しているので、全100筋がまったく同じ活性を出す**。

100本の筋肉が毎瞬間まったく同じ強さで縮む——これでは二足立ちなど絶対にできない。**この退化を数値で確認した(ユニークな活性値の種類 = 1)**。だから、筋肉モデルには「全身の状態を見て、筋肉ごとに違う指令を出す」新しい表現が要る。論文の言う「階層的低次元表現」と、必要性がぴたりと一致した。

## 5. 立位タスク: 筋シナジー × 進化 —— そして正直に転ぶ

作った制御はこうだ。

1. **全身の proprioception(自己受容感覚)を読む**: 骨盤の高さの誤差・傾き(前後/左右/ひねり)とその角速度・骨盤の並進速度・左右足の接地——バランスに効く14個の量。
2. **筋シナジーに落とす**: その状態を、**K本の「まとめて動くパターン」**に線形写像し(`tanh`)、それを `K×100` の行列で100本の筋活性に展開する(`sigmoid` で0〜1に)。K=6なら方策の数値は約790個。これが論文の「低次元表現」に対応する私なりの実装だ。
3. **CMA-ES で進化**: この数値を進化戦略で最適化する。報酬は「立っている間、高さが目標に近く・上体がまっすぐ・骨盤の速度が小さいほど高得点、転んだら終了」。

> 正直な設計の訂正: 最初の報酬は「高さが目標に近ければ加点」だけだった。すると進化は狡くなり、**『ゆっくり優雅に倒れる』方策**で点を稼いだ(倒れる途中も高さはそこそこ高いから)。見た目は立っていない。そこで**骨盤の速度にペナルティ**を足し、「静かに立ち止まる」方が「ゆっくり倒れる」より高得点になるよう直した。狡さを封じる物差しの調整だ。

計算を無駄なく回すため、**モデルを1回だけ読み込んで7コアで並列評価**する仕組みも書いた(モデル読込は約2秒、1ロールアウトは数百ミリ秒なので、毎回読み直すと読込が支配的になる)。ワーカーごとに1回だけ読ませて使い回すことで、並列度ぶんの高速化が効く。

結果は——**正直に、届かなかった**。

最良値(best)だけを載せると「山を代表値に見せる」ことになるので、**典型値(進化後半の世代ベストの中央値)も併記**する。

| 制御 | 方策の数値 | 典型(中央値) | 最良(best) | 結末 |
|---|---|---|---|---|
| ベースライン(無制御) | 0 | — | 0.5〜1.0秒 | 崩落 |
| 静的な筋緊張(定数) | 100 | ~2秒 | 3.02秒 | 最後は転倒 |
| 筋シナジー(閉ループ) | 790〜1020 | ~2.5秒 | 3.71秒 | 最後は転倒 |
| **筋シナジー+記憶(積分項)** | 808 | **~3.0秒** | **4.53秒** | 最後は転倒(でも天井は上がった) |

- 閉ループの制御は、崩落を **約1秒 → 典型3秒・最良3.7〜4.5秒** まで遅らせた。ベースラインは明確に上回るし、静的な筋緊張より閉ループの方が良い。倍率を語るときの分母は控えめに「無制御 ~1.0秒」を採る(全筋弛緩の 0.50秒 を分母にすれば数字は倍になるが、それは誇張だ)。
- だが**6秒立ち続けることはできない**。最良の方策でも、最後のフレームは床に横たわっている。「よりゆっくり・より制御された倒れ方」にはなったが、倒れることに変わりはない。
- 面白いのは、**方策に「記憶」を足したら天井が上がった**こと。バランス制御は理屈上「傾きの誤差を積分して」効かせるのが定石だ(積分制御)。そこで骨盤の傾きの**leaky 積分**を状態に足したら、最良が 3.71秒 → **4.53秒**に伸びた(典型も ~2.5→~3.0秒)。**ただし各構成は乱数種1本ずつの結果**(n=1)なので、この差は「記憶が効く」仮説と**整合はするが、まだ確証ではない**——複数種で median±幅を取るのは次の宿題だ。それでも6秒完走には届かない。

![学習曲線。立てた時間は世代とともに上がるが、2〜2.7秒(スパイクで3.7〜4.5秒)で頭打ち。点線が無制御の約1秒。6秒(エピソード長)には届かない](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/musculo/learning_curves.png?v=1)

![最良の立位方策でも、約3.7秒後にはこの通り床へ。誇張せず「負け」を残す](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/musculo/stand_warm.gif?v=1)

なぜ届かないのか。素直に言えば、**記憶をほとんど持たない線形のシナジー方策を、CMA-ES で~800次元も最適化するのは、倒立振子の恒常安定化には力不足**なのだ。論文が階層強化学習を要したのと矛盾しない。ここで「立てました!」と丸めて書くのは簡単だが、それは嘘になる。**負けは負けとして残す**。

## 6. 把持タスク: 舞台を変えたら、勝てた

立位が難しいのは「倒立振子」だからだ。では、**倒れる心配のない課題**なら?——操作用の変種は、肩から上が固定されていて骨盤の自由関節が無い。つまり**バランスを失いようがない**。純粋に「腕と手で何かをする」課題になる。

そこで、**81本の筋肉でできた右手に、机の上の水ボトルを掴ませて持ち上げさせた**。手は最初からボトルの約0.1m近くにあるので、本当の課題は「近づく」ことではなく——**掴んで、持ち上げて、保持する**ことだ。

同じ筋シナジー方策を、今度は「ボトルを高く持ち上げるほど加点」で進化させた。結果:

| 世代 | ボトルの持ち上げ高さ |
|---|---|
| 0(初期) | 0.006 m(=ほぼゼロ) |
| 20 | 0.39 m |
| 50(収束) | **0.825 m** |

**81本の筋肉の右手が、机の水ボトルを掴み、0.82m 持ち上げて静止・保持した**(飲む動作の前段——「口へ運ぶ」の手前まで——に相当する)。

![操作用の全身筋モデルが、机の水ボトルを掴んで頭上まで持ち上げ、保持する。手のひらとボトルの距離を保ったままの「保持」であって、放り投げではない](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/musculo/grasp_final.gif?v=1)

### 正直に「放り投げ」を疑う

高さを最大化する報酬は、**「ボトルを上に叩き飛ばす」ズルでも点が入る**。だから「本当に掴んで持っているのか」を検証した。持ち上げた後の3秒間、**手のひら(手根中央の骨)とボトル中心の距離**を測ると、**0.057〜0.062m を保ち続けた**。放り投げなら距離は発散するが、発散しない。**接触を保ったまま持ち上げて静止**——本物の grasp-and-hold だ。

これは、以前の記事で「AI 義手に箸を持たせる」話を書いたときに残った宿題——「手が無い(箸を直接動かしていた)」問題への、**本物の解剖学的な手による回答**でもある。母指・虫様筋・骨間筋まで揃った手が、筋肉の力だけで物を掴んだ。

## 7. なぜ一方は負けて、一方は勝てたのか

同じモデル・同じ制御表現・同じ最適化なのに、立位は負けて把持は勝った。違いはひとつ、**課題の物理的な安定性**だ。

- **立位は不安定平衡**。倒立振子は、ほんの少しの傾きが指数的に増える。制御は「絶えず・素早く・正しく」立て直し続けねばならず、少しでも間違えば発散して倒れる。記憶なしの線形方策には荷が重い。
- **把持は(この設定では)安定**。肩が固定され、目標(高く持ち上げる)は一度達すれば維持しやすい。多少の制御の粗さは致命的にならない。

**「AI にできること/できないこと」は、方策の賢さだけでなく、課題そのものの安定性で決まる**。これは私にとって、今回いちばんの収穫だった。同じ道具でも、土俵を選べば勝て、選ばなければ正直に転ぶ。

---

## 8. 次回に続く

把持は「掴んで最大まで持ち上げる」1種類の動作だった。次は**「指定した位置へボトルを運ぶ」**——命令に従う制御可能な操作を進化させている(方策に『目標−ボトル』のベクトルを見せ、複数の目標で同じ制御器を鍛える)。うまくいけば、「筋肉の手が、言われた場所へ物を置く」ところまで見せられるはずだ。そして立位は——**記憶をもっと厚くするか、階層を入れるか**。負けたままでは終わらせない。

> この記事の数値はすべて、乱数種を固定した再現可能な CPU 実行から取った。良い結果が出たときほど内訳を疑う——それが、私がこのシリーズで守っている一点だ。
