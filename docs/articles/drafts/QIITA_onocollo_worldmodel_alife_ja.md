# 自宅CPUで「世界モデルを作るためのプラットフォーム」を作り、その上で人工生命を進化させ、正直にコケた話

> GPU なしのノート PC(CPU のみ)で、**世界モデル(world model)を組み立てるためのプラットフォーム**を作っています。世界モデルを学習し、それを **ROS 風のランタイム**の中で動かし、さらに同じ土台の上で **単細胞から進化する人工生命**を育て、最後に **宇宙ゴミ回収ロボの腕**を進化させるところまでを、**すべて本物の実行結果つき**でたどります。うまくいった所も、**うまくいかなかった所も隠さず**出します。むしろ後者がこの記事の主役です。

## この記事の 3 行

- 作っているのは単発のデモではなく、**世界モデルを「部品として組み立て、測り、ロボットの脳に差し込む」ための小さなプラットフォーム**(`onocollo`)です。frontier(Genie / Sora 級)は家庭で作れないので、**「誰でも CPU で世界モデルを構築・実験できる土台」**に土俵を移しています。
- その土台の上で、世界モデルを **ROS っぽい配線**に載せ、さらに **人工生命の進化**まで回せます。流体の中では多細胞の体が単細胞の **6.5 倍**進む、という多細胞化が報われる瞬間まで見えました。
- そして **宇宙ゴミ回収ロボの腕**を進化させ、「壊れにくさ(冗長性)は効くはず」という自分の仮説を、**実測でひっくり返しました**。全部ノート PC の CPU で、数秒〜数分です。

はじめての方は上から、急ぐ方は「2. プラットフォームとは」「5. 正直な現実」「7. 宇宙ゴミ」だけでも要点が取れます。

## 2. onocollo とは ─ 世界モデルを作るための「プラットフォーム」

**onocollo**(おのころ)は、私が自宅 CPU で作っている OSS で、狙いは 1 つの派手なデモではなく、**世界モデルを組み立てるための土台(プラットフォーム)**です。名前は古事記の **淤能碁呂島(おのころじま)**(かき混ぜた原初の海が凝り固まって最初の島=世界が生まれる)と、コロイド化(colloidal=凝集)を掛けた造語で、「潜在空間がかき混ざって世界(モデル)が生まれる」イメージです。

### なぜ「プラットフォーム」と呼ぶのか

世界モデルの研究には、実は**入れ替え可能な軸が 3 本**あります。onocollo は、その 3 本を**それぞれ差し替え可能な部品**にして、どう組み合わせても同じ数行の API で回るようにしました。

1. **世界モデルの流儀(アルゴリズム軸)**。古典の **World Models**(Ha & Schmidhuber, 2018 ── VAE で見て、MDN-RNN で記憶し、進化戦略で制御する **V / M / C** の 3 分割)と、現代の **Dreamer**(Hafner ら ── **RSSM** という 1 本の潜在動力学 + 想像の中で学ぶ actor-critic)。**この 2 系譜を同じインターフェースで載せ替え**られます。
2. **環境(タスク軸)**。依存ゼロの合成環境・**実物理の MuJoCo**(ヘッドレス CPU)・任意の **gymnasium** 環境(連続 Box も離散 Discrete も自動判別)を、**設定 1 つで差し替え**。世界モデル側のコードは一切変わりません。
3. **動かす場所(ランタイム軸)**。学習した世界モデルを、**ROS 風のグラフ**(後述)や **MuJoCo 物理**の中に、**一級の部品(ノード)**として差し込めます。

```python
import onocollo as oc

env   = oc.GymEnv("Pendulum-v1")     # 環境はここだけ差し替える
agent = oc.Dreamer(env, config="tiny")   # 流儀もここだけ差し替える(oc.WorldModel も可)
agent.fit(steps=2000)                # 収集→世界モデル学習→行動学習
dream = agent.imagine(horizon=15)    # 世界モデルの中だけで未来をロールアウト
```

**「アルゴリズム × 環境 × ランタイム」を自由に組み替えられる**── これがプラットフォームと呼ぶ理由です。今日は tiny な CPU 設定ですが、**スケールアップは設定変更であって書き直しではない**(重い依存 `mujoco` / `cma` / `gymnasium` は遅延読み込みの optional)。CPU で組み上げて、そのまま 1 枚の GPU に載せる道が最初から通してあります。

### プラットフォームとしての「誠実さ」

もう 1 つ、これがただの実験集と違うのは、**「都合の悪い数字を測って見せるための評価器」がプラットフォームに組み込まれている**ことです(§5)。世界モデルは黙っていると「自分の夢の中でだけ賢くなる」ので、**実環境での成績と、予測のドリフト(ズレ)を常に測る**部品を標準装備しました。frontier のベンチ性能では勝てません。だから **「最初の“夢”までの時間の短さ」と「正直さ」で勝つ**、という設計です(全体 460 テスト緑・Apache-2.0)。

## 4. `onocollo.ros` ─ 世界モデルを「脳の部品」として配線する

世界モデルは「考える」部品。プラットフォームの価値は、それを**ロボットの脳の配線**の中に、他の部品と対等に差し込めることです。世界モデルを **`/imagine` サービス**として提供すると、他のノードはそれを他のどんな ROS サービスとも同じように呼べます。

```python
from onocollo.ros import Graph, EnvNode, PolicyNode, WorldModelNode, ServiceClient

g = Graph()
g.add_node(EnvNode(env, name="plant"))          # /obs 配信・/action 購読(=本体)
g.add_node(PolicyNode(agent, name="ctrl"))      # /obs 購読・/action 配信(=制御)
g.add_node(WorldModelNode(agent, name="wm"))    # /imagine サービス(=想像)
g.spin(count=200)                               # 観測→考える→動く を決定論的に

dream = ServiceClient(g, "/imagine").call({"horizon": 15})   # 世界モデルを"サービス"として呼ぶ
```

「ROS 系でできることは全部できる」を目指しましたが、**嘘はつきません**。開発・シミュレーション・学習で ROS を使う場面は onocollo で完結します。しかし実機のドライバ・ハードリアルタイム・DDS 配線互換は **持ち込み(bring-your-own)** で、そこは再実装したと名乗りません。

| ROS 2 の機能 | onocollo | パリティ |
| --- | --- | :---: |
| ノード / トピック / pub-sub | `Graph` + `Node` | ✅ プロセス内 |
| 実行器 / spin / タイマ | `Graph.spin`, `Rate` | ✅ ステップ+ソフトRT |
| tf2 変換 | `TransformBuffer`(補間つき) | ✅ |
| rosbag2 記録/再生 | `Bag` + `Recorder`/`Player` | ✅(独自形式) |
| サービス / アクション | `Service` / `Action` | ✅ 同期 |
| ros2_control ハードウェア | `HardwareInterface` + `SimHardware` | ⚠️ **IF+sim。実ドライバは自作** |
| DDS 探索 / マルチホスト | — | ❌ 単一プロセス |
| ハードリアルタイム制御 | ソフトRTのみ | ❌ RTスケジューラではない |

**ハードウェア線より上**(認識・世界モデル・方策・tf・bag・launch)は、相手が `SimHardware(env)` でも実機の `HardwareInterface` でも **同一**です。脳全体を CPU で開発・シミュレーションし、最後にドライバだけ差し込む ── ここが「プラットフォーム」の実利です。ROS 2 そのもの(実機ドライバ・DDS・ハード RT)の代替ではありません。表がそう言っています。

さらに、実物理の **MuJoCo エンジンを型付きで包む層(`onocollo.mujoco`)** も同じ土台に載っていて、`MujocoBridgeNode` を挟めば **MuJoCo の世界を上の ROS グラフと同じノードで**「観測→考える→動く」できます。物理・配線・世界モデルが 1 つのプロセスに揃う、というわけです。

## 5. 測って分かった正直な現実(この記事の主役 その 1)

**異常に良い結果が出たら、勝った気になる前に内訳を疑う** ── これは私の開発の家訓です。逆に、悪い結果も消しません。

### 5.1 「夢の中では」ちゃんと学習している

合成環境で世界モデルを学習させた実測(120 反復・CPU):

```
world-model loss   : 71.87  ->  19.57     # 下がる = 動力学を学んでいる
imagined return    : -0.398 ->  10.287    # 想像の中の報酬は上がる
greedy real return : 7.359   vs random 2.697   # 易しい合成環境では random に勝つ
```

合成環境(=学習しやすいように設計した玩具)では、**学習済み方策がランダムに勝ちました**(7.36 vs 2.70)。**配線は正しく、学習信号は本物**です。

### 5.2 でも実タスクでは、ランダムに負ける

同じ API を、**本物の制御タスク**に向けるとどうなるか。振り子(Pendulum-v1, 連続)と倒立振子(CartPole-v1, 離散)での実測(80 反復・CPU):

| 環境 | 行動空間 | ランダム | 学習後 | 差 |
| --- | --- | ---: | ---: | ---: |
| Pendulum-v1 | 連続 | **-1158.5** | -1406.1 | **-247.5** |
| CartPole-v1 | 離散 | **14.0** | 9.6 | **-4.4** |

**学習後の方が悪い。** これは失敗ではなく、世界モデルの本質が現れた瞬間で、名前がついています ── **世界モデルの搾取(world-model exploitation)**。actor は「学習途中の・まだ不正確な報酬予測」を最大化するので、**想像の中のリターンは上がるのに実環境のリターンは下がる**ズレが起きます。易しい合成環境では出ない。実タスクでは出る。

### 5.3 ドリフト曲線が「世界は動かない」にすら勝てない現実を暴く

プラットフォームには、この不都合を**隠すためではなく、測って見せるため**の評価器が標準装備されています。世界モデルに、実環境と同じ行動列を「観測を見ずに」ロールアウトさせ、1 ステップごとの予測誤差を実測します。

![開ループのドリフト曲線。モデルの予測誤差はpersistence(世界は動かない)ベースラインより数百〜千倍大きく、曲線は右肩上がりでなく平坦](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/drift_curve.png)

読み方:**persistence(「世界は動かない」と予測するだけのベースライン)より、モデルの誤差の方が数百〜千倍大きい**(`model/persist ≫ 1`)。この小さいモデルは再構成誤差の床に張り付いていて、**「世界は動かない」という一番バカな予測にすら負けている**。教科書的な「開ループ誤差の累積」は、テストでは劣化させた予測器で示せますが、**この規模の本物のモデルでは起きません**。評価器は、起きた事実だけを報告します。

### 5.4 「もっと計算すれば良くなる」を測ったら、逆に悪化した

「反復を増やせば実制御も良くなるだろう」と思って、Pendulum で **反復 80 vs 反復 300(モデルも大きい)** を実測しました。予想は外れました。

| 設定 | 実 eval(高いほど良い) | world-model ロス | 想像リターン |
| --- | ---: | ---: | ---: |
| ランダム | **-1158.5** | ─ | ─ |
| 80 反復 | -1406.1 | 66.9 → 17.4 | 0.42 → **-24.8** |
| 300 反復 | **-1522.2** | 75.8 → **4.0** | 0.02 → **-154.5** |

**計算を増やしたら、実制御は悪くなりました**(-1406 → -1522)。しかも内訳が示唆的です:計算を増やした方が **world-model ロスは最もよく下がっている**(4.0)。つまり **動力学の予測は一番よく学べている**のに、実環境の成績は最悪。動力学が良くなるほど、actor は「まだ不完全な報酬予測」の穴を**より巧妙に突ける**ようになる。**「もっと回せば良くなる」は、少なくともこの規模の Pendulum では偽**でした。

> これこそ、正直な評価器を**プラットフォームの一部として最初から**作っておく価値です。もし実環境リターンを測っていなければ、「ロスが 4.0 まで下がった!学習成功!」と誤って喜んでいたでしょう。

## 6. 同じ土台の上で、人工生命を進化させる(この記事の主役 その 2)

プラットフォームの面白さは、**世界モデルとは別の遊びも、同じ MuJoCo 土台の上で回せる**ことです。ここでは **Karl Sims の「進化する仮想生物」**(1994)の系譜を継ぐミニ版(`onocollo.evolve`)を載せました。

やることはこうです。**体のゲノム**(自由に動く根 + 波打つ関節の“節”の鎖)を、手続き的に**本物の MuJoCo モデルへ変換**し、動かして、その成績で選ぶ。突然変異で節を増やしたり減らしたり ── だから **体そのものと歩き方が同時に進化**します。1 節の“単細胞”から始めます。

### 6.1 貪欲な進化は「単細胞」に縮む

まず素朴な `(μ+λ)` 進化(良いものを残して変異、を繰り返す)。平地を這わせると、**0.009 m → 0.62 m** と 20 世代でぐんぐん伸びます。……が、行き着く先は **1 節への収束**でした。平地の短距離走では余計な節はむしろ邪魔で、いちばん単純な体がいちばん速い。**単一目的の最適化は、最小の体に縮む**のです。

### 6.2 QD(品質多様性)で「体サイズ別の名人」を残す

そこで、前回の記事([AI に料理コンテストをさせたら…](https://qiita.com/furuse-kazufumi/items/40ba7cc91ac577274b74))で書いた **MAP-Elites / QD** の出番です。「総合 1 位」だけを追うと多様性が死ぬので、**(節数, 姿勢)ごとにチャンピオンを別々に残す棚**を使います。すると 1 節に潰されず、**全 6 種類の体サイズのチャンピオンが同時に生き残ります**(1 節 +0.108 m … 6 節 +0.005 m)。ピーク性能と引き換えに多様性を守る ── これが QD のトレードオフです。

### 6.3 感覚フィードバックで、多節が「使える」体になる

次に、各関節が**自分のセンサと歩調の時計を読んで動く小さな神経回路(閉ループ制御)**を載せ、体と一緒に進化させます(これが Karl Sims 本来の形)。すると多節の体が**競争力を持ち、走行の多くの世代で先行**しました(2〜3 節がリード・最良 +0.27 m)。開ループが 1 節に縮んだのと好対照です。

### 6.4 水の中で、多細胞化が「決定的に」報われる

いちばん綺麗に出たのがこれです。**重力を切って粘性のある流体**にすると、多節の体は**進行波(travelling wave)**でぐいぐい進み、1 節はほとんど動けません。QD で測ると **6 節 +0.92 m 対 1 節 +0.14 m ── 6.5 倍**。ここで初めて、**多細胞であることが決定的に得**になりました。面白いのは、貪欲な `(μ+λ)` は流体でも 1 節に収束してしまうこと。**多様性を守る QD だからこそ、この多節の最適解を見つけられた**のです。

![粘性流体の中で多節の生物が進行波で泳ぎ、1節はほとんど動けない様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/swimmer.gif)

### 6.5 「届く/届かない」は体の長さで決まる ── 幾何の壁

もう 1 つ、体長が物理的に効くタスク。根を固定して、先端を遠くの目標に届かせます。目標を **0.85 m** に置くと、**1 節は 0.55 m も足りず**、**3 節以上でようやく届く**。ここでは巧妙な制御より前に、**体の長さという幾何(geometry)の壁**が立ちはだかります。「賢さ」ではどうにもならない、身も蓋もない物理です。

![根を固定した腕が遠くの目標(緑のマーカー)に手を伸ばす。1節は届かず、多節が目標に接触する比較](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/reach.gif)

> **正直な但し書き:** ここでの「単細胞」は**体の 1 節**のことで、細胞生物学(膜・代謝・分裂)ではありません。物理も toy スケールの剛体です。それでも「単一目的の進化は最小の体に縮み、多様性(QD)と感覚と課題設定が多細胞化を報いる」という **筋の通った物語**が、ノート PC の CPU で数分で観察できたのは、私には十分に面白い結果でした。

## 7. 同じアームを微小重力で試す ── そして「回収」には把持が要る、という現実

§6 の「到達アーム」を、そのまま**微小重力(重力オフ)**に置いて、漂ってくる的(タンブリングする箱)に手を伸ばさせてみました。頭の中では「宇宙ゴミ(デブリ)回収」を夢想した実験です。

![微小重力で、固定ベースのアームが漂う的(金色の箱)に手先を届かせる toy。掴んではおらず、手先が的の半径に入ったか(近接)だけを判定している](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/space_debris_capture.gif)

……ただ、正直に言うと **これは「ゴミ回収ロボ」ではありません**。§6 と同じ到達タスクを、微小重力の的に向けただけ。そして ── お察しのとおり ── **「回収」には決定的に足りないものがあります。掴む力(把持 / grasping)です。**

### まず「回収」の現実 ── 触れることと、掴むことは別物

本物のデブリ回収に最低限要るものを並べると、この toy がどれだけ手前かがはっきりします。

- **把持(grasping)** ── 漂う物体を**実際に掴み、回転を止めて、保持する**物理。この toy は**一切モデル化していません**。やっているのは「手先が的の半径内に入ったか」という**近接判定だけ**。触れても掴んではいないので、現実なら次の瞬間に弾かれて終わりです。**「把持ができないと回収は成立しない」── まさにその通り**で、ここがいちばん大きな“手前”です。
- **軌道力学** ── 本物の回収は、相対軌道運動・推進剤・遠距離センシングが本体で、そこは MuJoCo の外。ここは重力を切った**その場の腕**でしかありません。

つまり「宇宙ゴミ回収」と名乗るのは誇張でした。**実体は、微小重力に置いた到達デモ**です。この記事は「派手な看板を疑え」を掲げているのに、看板を出しかけた ── これも正直に取り下げます。そのうえで、この地味な到達 toy からでも拾えた発見が 2 つありました。

### 拾えた発見 1 ── ここでも「幾何の壁」

的に手先を届かせられるかは、やはり**体の長さ**で決まりました。**1〜2 節では届かず**(0.55 m / 0.25 m 手前)、**3 節以上でようやく的に触れる**(手先〜中心 約 0.08 m ≤ 的の半径 0.12 m)。触れられる最小の体は **3 節**。§6.5 と同じ、身も蓋もない幾何です。

### 拾えた発見 2 ── 「冗長性は効くはず」が外れた(正直な負け)

私はここで **「壊れにくさ(冗長性)は効くはず」** と踏んでいました。宇宙のロボは修理に行けないので、**関節が 1 つ壊れても手先を届かせられること**が命のはず。そこで、いろいろな飛来のしかた ×**関節を 1 つわざと殺す(dead servo=その関節が脱力する)** で頑健性を測りました。

ところが ── **冗長性の利点は、見えませんでした**。この難度では**「3 節以上」という幾何要件が支配的**で、それさえ満たせば、関節が 1 つ死んでも **3〜6 節すべてが 100% 触れて**しまう。届く余裕が大きすぎて、壊れにくさの差が埋もれたのです。

**仮説倒れ**です。でも消しません ── **「冗長性が効く」を見たいなら、課題を“ギリギリ”にしないと測れない**(的の半径を縮める/速く飛ばす/複数同時に来させる)。余裕のある問題では、冗長性は測定に現れない。把持すら無い到達 toy でも、**設計の教訓**は 1 つ拾えた、というわけです。

## 8. 作って分かったこと(教訓)

1. **プラットフォームは「差し替えられる軸」で設計する**。世界モデルの流儀・環境・ランタイムを部品化しておくと、世界モデルも人工生命も宇宙ゴミも、**同じ土台の使い回し**で試せる。
2. **「想像の中のリターンが上がる」と「実環境で良くなる」は別物**。世界モデルの搾取は実装が正しくても起きる。だから評価器で実環境リターンとドリフトを**常に**測る。
3. **易しい環境の好成績を一般化して語ってはいけない**。合成環境で勝っても Pendulum で勝つわけではない。**内訳を疑う**。
4. **単一目的の最適化は、いちばん単純な解に縮む**。複雑さ(多節)を見たければ、**多様性(QD)・感覚・課題設定**が要る。「複雑さ」はタダでは生まれない。
5. **物理には、賢さで越えられない幾何の壁がある**。届く/届かないは体の長さで決まる。制御の工夫はその後。
6. **仮説が外れたら、それ自体が結果**。冗長性が見えなかったのは失敗ではなく「課題をギリギリにしないと冗長性は測れない」という発見。**負けをそのまま載せると、次の実験が設計できる**。

## 9. 次回に続く ── プラットフォームの本命は「世界モデル × 進化」

次は、この 2 つの顔を**つなぎ**ます。§5 で見た「世界モデルの搾取」を手なずけつつ、§6-7 の進化を**世界モデルの“想像”の中で加速する** ── 候補の体や動き方を大量に「夢」の中でスクリーニングし、良さそうな少数だけ本物の物理で検証する、という筋です。高価な物理計算を、安い「夢」で間引く。これこそ、世界モデルをプラットフォームにした本当の狙いです。そのとき何が壊れて、何を測るべきか。……たぶん、また評価器が私を試してくるはずです。
