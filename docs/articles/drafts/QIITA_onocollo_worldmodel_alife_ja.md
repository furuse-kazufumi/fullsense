# 自宅CPUで「世界モデルを作るためのプラットフォーム」を作り、その上で人工生命を進化させ、正直にコケた話

> GPU なしのノート PC(CPU のみ)で、**世界モデル(world model)を組み立てるためのプラットフォーム**を作っています。この記事で出す数値・図・GIF は、**今回この場で再実行したものではなく、既に取得していた実行結果とログを整理したもの**です。そのうえで、世界モデルを学習し、それを **ROS 風のランタイム**の中で動かし、さらに同じ土台の上で **単細胞から進化する人工生命**を育て、最後にその腕を**微小重力の的**に向けて「壊れにくさは効くのか」を測るところまでを、**既取得の本物の実行結果**つきでたどります。うまくいった所も、**うまくいかなかった所も隠さず**出します。むしろ後者がこの記事の主役です。

## この記事の 3 行

- 作っているのは単発のデモではなく、**世界モデルを「部品として組み立て、測り、ロボットの脳に差し込む」ための小さなプラットフォーム**(`onocollo`)です。frontier(Genie / Sora 級)は家庭で作れないので、**「誰でも CPU で世界モデルを構築・実験できる土台」**に土俵を移しています。
- その土台の上で、世界モデルを **ROS っぽい配線**に載せ、さらに **人工生命の進化**まで回せます。流体の中では多細胞の体が単細胞の **6.5 倍**進む、という多細胞化が報われる瞬間まで見えました。
- 最後に、その到達アームを**微小重力の的**に向けてみました。頭では「宇宙ゴミ回収」を夢想しましたが、正直に言えば**把持(掴む力)すら無い到達 toy**で、むしろ「壊れにくさ(冗長性)は効くはず」という自分の仮説の方が**実測でひっくり返りました**。全部ノート PC の CPU で、数秒〜数分です。

はじめての方は上から、急ぐ方は「2. プラットフォームとは」「5. 正直な現実」「7. 微小重力の到達」だけでも要点が取れます。

## 2. onocollo とは ─ 世界モデルを作るための「プラットフォーム」

**onocollo**(おのころ)は、私が自宅 CPU で作っている OSS で、狙いは 1 つの派手なデモではなく、**世界モデルを組み立てるための土台(プラットフォーム)**です。名前は古事記の **淤能碁呂島(おのころじま)**(かき混ぜた原初の海が凝り固まって最初の島=世界が生まれる)と、コロイド化(colloidal=凝集)を掛けた造語で、「潜在空間がかき混ざって世界(モデル)が生まれる」イメージです。

### なぜ「プラットフォーム」と呼ぶのか

世界モデルの研究には、実は**入れ替え可能な軸が 3 本**あります。onocollo は、その 3 本を**それぞれ差し替え可能な部品**にして、どう組み合わせても同じ数行の API で回るようにしました。

1. **世界モデルの流儀(アルゴリズム軸)**。古典の **World Models**(Ha & Schmidhuber, 2018 ── VAE で見て、MDN-RNN で記憶し、進化戦略で制御する **V / M / C** の 3 分割)と、現代の **Dreamer**(Hafner ら ── **RSSM** という 1 本の潜在動力学 + 想像の中で学ぶ actor-critic)。**この 2 系譜を同じインターフェースで載せ替え**られます。
2. **環境(タスク軸)**。依存ゼロの合成環境・**実物理の MuJoCo**(ヘッドレス CPU)・任意の **gymnasium** 環境(連続 Box も離散 Discrete も自動判別)を、**設定 1 つで差し替え**。世界モデル側のコードは一切変わりません。
3. **動かす場所(ランタイム軸)**。学習した世界モデルを、**ROS 風のグラフ**(後述)や **MuJoCo 物理**の中に、**一級の部品(ノード)**として差し込めます。

```python
import onocollo as oc

env   = oc.GymEnv("Pendulum-v1")     # 環境はここだけ差し替える
agent = oc.Dreamer(env, config="tiny")   # 流儀もここだけ差し替える(oc.WorldModel も可)
agent.fit(steps=2000)                # 収集→世界モデル学習→行動学習
dream = agent.imagine(horizon=15)    # 世界モデルの中だけで未来をロールアウト
```

**「アルゴリズム × 環境 × ランタイム」を自由に組み替えられる**── これがプラットフォームと呼ぶ理由です。今日は tiny な CPU 設定ですが、**スケールアップは設定変更であって書き直しではない**(重い依存 `mujoco` / `cma` / `gymnasium` は遅延読み込みの optional)。CPU で組み上げて、そのまま 1 枚の GPU に載せる道が最初から通してあります。

### プラットフォームとしての「誠実さ」

もう 1 つ、これがただの実験集と違うのは、**「都合の悪い数字を測って見せるための評価器」がプラットフォームに組み込まれている**ことです(§5)。世界モデルは黙っていると「自分の夢の中でだけ賢くなる」ので、**実環境での成績と、予測のドリフト(ズレ)を常に測る**部品を標準装備しました。frontier のベンチ性能では勝てません。だから **「最初の“夢”までの時間の短さ」と「正直さ」で勝つ**、という設計です(全体 460 テスト緑・Apache-2.0)。

## 4. `onocollo.ros` ─ 世界モデルを「脳の部品」として配線する

世界モデルは「考える」部品。プラットフォームの価値は、それを**ロボットの脳の配線**の中に、他の部品と対等に差し込めることです。世界モデルを **`/imagine` サービス**として提供すると、他のノードはそれを他のどんな ROS サービスとも同じように呼べます。

```python
from onocollo.ros import Graph, EnvNode, PolicyNode, WorldModelNode, ServiceClient

g = Graph()
g.add_node(EnvNode(env, name="plant"))          # /obs 配信・/action 購読(=本体)
g.add_node(PolicyNode(agent, name="ctrl"))      # /obs 購読・/action 配信(=制御)
g.add_node(WorldModelNode(agent, name="wm"))    # /imagine サービス(=想像)
g.spin(count=200)                               # 観測→考える→動く を決定論的に

dream = ServiceClient(g, "/imagine").call({"horizon": 15})   # 世界モデルを"サービス"として呼ぶ
```

「ROS 系でできることは全部できる」を目指しましたが、**嘘はつきません**。開発・シミュレーション・学習で ROS を使う場面は onocollo で完結します。しかし実機のドライバ・ハードリアルタイム・DDS 配線互換は **持ち込み(bring-your-own)** で、そこは再実装したと名乗りません。

| ROS 2 の機能 | onocollo | パリティ |
| --- | --- | :---: |
| ノード / トピック / pub-sub | `Graph` + `Node` | ✅ プロセス内 |
| 実行器 / spin / タイマ | `Graph.spin`, `Rate` | ✅ ステップ+ソフトRT |
| tf2 変換 | `TransformBuffer`(補間つき) | ✅ |
| rosbag2 記録/再生 | `Bag` + `Recorder`/`Player` | ✅(独自形式) |
| サービス / アクション | `Service` / `Action` | ✅ 同期 |
| ros2_control ハードウェア | `HardwareInterface` + `SimHardware` | ⚠️ **IF+sim。実ドライバは自作** |
| DDS 探索 / マルチホスト | — | ❌ 単一プロセス |
| ハードリアルタイム制御 | ソフトRTのみ | ❌ RTスケジューラではない |

**ハードウェア線より上**(認識・世界モデル・方策・tf・bag・launch)は、相手が `SimHardware(env)` でも実機の `HardwareInterface` でも **同一**です。脳全体を CPU で開発・シミュレーションし、最後にドライバだけ差し込む ── ここが「プラットフォーム」の実利です。ROS 2 そのもの(実機ドライバ・DDS・ハード RT)の代替ではありません。表がそう言っています。

さらに、実物理の **MuJoCo エンジンを型付きで包む層(`onocollo.mujoco`)** も同じ土台に載っていて、`MujocoBridgeNode` を挟めば **MuJoCo の世界を上の ROS グラフと同じノードで**「観測→考える→動く」できます。物理・配線・世界モデルが 1 つのプロセスに揃う、というわけです。

## 5. 測って分かった正直な現実(この記事の主役 その 1)

**異常に良い結果が出たら、勝った気になる前に内訳を疑う** ── これは私の開発の家訓です。逆に、悪い結果も消しません。

### 5.1 「夢の中では」ちゃんと学習している

合成環境で世界モデルを学習させた実測(120 反復・CPU):

```
world-model loss   : 71.87  ->  19.57     # 下がる = 動力学を学んでいる
imagined return    : -0.398 ->  10.287    # 想像の中の報酬は上がる
greedy real return : 7.359   vs random 2.697   # 易しい合成環境では random に勝つ
```

合成環境(=学習しやすいように設計した玩具)では、**学習済み方策がランダムに勝ちました**(7.36 vs 2.70)。**配線は正しく、学習信号は本物**です。

### 5.2 でも実タスクでは、ランダムに負ける

同じ API を、**本物の制御タスク**に向けるとどうなるか。振り子(Pendulum-v1, 連続)と倒立振子(CartPole-v1, 離散)での実測(80 反復・CPU):

| 環境 | 行動空間 | ランダム | 学習後 | 差 |
| --- | --- | ---: | ---: | ---: |
| Pendulum-v1 | 連続 | **-1158.5** | -1406.1 | **-247.5** |
| CartPole-v1 | 離散 | **14.0** | 9.6 | **-4.4** |

**学習後の方が悪い。** これは失敗ではなく、世界モデルの本質が現れた瞬間で、名前がついています ── **世界モデルの搾取(world-model exploitation)**。actor は「学習途中の・まだ不正確な報酬予測」を最大化するので、**想像の中のリターンは上がるのに実環境のリターンは下がる**ズレが起きます。易しい合成環境では出ない。実タスクでは出る。

### 5.3 ドリフト曲線が「世界は動かない」にすら勝てない現実を暴く

プラットフォームには、この不都合を**隠すためではなく、測って見せるため**の評価器が標準装備されています。世界モデルに、実環境と同じ行動列を「観測を見ずに」ロールアウトさせ、1 ステップごとの予測誤差を実測します。

![開ループのドリフト曲線。モデルの予測誤差はpersistence(世界は動かない)ベースラインより数百〜千倍大きく、曲線は右肩上がりでなく平坦](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/drift_curve.png?v=1)

読み方:**persistence(「世界は動かない」と予測するだけのベースライン)より、モデルの誤差の方が数百〜千倍大きい**(`model/persist ≫ 1`)。この小さいモデルは再構成誤差の床に張り付いていて、**「世界は動かない」という一番バカな予測にすら負けている**。教科書的な「開ループ誤差の累積」は、テストでは劣化させた予測器で示せますが、**この規模の本物のモデルでは起きません**。評価器は、起きた事実だけを報告します。

### 5.4 「もっと計算すれば良くなる」を測ったら、逆に悪化した

「反復を増やせば実制御も良くなるだろう」と思って、Pendulum で **反復 80 vs 反復 300(モデルも大きい)** を実測しました。予想は外れました。

| 設定 | 実 eval(高いほど良い) | world-model ロス | 想像リターン |
| --- | ---: | ---: | ---: |
| ランダム | **-1158.5** | ─ | ─ |
| 80 反復 | -1406.1 | 66.9 → 17.4 | 0.42 → **-24.8** |
| 300 反復 | **-1522.2** | 75.8 → **4.0** | 0.02 → **-154.5** |

**計算を増やしたら、実制御は悪くなりました**(-1406 → -1522)。しかも内訳が示唆的です:計算を増やした方が **world-model ロスは最もよく下がっている**(4.0)。つまり **動力学の予測は一番よく学べている**のに、実環境の成績は最悪。動力学が良くなるほど、actor は「まだ不完全な報酬予測」の穴を**より巧妙に突ける**ようになる。**「もっと回せば良くなる」は、少なくともこの規模の Pendulum では偽**でした。

> これこそ、正直な評価器を**プラットフォームの一部として最初から**作っておく価値です。もし実環境リターンを測っていなければ、「ロスが 4.0 まで下がった!学習成功!」と誤って喜んでいたでしょう。

## 6. 同じ土台の上で、人工生命を進化させる(この記事の主役 その 2)

プラットフォームの面白さは、**世界モデルとは別の遊びも、同じ MuJoCo 土台の上で回せる**ことです。ここでは **Karl Sims の「進化する仮想生物」**(1994)の系譜を継ぐミニ版(`onocollo.evolve`)を載せました。

やることはこうです。**体のゲノム**(自由に動く根 + 波打つ関節の“節”の鎖)を、手続き的に**本物の MuJoCo モデルへ変換**し、動かして、その成績で選ぶ。突然変異で節を増やしたり減らしたり ── だから **体そのものと歩き方が同時に進化**します。1 節の“単細胞”から始めます。

### 6.1 貪欲な進化は「単細胞」に縮む

まず素朴な `(μ+λ)` 進化(良いものを残して変異、を繰り返す)。平地を這わせると、**0.009 m → 0.62 m** と 20 世代でぐんぐん伸びます。……が、行き着く先は **1 節への収束**でした。平地の短距離走では余計な節はむしろ邪魔で、いちばん単純な体がいちばん速い。**単一目的の最適化は、最小の体に縮む**のです。

### 6.2 QD(品質多様性)で「体サイズ別の名人」を残す

そこで、前回の記事([AI に料理コンテストをさせたら…](https://qiita.com/furuse-kazufumi/items/40ba7cc91ac577274b74))で書いた **MAP-Elites / QD** の出番です。「総合 1 位」だけを追うと多様性が死ぬので、**(節数, 姿勢)ごとにチャンピオンを別々に残す棚**を使います。すると 1 節に潰されず、**全 6 種類の体サイズのチャンピオンが同時に生き残ります**(1 節 +0.108 m … 6 節 +0.005 m)。ピーク性能と引き換えに多様性を守る ── これが QD のトレードオフです。

### 6.3 感覚フィードバックで、多節が「使える」体になる

次に、各関節が**自分のセンサと歩調の時計を読んで動く小さな神経回路(閉ループ制御)**を載せ、体と一緒に進化させます(これが Karl Sims 本来の形)。すると多節の体が**競争力を持ち、走行の多くの世代で先行**しました(2〜3 節がリード・最良 +0.27 m)。開ループが 1 節に縮んだのと好対照です。

### 6.4 水の中で、多細胞化が「決定的に」報われる

いちばん綺麗に出たのがこれです。**重力を切って粘性のある流体**にすると、多節の体は**進行波(travelling wave)**でぐいぐい進み、1 節はほとんど動けません。QD で測ると **6 節 +0.92 m 対 1 節 +0.14 m ── 6.5 倍**。ここで初めて、**多細胞であることが決定的に得**になりました。面白いのは、貪欲な `(μ+λ)` は流体でも 1 節に収束してしまうこと。**多様性を守る QD だからこそ、この多節の最適解を見つけられた**のです。

![粘性流体の中で多節の生物が進行波で泳ぎ、1節はほとんど動けない様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/swimmer.gif?v=1)

### 6.5 「届く/届かない」は体の長さで決まる ── 幾何の壁

もう 1 つ、体長が物理的に効くタスク。根を固定して、先端を遠くの目標に届かせます。目標を **0.85 m** に置くと、**1 節は 0.55 m も足りず**、**3 節以上でようやく届く**。ここでは巧妙な制御より前に、**体の長さという幾何(geometry)の壁**が立ちはだかります。「賢さ」ではどうにもならない、身も蓋もない物理です。

![根を固定した腕が遠くの目標(緑のマーカー)に手を伸ばす。1節は届かず、多節が目標に接触する比較](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/reach.gif?v=1)

> **正直な但し書き:** ここでの「単細胞」は**体の 1 節**のことで、細胞生物学(膜・代謝・分裂)ではありません。物理も toy スケールの剛体です。それでも「単一目的の進化は最小の体に縮み、多様性(QD)と感覚と課題設定が多細胞化を報いる」という **筋の通った物語**が、ノート PC の CPU で数分で観察できたのは、私には十分に面白い結果でした。

## 7. 同じアームを微小重力で試す ── そして「回収」には把持が要る、という現実

§6 の「到達アーム」を、そのまま**微小重力(重力オフ)**に置いて、漂ってくる的(タンブリングする箱)に手を伸ばさせてみました。頭の中では「宇宙ゴミ(デブリ)回収」を夢想した実験です。

![微小重力で、固定ベースのアームが漂う的(金色の箱)に手先を届かせる toy。掴んではおらず、手先が的の半径に入ったか(近接)だけを判定している](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/space_debris_capture.gif?v=1)

……ただ、正直に言うと **これは「ゴミ回収ロボ」ではありません**。§6 と同じ到達タスクを、微小重力の的に向けただけ。そして ── お察しのとおり ── **「回収」には決定的に足りないものがあります。掴む力(把持 / grasping)です。**

### まず「回収」の現実 ── 触れることと、掴むことは別物

本物のデブリ回収に最低限要るものを並べると、この toy がどれだけ手前かがはっきりします。

- **把持(grasping)** ── 漂う物体を**実際に掴み、回転を止めて、保持する**物理。この toy は**一切モデル化していません**。やっているのは「手先が的の半径内に入ったか」という**近接判定だけ**。触れても掴んではいないので、現実なら次の瞬間に弾かれて終わりです。**「把持ができないと回収は成立しない」── まさにその通り**で、ここがいちばん大きな“手前”です。
- **軌道速度と軌道力学** ── 本物のデブリは**秒速数 km の軌道速度**で飛んでいて、回収機は推進剤を使って**相対速度をほぼゼロまで合わせにいく**相対軌道運動が本体です。この toy は重力を切って**ゆっくり漂う的**を置いただけ。その高速も軌道力学も、MuJoCo の外です。

つまり「宇宙ゴミ回収」と名乗るのは誇張でした。**実体は、微小重力に置いた到達デモ**です。この記事は「派手な看板を疑え」を掲げているのに、看板を出しかけた ── これも正直に取り下げます。そのうえで、この地味な到達 toy からでも拾えた発見が 2 つありました。

### 拾えた発見 1 ── ここでも「幾何の壁」

的に手先を届かせられるかは、やはり**体の長さ**で決まりました。**1〜2 節では届かず**(0.55 m / 0.25 m 手前)、**3 節以上でようやく的に触れる**(手先〜中心 約 0.08 m ≤ 的の半径 0.12 m)。触れられる最小の体は **3 節**。§6.5 と同じ、身も蓋もない幾何です。

### 拾えた発見 2 ── 「冗長性は効くはず」が外れた(正直な負け)

私はここで **「壊れにくさ(冗長性)は効くはず」** と踏んでいました。宇宙のロボは修理に行けないので、**関節が 1 つ壊れても手先を届かせられること**が命のはず。そこで、いろいろな飛来のしかた ×**関節を 1 つわざと殺す(dead servo=その関節が脱力する)** で頑健性を測りました。

ところが ── **冗長性の利点は、見えませんでした**。この難度では**「3 節以上」という幾何要件が支配的**で、それさえ満たせば、関節が 1 つ死んでも **3〜6 節すべてが 100% 触れて**しまう。届く余裕が大きすぎて、壊れにくさの差が埋もれたのです。

**仮説倒れ**です。でも消しません ── **「冗長性が効く」を見たいなら、課題を“ギリギリ”にしないと測れない**(的の半径を縮める/速く飛ばす/複数同時に来させる)。余裕のある問題では、冗長性は測定に現れない。把持すら無い到達 toy でも、**設計の教訓**は 1 つ拾えた、というわけです。

### だから、これは「別プロジェクト」に切り出します

ここまで正直に書いたとおり、いまの実装は**回収を名乗るには手前すぎ**ます。そこで「微小重力での回収」は、この記事の toy で終わらせず、**独立したプロジェクトに切り出すことにしました**。看板倒れにしないために、次は足りないものを本気で足します ── **把持(実際に掴んで回転を止める物理)・センシング(漂う的をどう認識するか)・制御**。そして「どうすれば回収が**本当に成立する**のか」を、あてずっぽうでなく **進化アルゴリズムで追い込む**研究スタイルにします。ここで得た「幾何が支配する/課題をギリギリにしないと冗長性は測れない」という学びが、その最初の設計図です。**看板を出す前に中身を作る** ── この記事の家訓を、次はプロジェクトの進め方そのものに効かせます。

## 7.5【追記】切り出した先で、宇宙ゴミ捕捉を「進化」で深掘りした ── 正直な続報

§7 で「別プロジェクトに切り出す」と書いた宇宙ゴミ捕捉を、その後じっくり進化させてみました。最初に開示しておくと、以下は**今回この場で再実行した検証ではなく、既に取得していた実行結果・ログ・GIF を整理した続報**です。まず正直な前段: 最初の把持は実は**ファサード(見せかけ)**でした ── 箱がジョーの外で溶接され、指も動いていなかった。そこを**実摩擦の把持**(ジョーを的の軸に挿し込み、指を実際に閉じて両側の接触が続いたら「捕捉」)に書き直したうえで、腕の形を進化させています。

位置づけを一言で言うと、これは「**grasping は疎な報酬・疎な相互作用で難しい**」という既知の流れの、**微小重力 + 故障注入 + 形態進化**版です。QD の grasping 先行研究(Huber ら, 2023)は、**15 手法を 10 の grasping domain で比較し、成功解を優先して残す MAP-Elites 変種が、その比較設定では studied metrics で大きく優位だった**と要約できます。MAP-Elites 自体も、総合 1 位 1 個ではなく**ユーザが選んだ行動記述子ごとに高性能解の地図を作る**設計です(Mouret & Clune, 2015)。今回の違いは、その棚を**体サイズごと**に持ちながら、「触れたか」ではなく**捕捉・de-tumble・故障時の劣化**まで同じ土台でまとめて測っている点です。

### まず「つかむ」証拠

![宇宙ゴミを掴んで運ぶ](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_grasp.gif?v=1)

近づいて、ジョーを差し込んで、閉じて、保持して、回転を止める(de-tumble)。重い箱(5kg)でも掴めますし、失敗するときは正直に失敗します ── 速すぎる的は逃げ、関節が固着すると届きません。

![5kg の重い箱でも掴む](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_heavy.gif?v=1)
![正直な失敗①: 速い飛来は逃げる](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_escape.gif?v=1)
![正直な失敗②: 固着した関節では届かない](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_frozen.gif?v=1)

### 正直な驚き ① ── 今回の設定では `greedy < random < QD` になった

同じ評価予算で 3 手法を比べました(耐故障目的 = ランダムなゴミの飛来 × 各関節を 1 個ずつ固着させたときの捕捉率)。

![手法比較: greedy vs random vs QD](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_methods.png?v=1)

| 手法 | 最良 fitness(平均[最小,最大]) | 5 seed 中の勝ち |
|---|---|---|
| ランダム探索 | 0.59 [0.50, 0.80] | 2 |
| 貪欲な (μ+λ) 進化 | 0.55 [0.50, 0.58] | **0** |
| QD(品質多様性) | **0.72** [0.50, 0.80] | 3 |

**今回の 5 seed・今回の評価予算では、平均値として `greedy < random < QD` になりました。** ただしレンジは一部重なっており、ここから **「QD が常に優位」** までは言えません。観測事実として言えるのは、**この設定では貪欲進化が最下位で、QD が最良平均だった**ことです。

因果の解釈は、現時点では**探索経路依存を示唆する仮説**として置くのが正確です。故障に強い設計が 4〜6 節側に多く、1 節から少しずつ変異する貪欲探索はそこへ届きにくい可能性があります。実際、**貪欲を最初から 4 節で始めると 0.55→0.68 に改善**したので、少なくとも初期化と探索経路が効いていることは示唆されます。ただし、これだけで **「探索の問題であって適応度地形の問題ではない」** と断定はしません。ここで言えるのは、**今回の設定では QD 的な多様性保持が効いた**、というところまでです。

### 体サイズ別の「名人図鑑」

![QD レパートリー: 体サイズごとの名人](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_zoo.gif?v=1)

QD は 1→6 節それぞれに名人を残します。1 節は短すぎて失敗し、3 節と 6 節が 0.75 近辺まで伸びました ── 「体の大きさごとの最適な掴み方」が並びます。

### 正直な驚き ② ── 「冗長性は効くはず」は、単純には効かなかった

各名人を「関節故障あり/なし」で採点した、冗長性の代償カーブです。**図は 1〜6 節を全部描いていますが、表は本文で触る代表点だけ**を抜いています。1 節はそもそも捕捉が成立せず、4 節は後段の高速タンブリングで主役になるため、ここでは **2/3/4/5/6 節**を並べます。

![冗長性の代償カーブ](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_redundancy.png?v=1)

| 節数 | 故障なし | 故障あり | 代償 |
|---|---|---|---|
| 2 | 0.83 | 0.56 | +0.27 |
| 3 | **1.00** | 0.75 | +0.25 |
| 4 | 0.92 | 0.74 | +0.18 |
| 5 | 0.67 | 0.61 | **+0.06(最小=最も優雅に劣化)** |
| 6 | **1.00** | 0.74 | +0.26 |

**代償は節数に単調ではありません。** 5 節がいちばん優雅に劣化するが素の性能は低め、3 節は素で強いが代償がやや大きい、4 節は中間、6 節は関節が最多なのに代償が大きい。故障を生き延びられるかは「**どの**関節が壊れ、残りの幾何で補えるか」で決まり、単に「関節が多いほど良い」ではない ── §7 で「冗長性は効くはず が外れた」と書いた話の、今回設定での裏取りです。

### 正直な発見 ③ ── 速く回るゴミ、そして「掴んだ」と「止めた」は別物

高速タンブリング(自転)を掃引すると、**「掴んだ(gripped)」と「回転を止めた(de-tumbled)」が乖離**します。速いタンブラーはジョーの中で**ガタつきながら掴まれている**だけのこともある(掴めても回転は止まっていない)。

![タンブリング frontier](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_tumble.png?v=1)
![速いタンブラーを止める 4 節名人](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_fast_tumble.gif?v=1)

それでも生き残った良い結果: QD の 2 節・4 節名人は **2.0 rad/s まで ~100% de-tumble** できます。一方 3 節は ~1 rad/s まで、6 節は不安定 ── ここでも「多ければ良い」ではありません。**捕捉(gripped)できること**と、**回転まで止め切る(de-tumbled)こと**は別指標だ、と分かったのが重要でした。

### ゴミの中で掴む(clutter)

![ゴミ畑の中で的だけ掴む](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/adr_clutter.gif?v=1)

まわりにダミーのゴミ(灰色)を散らすと、ジョーの通り道にある物が把持を邪魔します。ここでも貪欲が最下位、ランダムが安定 ── パターンは変わりません。

### 正直な発見 ④ ── 「腕の形」と同じくらい「制御」が効く

形を固定して**制御方針**(差し込み前の standoff / 挿入速度 / 速度ゲート)だけ調整すると、難しい飛来(速いドリフト+タンブル)での捕捉が **既定 40% → 調整後 100%(+60 ポイント)**。さらに開ループ→反応閉ループ(ライブでゴミを見て Jacobian でサーボ)で、素の捕捉 0.58→0.75、ドリフト待ちの局面 0.00→0.50 と改善しました。**ただし正直に**: その上に載せた**学習型コントローラは、強い手設計の反応ベースラインを超えませんでした**(この小さな CPU 予算では「学習した先読み」が hand-set を上回る証拠はまだ無し)。制御は形態と並んで成否を決める、しかし「学習=勝ち」ではない ── というのが、今の正直な結論です。

## 7.6【追記】同じ MuJoCo 土台で、今度はロケットを着陸させてみた

最近は同じ `onocollo` の MuJoCo 土台で、**レトロプロパルシブな垂直着陸**も試しました。剛体のブースターが重力下で落ちてきて、**ジンバル付き主エンジン**だけで減速し、まっすぐ・ゆっくり・着地点中央へ降りる toy です。位置づけとしては `onocollo` の「形態固定で制御を進化させる」枝で、宇宙ゴミ捕捉とは逆に **形ではなく制御**を主役にした例です。

![ジンバル付き主エンジンでブースターを垂直着陸させる toy。まずは手設計の PD 制御で着陸できることを確認した](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/rocket_land.gif?v=1)

![発射から上昇、反転、降下、着地までを 1 枚で示す static-safe 補助図。Qiita では GIF が本編、こちらは流れの要約として使う](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/motion_pack/rocket_launch_land_cycle.svg?v=1)

この補助図は **公開時に assets 側が `main` へ反映済みであることが前提**です。Qiita では GIF を本編、static-safe SVG を要約図として併用します。

ここで面白かったのは、**regime を分けて見ると話が変わる**ことです。強い手設計ベースラインとして PD をきちんと調整すると、32 episode の **nominal 条件**では **land_rate 1.00 / soft_rate 約 0.85** まで出ます。その上で zero-init の残差 MLP を `(μ+λ)` 進化で載せると、**nominal では** `76.60 → 77.78 (+1.5%)` の **modest win** が出ました。これは honest に「少し効いた」であって、ここ自体は null ではありません。

崩れるのは **persistent wind を入れた hard 条件**です。こちらでは memoryless な学習残差が、訓練側では **`52.8 → 66.4` (+26%)** や **`52.2 → 59.0` (+13%)** と大きく伸びるのに、held-out へ出すと tuned PD と **同等か微妙に負ける**ところまで落ちます。つまり transfer 失敗は nominal の `+1.5%` とは別件で、**風あり regime に特有の overfit** です。mechanistic には理由がはっきりしていて、**定常風を打ち消すには積分や内部状態が要るのに、memoryless な残差にはそれが無い**からです。

そこで PID の積分項を足すと、wind 下でも **soft landing rate が 0.09→0.16 / 0.12→0.16 / 0.03→0.06** と modest に改善しました。ここまでは「状態を 1 つ足すだけでも、定常外乱には意味がある」という素直な話です。さらに学習側にも recurrent policy を与えると、**訓練では** `PID 53.8 → recurrent 65.1 (+21%)` と大きく勝ちます。ところが **held-out では 1 勝 1 敗 1 分**で、しかも **soft landing は全敗**。つまり、学習に内部状態を与えても、有限の評価 episode の上では **objective をずらす方向へ過学習し得る**ことまで見えました。

宇宙ゴミ捕捉で見た「学習=自動的に勝ちではない」が、別タスクでもそのまま再現されたわけです。公開読者が辿れる参照としては、この FullSense repo の `main` ブランチ上にある [`rocket_land.gif`](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/rocket_land.gif?v=1) と、同じ節に置いた static-safe 補助図を使います。実装・生ログ・数値の canonical source 自体は repo 外で管理しているため、この場では **公開読者が到達できる公開 asset と本文参照**を先に置いたうえで、`onocollo` が **世界モデル・人工生命・把持だけでなく、制御タスクの honest benchmark 置き場としても使える**ことを示す recent update として触れました。差別化点は「ロケットを着陸させた」こと自体ではなく、**強い手設計 baseline を先に置き、regime ごとに held-out で学習の勝ちを取り消せるようにしてから語る**、という評価の作法です。

## 7.7【追記】同じ土台で「多様な体」を動かす ── 蛇・全身骨格・虫・四足、そして自分で見破った 3 つの「偽の成功」

ここまでは腕・アーム・ロケットと、どちらかといえば「1 つの体」を掘り下げてきました。今度は同じ `onocollo` の MuJoCo 土台の上で、**まったく違う形の体を 4 つ**動かしてみます ── くねる蛇、700 本の筋肉の全身骨格、羽ばたく虫、四足のロボット犬。狙いは「たくさんの体で動かせた!」という自慢ではありません。**この 4 つのうち 3 つで "うまくいったように見える偽の成功" が途中で出て、それを公開前に自分で見破って捨てた**のです。今回の主役は、その**偽陽性(false positive)の潰し方**の方です。

> 追加の用語(この節だけで使います)

| 用語 | 読み/英語 | ひとことで言うと |
| --- | --- | --- |
| 異方性摩擦 | Anisotropic friction | 「前には滑りやすく、横には滑りにくい」向きで変わる摩擦。蛇が脚なしで進める理由。 |
| 抵抗力理論 | RFT (Resistive Force Theory) | 接地を「向きで変わる抵抗力」に置き換えて、脚なしのくねり移動を近似する古典モデル。 |
| serpenoid | ─ | 蛇の胴を進む波(travelling wave)で表す古典的な蛇型歩容。振幅・周波数・関節間の位相差で決まる。 |
| 筋シナジー | Muscle synergy | 100 本の筋肉を少数(K 本)の「まとめて動くパターン」に圧縮した表現。 |
| 報酬ハッキング | Reward hacking | AI が「点数の抜け穴」を突き、狙った行動をせずに高得点だけ取ってしまうこと。**この節の裏の主役**。 |
| held-out | ─ | 学習に使っていない初見の条件で測ること。過学習(狙い撃ちの過剰適応)を暴くための物差し。 |

### 体① 蛇の這行 ── 「異方性摩擦がある時だけ前に進む」を実測(成功)

蛇には脚も車輪もありません。ではなぜ前に進めるのか。答えは**摩擦の向き**です。胴のうろこは「前にはスッと滑るが、横には引っかかる」。この**異方性摩擦(anisotropic friction)**があるから、胴を横にくねらせた波が、正味の前進に変わります。逆に摩擦が**等方(どの向きも同じ)**だと、いくらくねっても、その場でうねうねするだけで一歩も進めません。

ここで正直な落とし穴が 1 つ。**MuJoCo の接触摩擦は接線方向には等方**(1 つのスカラーで決まる)なので、**そのままでは蛇の這行を再現できません**。そこで脚なし移動の定番、**抵抗力理論(RFT)**を使います ── 地面との接触を、胴の各分節ごとの「向きで変わる抗力」に置き換えるのです(前方向の抗力 `c_t` より、横方向の抗力 `c_n` をずっと大きくする)。胴は平面上の分節チェーンにして、古典的な **serpenoid の進む波**で駆動し、この異方性抗力を毎ステップ加えます。制御パラメータはたった 4 つ(振幅・周波数・関節間の位相差・曲がりバイアス)。これを CMA-ES で進化させました。

そして、この記事の流儀どおり **ablation(切り分け実験)**で機構を確かめます。「異方性のとき」と「等方のとき」で、**同じ進化した歩容**がどうなるか:

| 歩容＼摩擦 | 異方性(横抗力が 10 倍) | 等方(前も横も同じ) |
| --- | --- | --- |
| **異方性で進化した歩容** | **+3.65 m 前進**(横ずれ 0.08 m=ほぼ直線) | **0.00 m**(その場) |
| 等方で進化した歩容 | −0.54 m(後ろ・横へ) | +0.37 m(退化した定在波) |

読み方はこうです。**同じ 1 つの歩容**が、異方性摩擦の下では 3.65 m まっすぐ這って進み、摩擦を等方にした瞬間、正味の移動がぴたりと **0.00 m** になります。しかも「等方でもがんばって進化させた最良の歩容」ですら 0.37 m しか出ず(異方性の約 1/10)、中身は前に進む波ではなく、ほとんどその場の定在波でした。**「異方性摩擦こそが、横のうねりを前進に変えている」**という教科書どおりの機構が、自宅 CPU 上でそのまま出ました。

![異方性摩擦(RFT)の下で、serpenoid 歩容の蛇が真っすぐ這って前進する(上から見た図。8 秒で約 5 m)](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/snake_crawl.gif?v=1)

![同じ歩容を「等方摩擦」に切り替えると、胴はくねり続けているのに正味では一歩も進まない(その場のうねり)。これが ablation の対照](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/snake_isotropic.gif?v=1)

(正直な注記:進化した周波数は探索範囲の上限に張り付きました=「もっと速く振りたい」。前進距離は安定していますが、上限を上げれば数字は変わり得ます。)

これは §9 で「いつか載せたい」と書いていた**多節の体で瓦礫を越える**の、最初の一歩でもあります。まだ平地の這行までですが、脚なし移動の土台はできました。

### 体② 全身骨格で「掴んで、指定した場所へ運ぶ」 ── 報酬ハッキングを潰して部分的に成功

§7.5 の腕とは別に、700 本の筋肉の全身骨格(別記事で詳報)の右手が、水ボトルを**掴んで持ち上げ**られることは実証済みでした。今回はその先 ──「**掴んだボトルを、命令した場所まで運ぶ**」に挑みます。方策は「目標 − ボトル」のベクトルを見て、同じ 1 つの制御で違う目標に従えるか。これができれば「制御可能な操作」の実証になります。

最初の素朴なやり方は、みごとに**報酬ハッキング**に落ちました。「ボトルが目標に近いほど加点」だけにすると、**掴むことへの勾配がゼロ**になるのです。ボトルの初期位置がもともと点数の底値を稼ぐので、最適化は「筋肉を弛緩させ、ボトルを置いたまま放置」して、動かないことで点を稼ぎました(前回、200 世代回しても目標への最接近距離が一切改善せず凍結していたのが、まさにこれ)。**点数は上がるのに、掴んでも運んでもいない**。

直し方は 2 つ。ひとつは **warm-start**:すでに検証済みの「掴んで持ち上げる」方策を初期値にします(観測の次元が違うので、命令ベクトルの入力重みをゼロで埋めて "最初は掴む方策とまったく同じ" にする)。すると進化の仕事は「動く把持の上に、命令追従だけを足す」ことになり、"放置ハック" の窪地を最初から飛び越えます。もうひとつは**把持維持の加点**と、終端ボーナスを**「掴み続けている時だけ有効」にゲート**すること(落としたり放り投げたら得点ゼロ)。

結果、点数は前述の "放置ハック" の底値 ~18.9 から **~1021 に跳ね上がり**、中身も本物になりました:

| 命令した目標 | 最接近 | 掴み続けた? | 目標に置けた?(12 cm 以内) |
| --- | --- | --- | --- |
| 真上 +0.30 m | **0.108 m** | ✓ | **✓** |
| 体側・飲むポーズ | 0.265 m | ✓ | ✗ |
| 前方やや上 | 0.120 m | ✓ | 境界 |

**3 つの命令すべてでボトルを掴み続け**(手のひらとボトルの距離 0.043〜0.046 m を維持)、命令の方向へ運びました。真上の目標は置くところまで成功、前方やや上は境界。**正直な限界**は「体側の飲むポーズ」で、ここは 0.27 m で届きませんでした ── 真上の把持から warm-start した**単一の線形シナジー**では、腕を体側へ引き戻す姿勢に容量が足りないのです。**掴みは完全、命令追従は 3 分の 1 が厳密成功**。誇張せず、部分的な「制御可能な操作」として置きます。

![全身骨格の右手が水ボトルを掴んだまま、命令された「真上」の目標へ運んで保持する。放り投げでなく、掴み続けたまま運んでいる](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/musculo_place.gif?v=1)

### 体③ 虫の飛行 ── ホバリングは未達、しかも前の「飛べた」は勘違いだった(正直な訂正)

MuJoCo には空力つきのショウジョウバエモデル(flybody)があります。翼だけを動かすホバリング制御を進化させ、honest disclosure に従って内訳を検証したところ、**過去の自分の主張を訂正する**羽目になりました。

以前「翼を 150 Hz で振ると胴が +0.97 cm 上昇=飛行は feasible」と書いていました。ところがこれは**床のあるシーンで測った、脚で床を押した分**だったのです。同じシーンでは、**翼を止めても胴は 0.5 秒で 0.005 cm しか落ちません**(脚 6 点で床に立っている)。進化後に一時「ホバリング成功率 100%」と出たのも、**離陸せず床に立ったまま翼を振っていた**だけの偽陽性でした ──「羽ばたきを止めたら落ちるか?」という一発の確認で発覚しました。

床を外したモデルで正直に測り直すと:羽ばたきなしでは 1 秒で **−128.9 cm 落下**。最良の開ループ翼打ちでも、落下を **−56 → −43 cm/0.5 秒(約 23% 抑制)**するのが精一杯で、揚力は体重の遥か下。**単純な「振り + ひねり」の正弦翼打ちでは飛べません**。本物の昆虫ホバリングには、ストローク反転時の精密な迎角制御(figure-8 軌道)が要り、flybody 論文でも全翼自由度の**学習**方策で飛ばしています。翼は**実揚力を生む**(落下を有意に抑える)ものの、ホバリングには遠い ── これが正直なところです。

### 体④ 四足ロボットの登坂 ── 専用の報酬を作っても登れない(そして rear-up ハックを潰した)

平地では四足ロボット犬(unitree go2)は直立歩行できます(別節)。では坂は? 前回は登れませんでした。今回は「**登った分だけ加点する**専用報酬(重心の高さ獲得)」を足して再挑戦しました ── が、ここでも**報酬ハッキング**が出ました。素朴に「高さが上がったら加点」とすると、点数は上がる(1.48)のに中身は、**前進せず(前後移動ほぼ 0)、体を反らして重心だけ持ち上げて、そのまま転倒**。高さだけを稼ぐ "のけぞり" です。

そこで「**前に進み、かつ倒れていない時だけ、高さの加点を認める**」ゲートを入れて直しました。すると gaming は消えましたが ── 直した報酬の下では、6° の坂で前進わずか 0.22 m、しかも**重心はむしろ下がる**(登高ゼロ)。**専用の報酬を作っても、この方策の容量と CPU 予算では坂を登れない**、というのが非 gaming で確定した結論です(前回の負けを、今度は "抜け穴のない物差し" で追認した形)。平地は容易なのに、世界を少し傾けた瞬間に同じ手法が崩れる ── 実ロボティクスと同じ「平地と登坂の断崖」です。

### この 4 つで、何を持ち帰ったか

いちばん大事なのは、**4 つのうち 3 つで "偽の成功" が出て、そのすべてを公開前に自分で見破った**ことです。

- **place の "放置ハック"**:点数は上がるのに掴んでいない → 目標への最接近距離が凍結していたことで発覚。
- **虫の "床ホバリング"**:成功率 100% が、そもそも離陸していないだけ → 羽ばたき停止テストで発覚。
- **登坂の "のけぞりハック"**:高さは上がるのに前進せず転倒 → 前進 × 非転倒ゲートで矯正。

異常に良い数字(fit=1.000、成功率 100%)を見たら、まず内訳を疑う。**床を外す/羽ばたきを止める/held-out で測る/最接近距離の時系列を見る** ── そういう **"一発の意地悪な確認"** が、誇張した記事を 3 回、公開直前で止めてくれました。これは §5 の「世界モデルの搾取」と同じ話が、体を変えても何度でも顔を出す、という実例です。成功は 1.5 個(蛇=成功・全身骨格の運搬=半分)、正直な負けは 2 個(虫・登坂)。この比率のまま出すのが、この土台の正しい使い方だと思っています。

## 8. 作って分かったこと(教訓)

1. **プラットフォームは「差し替えられる軸」で設計する**。世界モデルの流儀・環境・ランタイムを部品化しておくと、世界モデルも人工生命も微小重力の到達実験も、**同じ土台の使い回し**で試せる。
2. **「想像の中のリターンが上がる」と「実環境で良くなる」は別物**。世界モデルの搾取は実装が正しくても起きる。だから評価器で実環境リターンとドリフトを**常に**測る。
3. **易しい環境の好成績を一般化して語ってはいけない**。合成環境で勝っても Pendulum で勝つわけではない。**内訳を疑う**。
4. **今回の設定では `greedy < random < QD` になったが、これは観測事実であって一般法則ではない**。少なくとも今回の予算では、**多様性(QD)を残す探索が有利**でした。いっぽうで 5 seed の範囲なので、「常に QD が勝つ」とは言いません。
5. **`gripped` と `de-tumbled` は別物**。掴めたように見えても、速いタンブラーは中でガタつくだけのことがある。到達・捕捉・減速・保持は、分けて測る必要があります。
6. **冗長性の効き方は単調ではない**。関節が多いほど壊れにくい、ではなかった。形態の良し悪しは、どの節が壊れ、どの幾何が残るかで決まります。
7. **形態と制御は両方効く**。腕の形だけでなく、差し込み方や速度ゲート、ロケット着陸での PD/PID のような制御方針でも成功率は大きく動いた。逆に、この CPU 予算では学習型コントローラが hand-set を安定して上回る証拠はまだ出ていません。
8. **仮説が外れたら、それ自体が結果**。今回の結論は「冗長性が効かない」ではなく、**今回の課題設定では冗長性の差が露出しにくかった**ということです。負けをそのまま載せると、次の実験が設計できます。
9. **異常に良い数字は、"一発の意地悪な確認" で潰す**(§7.7)。fit=1.000 やホバリング成功率 100% のような "出来すぎ" は、たいてい報酬の抜け穴です。**床を外す/羽ばたきを止める/held-out で測る/最接近距離の時系列を見る** ── 実装を疑う前に、まず 1 つだけ意地悪な確認を挿す。今回はこれで、誇張した記事を 3 回、公開直前で止めました。良い評価器を作ることと、良い方策を作ることは、同じくらい大事です。

## 9. 次回に続く ── プラットフォームの本命は「世界モデル × 進化」

次は、この 2 つの顔を**つなぎ**ます。§5 で見た「世界モデルの搾取」を手なずけつつ、§6-7 の進化を**世界モデルの“想像”の中で加速する** ── 候補の体や動き方を大量に「夢」の中でスクリーニングし、良さそうな少数だけ本物の物理で検証する、という筋です。高価な物理計算を、安い「夢」で間引く。これこそ、世界モデルをプラットフォームにした本当の狙いです。そのとき何が壊れて、何を測るべきか。……たぶん、また評価器が私を試してくるはずです。

### この土台の上に「次に載せられる」候補

onocollo は、ここまで出てきた**ロケット着陸・宇宙デブリ把持・水中遊泳**を、すべて**同じ型**で作ってきました ── **物理モデル + 正直な採点環境 + 手作りベースライン + そこから“温かくスタート”させる進化 + held-out(初見条件)での評価**。この型に、まだ試していない題材をそのまま載せられます。

正直に先に言っておくと、**以下の多くはまだ構想**です(ただし §7.7 で、下の 1「箸」と 3「多節の体(蛇)」は、素手把持の運搬と平地の這行まで実装まで進めました)。しかも能力そのものは、すでに立派な先行研究が山ほどある領域ばかり。だから onocollo の値打ちは「世界初」ではなく、**同じ正直な評価器で、都合の良い数字に逃げずに最後まで測りきる**ことの側にあります。手前にあるものから順に並べます。

1. **「掴む」の次は「道具で掴む」── 箸** ── 素手のグリッパを、**箸2本の複合接触**に拡張します。採点は「掴めた/掴めない」の二値ではなく、`整列 → 両側で接触が成立 → 滑りを抑える → 持ち上げる → 運ぶ` を**別々の軸**で測る。§7.5 で作った「本物の両面接触判定」がほぼそのまま流用できるので、候補のなかで**いちばん手前**にあります。(実機で箸を操るロボットや、箸を使う物理キャラクターの研究は既にあります。私が新しくできるのは能力ではなく、**進化 × 評価分割 × ノート CPU だけで正直に**、という測り方の側です。)
2. **海底で「泳いで、近づいて、掴む」** ── §6.4 で多細胞の体が水を泳いだ話と、§7.5 で腕がゴミを掴んだ話の、ちょうど**合流点**です。宇宙(§7)ではアームを固定していましたが、水中で自由に漂う体が物を掴むと、**その反作用で自分の体が蹴られます**。この「掴んだら自分が動く」難しさこそ面白いところ。まずは把持を足す前に、「**泳いでブイの上でピタッと止まれるか**」だけを切り出して測るつもりです(たぶん、これが最初の壁になります)。
3. **瓦礫を越える、多節の体** ── §6 の「泳ぐ多節の体」を地上に持ち出し、段差やバーを越えられるか。ただし、くねって進む歩き方そのものは古典的な研究が本当に山ほどあるので、私が測る値打ちがあるのは「**関節が1つ動かなくなっても越えられるか**」という冗長性の側(§7.5 の“壊れにくさ”調べの兄弟)だと思っています。
4. **双腕で、布をたたむ** ── これは正直、いちばん遠い。布のような**柔らかい物**は、硬い物を前提にした今の土台では計算が重く、素直には載りません。やるとしても、まず「**半日で、布がグシャッと破綻せずに動くか**」を最初の関門(go/no-go)にします。

どれを最初に選んでも、きっとまた ── **良い数字より先に、コケ方の方が先に見える**はずです。そして、それがこの土台の正しい使い方だと思っています。
