# 自宅のCPUだけで「世界モデル」と「進化する人工生命」を作り、宇宙ゴミ回収ロボまで進化させて、正直にコケた話

> GPUなしのノートPC(CPUのみ)で、**世界モデル(world model)** を学習し、それを **ROS 風のランタイム** の中で動かし、さらに **単細胞から進化する人工生命** を育て、最後に **宇宙ゴミ回収ロボの腕** を進化させるところまでを、**すべて本物の実行結果つき**でたどります。うまくいった所も、**うまくいかなかった所も隠さず**出します。むしろ後者がこの記事の主役です。

## この記事の 3 行

- **世界モデル**=AI が頭の中に持つ「次に何が起きるか」のシミュレータ。これがあると、実機を動かさず**想像(imagination)の中**で練習できる。それを **ROS っぽい配線**の上に載せ、「観測 → 考える → 動く」を 1 つの Python プロセスで回した。
- その上に **人工生命の進化**を載せた。1 節の“単細胞”から体を進化させ、**流体の中では多細胞の体が単細胞の 6.5 倍進む**という、多細胞化が報われる瞬間まで見えた。
- そして **宇宙ゴミ回収ロボの腕**を進化させ、「壊れにくさ(冗長性)は効くはず」という自分の仮説を、**実測でひっくり返した**。全部ノート PC の CPU で、数秒〜数分です。

はじめての方は上から、詳しい方は「5. 測って分かった正直な現実」と「7. 宇宙ゴミ回収ロボ」だけでも要点が取れます。用語は最初にまとめました。

## 1. 用語集(先に言葉を揃える)

| ノード / トピック / サービス | Node / Topic / Service | ROS の部品(役割)・その間を流れる名前付きの通り道(pub/sub)・要求→応答の 1 往復。 |

| 報酬ハッキング / 世界モデルの搾取 | Reward Hacking / World-Model Exploitation | AI が「中身を良くする」でなく「採点の穴を突く」方向に進化してしまうこと。**この記事の裏の主役**。 |

## 2. onocollo とは ─ 3 つの顔

**onocollo**(おのころ)は、私が自宅 CPU で作っている小さな OSS です。名前は古事記の **淤能碁呂島(おのころじま)**(かき混ぜた原初の海が凝り固まって最初の島=世界が生まれる)と、コロイド化(colloidal=凝集)を掛けた造語で、「潜在空間がかき混ざって世界(モデル)が生まれる」イメージです。

onocollo には 3 つの顔があります。

1. **世界モデルのコア** — 2 つの系譜を小さく再現:
   - **World Models**(Ha & Schmidhuber, 2018):VAE(視覚)+ MDN-RNN(記憶)+ 進化戦略で学ぶ制御器、という **V / M / C** の 3 分割。
   - **Dreamer**(Hafner ら):**RSSM** という 1 本の潜在動力学モデル + 想像の中で学ぶ actor-critic。
2. **`onocollo.ros` ランタイム** — ROS 2 風の **デーモン不要・1 プロセス**のグラフ(ノード/トピック/pub-sub/タイマ/tf/bag/サービス&アクション/teleop/launch/ハードウェア抽象)。**世界モデルそのものを一級のノード**として動かせます。
3. **`onocollo.mujoco` + `onocollo.evolve`** — 実在の物理エンジン **MuJoCo** を型付きで包んだ薄い層と、その上で**単細胞から体を進化させる人工生命**の遊び場。

設計思想は一貫して **「能力ではなく使いやすさで勝つ・そして正直であること」**。frontier(Genie や Sora 級)の汎用世界モデルは家庭では作れません。だから土俵を「特定環境の実用世界モデルを、CPU で誰でも数行で回せる」に移しています。

## 3. 5 行で世界モデルを学習して「夢を見る」

まず世界モデルのコアだけ。sklearn 風の API(facade)で、任意の gymnasium 環境に対して 5 行です。

```python
import onocollo as oc

env   = oc.GymEnv("Pendulum-v1")        # 任意のgymnasium環境(連続Box/離散Discreteどちらも可)
agent = oc.Dreamer(env, config="tiny")  # RSSM + 想像で学ぶactor-critic、CPU
agent.fit(steps=2000, progress=print)   # 収集→世界モデル学習→行動学習
print("greedy return:", agent.eval(episodes=5))
dream = agent.imagine(horizon=15)       # 世界モデルの中だけで未来をロールアウト(実機ゼロ)
```

## 4. `onocollo.ros` ─ 「ROS より簡単」を、正直な境界つきで

世界モデルは「考える」部品。それを**ロボットの脳の配線**の中で動かします。世界モデルを **`/imagine` サービス**として提供すると、他のノードはそれを他のどんな ROS サービスとも同じように呼べます ── これが「世界モデルを脳に組み込む」ということの具体像です。

```python
from onocollo.ros import Graph, EnvNode, PolicyNode, WorldModelNode, ServiceClient

g = Graph()
g.add_node(EnvNode(env, name="plant"))                # /obs を配信・/action を購読(=ロボット本体)
g.add_node(PolicyNode(agent, name="controller"))      # /obs を購読・/action を配信(=制御)
g.add_node(WorldModelNode(agent, name="world_model")) # /imagine サービスを提供(=想像)
g.spin(count=200)                                     # 観測→考える→動く を決定論的に回す

dream = ServiceClient(g, "/imagine").call({"horizon": 15})  # 世界モデルを "サービス" として問い合わせ
```

「ROS 系でできることは全部できる」を目指しましたが、**嘘はつきません**。開発・シミュレーション・学習で ROS を使う場面は onocollo で完結します。しかし実機のドライバ・ハードリアルタイム・DDS 配線互換は **持ち込み(bring-your-own)** で、そこは再実装したと名乗りません。

| ROS 2 の機能 | onocollo | パリティ |
| --- | --- | :---: |
| ノード / トピック / pub-sub | `Graph` + `Node` / `Publisher` / `Subscription` | ✅ プロセス内 |
| 実行器 / spin / タイマ / レート | `Graph.spin`, `create_timer`, `Rate` | ✅ ステップ+ソフトRT |
| tf2 変換 | `TransformBuffer`(時間バッファ+slerp補間) | ✅ 数学+管理 |
| rosbag2 記録/再生 | `Bag` + `RecorderNode`/`PlayerNode` | ✅(独自JSON形式) |
| サービス / アクション | `ServiceServer/Client`, `ActionServer/Client` | ✅ 同期 |
| ros2_control ハードウェア | `HardwareInterface` ABC + `SimHardware` | ⚠️ **IF+sim。ドライバは自作** |
| DDS 探索 / マルチホスト | — | ❌ 単一プロセス |
| ハードリアルタイム制御 | ソフトRTのみ | ❌ RTスケジューラではない |

**ハードウェア線より上**(認識・世界モデル・方策・tf・bag・launch)は、相手が `SimHardware(env)` でも実機の `HardwareInterface` でも **同一**です。脳全体を CPU で開発・シミュレーションし、最後にドライバだけ差し込む ── これが狙いです。ROS 2 そのものの代替(実機ドライバ・DDS・ハード RT)ではありません。表がそう言っています。

## 5. 測って分かった正直な現実(この記事の主役 その 1)

**異常に良い結果が出たら、勝った気になる前に内訳を疑う** ── これは私の開発の家訓です。逆に、悪い結果も消しません。教訓として残します。

### 5.1 「夢の中では」ちゃんと学習している

合成環境で世界モデルを学習させた実測(120 反復・CPU):

```
world-model loss   : 71.87  ->  19.57     # 下がる = 動力学を学んでいる
imagined return    : -0.398 ->  10.287    # actorの目的関数。想像の中の報酬は上がる
greedy real return : 7.359   vs random 2.697   # 易しい合成環境では random に勝つ
```

合成環境(=学習しやすいように設計した玩具)では、**学習済み方策がランダムに勝ちました**(7.36 vs 2.70)。**配線は正しく、学習信号は本物**です。

### 5.2 でも実タスクでは、ランダムに負ける

同じ facade を、**本物の制御タスク**に向けるとどうなるか。振り子(Pendulum-v1, 連続)と倒立振子(CartPole-v1, 離散)での実測(80 反復・CPU):

| 環境 | 行動空間 | ランダム | 学習後(グリーディ) | 差 |
| --- | --- | ---: | ---: | ---: |
| Pendulum-v1 | 連続(tanh-Gauss) | **-1158.5** | -1406.1 | **-247.5** |
| CartPole-v1 | 離散(Categorical) | **14.0** | 9.6 | **-4.4** |

**学習後の方が悪い。** これは失敗ではなく、世界モデルの本質が現れた瞬間です。名前がついています ── **世界モデルの搾取(world-model exploitation)**。actor は「学習途中の・まだ不正確な報酬予測ヘッド」を最大化するので、**想像の中のリターンは上がるのに実環境のリターンは下がる**、というズレが起きます。合成環境ではこのズレが出ない(易しくすぐ正確になる)。実タスクでは出る。

### 5.3 ドリフト曲線が「世界は動かない」にすら勝てない現実を暴く

onocollo には、この不都合を**隠すためではなく、測って見せるため**の評価器があります。世界モデルに、実環境と同じ行動列を「観測を見ずに」ロールアウトさせ、1 ステップごとの予測誤差を実測します。

![開ループのドリフト曲線。モデルの予測誤差はpersistence(世界は動かない)ベースラインより数百〜千倍大きく、曲線は右肩上がりでなく平坦](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/drift_curve.png)

読み方:**persistence(「世界は動かない」と予測するだけのベースライン)より、モデルの誤差の方が数百〜千倍大きい**(`model/persist ≫ 1`)。つまりこの小さいモデルは、再構成誤差の床に張り付いていて、**「世界は動かない」という一番バカな予測にすら負けている**。教科書的な「開ループ誤差の累積」は、テストでは劣化させた予測器で示せますが、**この規模の本物のモデルでは起きません**。評価器は、起きた事実だけを報告します。

### 5.4 「もっと計算すれば良くなる」を測ったら、逆に悪化した

「反復を増やせば実制御も良くなるだろう」と思って、Pendulum で **反復 80(tiny)vs 反復 300(default, モデルも大きい)** を実測しました。予想は外れました。

| 設定 | 実 eval(高いほど良い) | world-model ロス | 想像リターン |
| --- | ---: | ---: | ---: |
| ランダム | **-1158.5** | ─ | ─ |
| tiny・80 反復 | -1406.1 | 66.9 → 17.4 | 0.42 → **-24.8** |
| default・300 反復 | **-1522.2** | 75.8 → **4.0** | 0.02 → **-154.5** |

**計算を増やしたら、実制御は悪くなりました**(-1406 → -1522)。しかも内訳が示唆的です:計算を増やした方が **world-model ロスは最もよく下がっている**(4.0)。つまり **動力学の予測は一番よく学べている**。にもかかわらず、実環境の成績は最悪。

これは「世界モデルの搾取」が **計算を増やすほど強まる**ことを実測で捉えた瞬間です。動力学が良くなるほど、actor は「まだ不完全な報酬ヘッド」の穴を**より巧妙に突ける**ようになる。**「もっと回せば良くなる」は、少なくともこの規模の Pendulum では偽**でした。

> これこそ、正直な評価器を最初から作っておく価値です。もし実環境リターンを測っていなければ、「ロスが 4.0 まで下がった!学習成功!」と誤って喜んでいたでしょう。

## 6. 単細胞から進化させる ─ 人工生命(この記事の主役 その 2)

ここから 3 つ目の顔です。onocollo は実在の **MuJoCo** エンジンを型付きで包む薄い層(`onocollo.mujoco`)を持っていて、その上に **Karl Sims の「進化する仮想生物」** の系譜を継ぐミニ版(`onocollo.evolve`)を載せました。

やることはこうです。**体のゲノム**(自由に動く根 + 波打つ関節の“節”の鎖)を、手続き的に**本物の MuJoCo モデルへ変換**し、動かして、その成績で選ぶ。突然変異で節を増やしたり減らしたりする ── だから **体そのものと歩き方が同時に進化**します。1 節の“単細胞”から始めます。

### 6.1 貪欲な進化は「単細胞」に縮む

まず素朴な `(μ+λ)` 進化(良いものを残して変異、を繰り返す)。平地を這わせると、**0.009 m → 0.62 m** と 20 世代でぐんぐん伸びます。……が、行き着く先は **1 節への収束**でした。平地の短距離走では、余計な節はむしろ邪魔。いちばん単純な体がいちばん速いのです。

![(μ+λ)進化で這う生物が世代を追って速くなるが、体は1節に収束していく様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/evolving_creature.gif)

### 6.2 QD(品質多様性)で「体サイズ別の名人」を残す

これは前回の記事(進化ループの話)で書いた **MAP-Elites / QD** の出番です。「総合 1 位」だけを追うと多様性が死ぬので、**(節数, 姿勢)ごとにチャンピオンを別々に残す棚**を使います。すると 1 節に潰されず、**全 6 種類の体サイズのチャンピオンが同時に生き残ります**(1 節 +0.108 m … 6 節 +0.005 m)。ピーク性能と引き換えに多様性を守る ── これが QD のトレードオフです。

![体の節の数ごとにチャンピオンを並べたMAP-Elitesの棚。1節に潰されず、節数というタイプ別の名人が共存している](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/qd_creature_zoo.gif)

### 6.3 感覚フィードバックで、多節が「使える」体になる

次に、各関節が**自分のセンサと歩調の時計を読んで動く小さな神経回路(閉ループ制御)**を載せ、体と一緒に進化させます(これが Karl Sims 本来の形)。すると多節の体が**競争力を持ち、走行の多くの世代で先行**するようになりました(2〜3 節がリード・最良 +0.27 m)。開ループの CPG が 1 節に縮んだのと好対照です。

![各関節が感覚フィードバックで動く神経制御を載せた多節生物が這う様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/neural_creature.gif)

### 6.4 水の中で、多細胞化が「決定的に」報われる

いちばん綺麗に出たのがこれです。**重力を切って粘性のある流体**にすると、多節の体は**進行波(travelling wave)**でぐいぐい進み、1 節はほとんど動けません。QD で測ると **6 節 +0.92 m 対 1 節 +0.14 m ── 6.5 倍**。ここで初めて、**多細胞であることが決定的に得**になりました。面白いのは、貪欲な `(μ+λ)` は流体でも 1 節に収束してしまうこと。**多様性を守る QD だからこそ、この多節の最適解を見つけられた**のです。

![粘性流体の中で多節の生物が進行波で泳ぎ、1節はほとんど動けない様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/swimmer.gif)

### 6.5 「届く/届かない」は体の長さで決まる ── 幾何の壁

もう 1 つ、体長が物理的に効くタスク。根を固定して、先端を遠くの目標に届かせます。目標を **0.85 m** に置くと、**1 節は 0.55 m も足りず**、**3 節以上でようやく届く**。ここでは巧妙な制御より前に、**体の長さという幾何(geometry)の壁**が立ちはだかります。「賢さ」ではどうにもならない、身も蓋もない物理です。

![根を固定した腕が遠くの目標に手を伸ばす。1節は届かず、多節が目標マーカーに接触する比較](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/reach.gif)

> **正直な但し書き:** ここでの「単細胞」は**体の 1 節**のことで、細胞生物学(膜・代謝・分裂)ではありません。物理も toy スケールの剛体です。それでも「単一目的の進化は最小の体に縮み、多様性(QD)と感覚と課題設定が多細胞化を報いる」という **筋の通った物語**が、ノート PC の CPU で数分で観察できたのは、私には十分に面白い結果でした。

## 7. 宇宙ゴミ回収ロボを進化させる ── そして正直にコケた

最後に、少し未来の話。**宇宙ゴミ(デブリ)の回収**は、これからの人類の宿題です。そこで「**微小重力の中で、通り過ぎるタンブリング(回転)デブリに手を伸ばして掴むアーム**」を、同じ進化の仕組みで育ててみました。

![微小重力の中で、固定ベースのアームが通過するタンブリングデブリに手を伸ばして捕捉する様子](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/onocollo/space_debris_capture.gif)

結果、ここでも **幾何の壁**がくっきり出ました。**1〜2 節では届かず**(0.55 m / 0.25 m 手前)、**3 節以上でデブリに到達して捕捉**(中心まで約 0.08 m ≤ デブリ半径 0.12 m)。捕捉できる最小の体は **3 節**でした。

### そして、自分の仮説が外れた

私はここで **「壊れにくさ(冗長性)は効くはず」** という仮説を立てていました。宇宙のロボは修理に行けないので、**関節が 1 つ壊れても掴めること**が命のはず。そこで、いろいろな飛来のしかた × **関節を 1 つわざと殺す(dead servo=その関節が力を失う)**、という条件で頑健性を測りました。

ところが ── **冗長性の利点は、見えませんでした**。この難度では**「3 節以上」という幾何要件が支配的**で、それさえ満たせば、**関節が 1 つ壊れても 3〜6 節すべてが 100% 捕捉**してしまう。捕捉の余裕が大きすぎて、壊れにくさの差が埋もれてしまったのです。

これは **仮説倒れ**です。でも消しません。教訓として残します ── **「冗長性が効く」を見たいなら、まず課題を"ギリギリ"にしないといけない**(デブリ半径を縮める/高速で飛ばす/複数同時に来る)。余裕のある問題では、冗長性は測定にすら現れない。これは、実機の航空宇宙で語られる「軽さ 対 壊れにくさ」の矛盾を、toy の中で追体験した瞬間でした。

> **honest scope(核心):** これは**軌道力学ではありません**。本物のデブリ除去は、相対軌道運動・推進剤・遠距離センシングが本体で、そこは MuJoCo の外です。ここでやったのは **終端の近接捕捉だけ**、しかも「capture」は**近接判定**であって、まだ**掴む物理(把持)そのものはモデル化していません**。派手な絵に見えますが、実体は等身大です。派手な見出しと実体の距離を、自分で測って正直に書く ── それがこの記事全体でやりたかったことです。

## 8. 作って分かったこと(教訓)

1. **「想像の中のリターンが上がる」と「実環境で良くなる」は別物**。世界モデルの搾取は、実装が正しくても起きる。だから評価器で実環境リターンとドリフトを**常に**測る。
2. **易しい環境の好成績を一般化して語ってはいけない**。合成環境で勝ったからといって、Pendulum で勝つわけではない。**内訳を疑う**。
3. **単一目的の最適化は、いちばん単純な解に縮む**。多細胞化(多節)を見たければ、**多様性を守る仕組み(QD)・感覚・課題設定**が要る。「複雑さ」はタダでは生まれない。
4. **物理には、賢さで越えられない幾何の壁がある**。届く/届かないは体の長さで決まる。制御の工夫はその後の話。
5. **仮説が外れたら、それ自体が結果**。冗長性が見えなかったのは失敗ではなく、「課題をギリギリにしないと冗長性は測れない」という発見。**負けをそのまま載せると、次の実験が設計できる**。

## 9. 次回に続く

次は、**世界モデルと進化をつなぐ**話です。§5 で見た「世界モデルの搾取」を手なずけつつ、§6-7 の進化を**想像の中で加速する** ── 候補の体や動き方を大量に「夢」の中でスクリーニングし、良さそうな少数だけ本物の物理で検証する、という筋です。高価な物理計算を、安い「夢」で間引く。そのとき何が壊れて、何を測るべきか。……たぶん、また評価器が私を試してくるはずです。
