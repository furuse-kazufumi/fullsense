# 骨と筋だけの人体模型に mocap で踊らせ・箸を持たせた話 — マスコット「evis」誕生と、猫背バグの真犯人を測って捕まえた記録

> 対象: MuJoCo・モーションキャプチャ・リターゲット・キャラクター表現に興味がある人 / 「解剖学的な人体モデルを自宅で動かしてみたい」人 / 成功談より**バグを測って追い詰める過程**を読みたい人。
> 前提知識: Python が読めれば十分。IK・mocap の用語は都度かみくだきます。
> すべてノート PC の CPU で実測。この記事の主役は、きれいな最終形ではなく、**「歩かせたら老人みたいに猫背になった」その原因を推測でなく数値で特定した過程**です。

---

## 0. 3行まとめ(先に結論)

- **骨と筋だけの人体モデル**(MuJoCo Menagerie の `MS-Human-700`、皮膚も内臓もなく骨・腱・700本の筋肉だけ)を、**マスコット「evis」**に仕立て、**モーションキャプチャ(mocap)をリターゲット**して歩かせ・踊らせ・**箸で食べさせた**。
- 途中、歩かせたら**「腰が曲がって老人みたいな猫背」**になった。推測で直そうとすると外す。**関節角を1個ずつ測った**ら、真犯人は「頭が前に出ている」でも「筋肉が足りない」でもなく——**マーカーの高さの取り違え**で、脊椎の関節が可動域の限界(±29°)に張り付いていた。測って初めて分かった。
- 目は最初 2D の描き込み(投影+深度で顔に貼る)だったが、横を向くと目が宙に浮くバグが出た。**「白目を 3D の球としてモデルに一体化し、表情をテクスチャで描けば?」**という一言で全部解決した。頭と一緒に回り、骨で自然に隠れる。**2D の小細工 80 行が消えた。**

honest な但し書き: これは**キネマティック再生**(各フレームの関節角を直接与える)であって、筋肉で駆動しているわけではない。箸も「持っている風」の剛体プロップで、指で握ってはいない。**どこが本物でどこが演出か**は最後に明記する。

![evis motion reel](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/evis_mocap/reel.gif?v=1)

*↑ 歩く→走る→跳ぶ→サルサ→キック→箸で食べる。全部同じ evis を、無料の mocap を移し替えて動かしている。*

## 1. 用語集(先に言葉だけ)

| 用語 | かみくだき |
|------|-----------|
| **モーションキャプチャ(mocap)** | 人の動きを記録した関節角の時系列データ。ここでは無料の BVH ファイルを使う。 |
| **リターゲット(retargeting)** | ある骨格の動きを、**別の骨格**(体型も関節構造も違う)に移し替えること。ここが本題。 |
| **BVH** | mocap の代表的なファイル形式。骨の階層と各フレームの回転角がテキストで入っている。自己完結(外部モデル不要)。 |
| **IK(逆運動学 / Inverse Kinematics)** | 「手先をここに置きたい」から**関節角を逆算**する計算。 |
| **キネマティック再生** | 物理(力・筋肉)を回さず、関節角を直接セットして絵にすること。動きの見た目だけを作る。 |
| **equality constraint(等式拘束)** | 「この関節はあの関節の関数で決まる」という連動ルール。膝や肩に大量に入っている(後述の罠)。 |

## 2. 出発点 — evis という「中身が見える人体」

evis の素体は **MS-Human-700**(Zuo et al., ICRA2024、Apache-2.0)。全身 85 自由度・**700本の筋肉**を持つ、生体力学研究用の人体モデルだ。皮膚がなく、骨(白)と筋・腱(赤)がむき出しになっている。

この「肉を剥いだ姿」が、そのままキャラクターの魅力になった。名前は **evis** に決めた。由来は二層で、

- **恵比寿(えびす)** — 七福神のひとり。縁起がいい。
- **eviscerate** — 内臓・肉を取り除くこと。まさに骨と筋だけになった evis の姿そのもの。

house style は「白い骨 + 暗いスタジオ + 発火するように光る赤い筋 + コミカルな白目」。筋の光り方には仕掛けがあって、キネマティック再生では筋活性がないので、**各部位の"速さ"で赤く発火させている**——速く振った腕がパッと光る。動きが要求する運動パターンが可視化される、というわけだ。

## 3. なぜ mocap か、そして最初の壁は「技術」ではなく「ライセンス」だった

手作りのサイン波で腕を振らせても、動きが硬い。**本物の mocap を移植したい。**

ところが最初にぶつかったのは技術ではなく**ライセンス**だった。evis を含む FullSense は商用 dual-license なので、mocap データも商用利用できる必要がある。有名どころを調べると:

| ソース | 形式 | 商用 | SMPL 依存 | 判定 |
|--------|------|------|-----------|------|
| **CMU MoCap**(cgspeed BVH 変換) | BVH | **可** | なし | **採用** |
| AMASS / AIST++ | SMPL パラメータ | 不可 | **要 SMPL(非商用)** | 除外 |
| Mixamo | FBX | 製品内は可 | なし | 要アカウント・データ再配布不可 |
| Bandai-Namco / LaFAN1 | BVH | **不可(NC)** | なし | 除外 |

`SMPL` は人体メッシュの標準モデルだが**研究用(非商用)**ライセンスで、AMASS も AIST++ もこれに依存する。よく使われるが、商用製品には持ち込めない。

答えは **CMU Graphics Lab Motion Capture Database** だった。ライセンス文は "This data is free for use in **research and commercial** projects worldwide"。BVH で自己完結(SMPL 不要)、git で自動取得できる。ただし一点 caveat があって、**「データそのものを(変換形でも)再販してはいけない」**。だから「リターゲット済みのアニメ」を配るのは OK だが、**BVH ファイルそのものはリポジトリに同梱しない**(必要なとき取得スクリプトで落とす)ようにした。

> 教訓: 「技術的に最良」と「配れる」は別。研究デモが商用製品に化けるなら、**最初にライセンスを敵対的に裏取り**したほうがいい。

## 4. リターゲットの仕組み — 「回転」ではなく「位置」を移す

BVH の骨格(標準的な人型)と MS-Human-700 は、関節の構造がまるで違う。MS-Human-700 は OpenSim 由来で、たとえば骨盤ルートが quaternion の自由関節ではなく**6個の独立した1自由度関節**、肩は「幽霊ボディ」を挟んだ複雑な肩帯になっている。回転角をそのまま移すのは地獄だ。

そこで**位置ベース**にした。BVH を forward kinematics で解いて、**各関節の"世界座標(点群)"だけ**を取り出す。回転は一切移さない。あとは、その点群に evis の対応する体の部位が重なるように、**差分 IK(Levenberg-Marquardt)**で関節角を解く。手・肘・膝・足・頭など約 16 点を狙い、前フレームから温めて解くので 1 フレーム約 30ms、5 秒のクリップが数秒でリターゲットできる。回転を移さないので、あの厄介な肩帯も IK が勝手に良い形に配分してくれる。

**ここに一つ、キネマティック特有の罠があった。** MS-Human-700 には膝・肩帯・手首に **42 個の等式拘束**(「この幽霊関節は肩の角度の多項式で決まる」)が入っている。ところが**関節角を直接セットする再生では、この拘束が MuJoCo に強制されない**。放っておくと IK が肩甲骨の幽霊ボディをあらぬ方向にねじり、肩が壊れる。なので毎フレーム、**多項式を自分で評価して従属関節を書き込んで**やる。これで膝の screw-home も肩のリズムも正しく動く。

## 5. 本題 — 「腰が老人みたいに曲がる」その真犯人を、測って捕まえる

歩かせてみた。動く。……が、レビューしてもらうと一言、**「腰の曲がり方が老人みたいだ」**。

側面から見ると確かに、頭が前に突き出て背中が丸い、典型的な"猫背"だった。ここからが本記事の主役、**推測でなく測定で真犯人を追い詰める過程**だ。

**仮説1: 頭が前に出ている?** 頭と骨盤の前後差を測ると **0.3cm**。ほぼ真上。**却下。**

**仮説2: mocap 元がそもそも猫背?** 元 BVH の首の前傾を測ると **9.5°**(軽い)、胴体は 1.6°。ほぼ直立。**元データは猫背ではない。リターゲットが増幅している。**

**仮説3: 姿勢正則化が弱い?** 脊椎・首を中立へ弱く引く項を足したが、**視覚的にほぼ変わらない**。マーカーの引く力に負けている。

推測をやめて、**関節角を1個ずつ実測した**。すると犯人が現れた:

```
T12_L1_FE      = -28.6°   (可動域 [-29, +29] → 負の限界に張り付き)
T1_head_neck_FE = +28.6°  (可動域 [-29, +29] → 正の限界に張り付き)
```

中背(胸腰椎)が片方の限界へ、首が反対の限界へ、**両方が飽和**していた。これが猫背の正体。ではなぜ IK は関節を限界まで曲げたのか。

**真犯人はマーカーの高さの取り違えだった。** evis の `thoracic1`(胸椎最上部)は骨盤の **0.50m 上**(肩の高さ)にある。ところが私はそこに、BVH の `Spine1`(骨盤の **0.26m 上**=中背)を割り当てていた。**肩の高さの体を、中背の高さへ引き下げようとして、3 セグメントしかない脊椎が圧縮され、関節が限界まで折れ曲がっていた**のだ。

修正はシンプルだった。**高さの合う対応に変える**——`thoracic12`(0.23m 上)←`Spine1`(0.26m)。過剰に拘束していた `thoracic1`/`lumbar5`/頭頂マーカーを外し、姿勢正則化を少し効かせる。結果、歩行は直立になり、しかも**ダンスの表現的な背屈(前屈・のけぞり)は残った**——あれはマーカーが本当に要求している動きだからだ。

> 教訓(この記事で一番言いたいこと): **「たぶん頭が前に出ている」で直そうとすると外す。** 採点の内訳(関節角)を測ると、犯人は思いもよらない「マーカーの高さ」だった。[[feedback_benchmark_honest_disclosure]] の精神——**推測する前に測れ。**

## 6. 目を 3D にしたら、小細工が全部消えた

evis の表情(＞＜ や ◎◎)は、最初は **2D の描き込み**だった。頭の 3D 位置をカメラに投影して画面上の目の座標を求め、深度バッファで「今この目は見えているか(横を向いて骨に隠れていないか)」を判定して、白目と顔文字を上から貼る……という仕組み。動くには動いたが、**横顔のとき遠い方の目が背景に投影されて宙に浮く**バグが残った。

そこへ一言、**「白目を 3D の球としてモデルに一体化して、眼球に直接テクスチャで表情を描けば?」**

これが正解だった。表情ごとに**キューブテクスチャ**(前面に顔文字、他は白い強膜)を焼いた眼球の球を頭に埋め込み、感情ごとに用意しておいて、**表示する 1 組だけ残して他を透明にする**ことで挿げ替える。すると:

- 目が**頭と一緒に回転**する。
- **骨(頭蓋)が自然に目を隠す**——後頭部を向けば目は消える。深度判定も投影計算も要らない。
- どの角度でも正しい。ダンスで頭が回っても、転倒で下を向いても勝手に正しくなる。

投影+深度の 2D 小細工が丸ごと(約 80 行)消えた。感情は 8 種(neutral の丸目 ●●、joy ＞＜、surprise ◎◎、love ♥♥ はカラー絵文字テクスチャ、dead ×× ……)+ 任意の文字。ちなみに全角の ＞＜ が絵文字と誤判定されて薄く描かれるバグもここで直した(判定範囲の取り違え)。

![evis expressions](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/evis_mocap/evis_expressions.png?v=1)

物理の転倒 rollout に配線すると、evis が倒れながら **neutral → ◎◎(落下)→ ××(崩落)**と表情を変える"実況"にもなる。

![evis reaction fall](https://raw.githubusercontent.com/furuse-kazufumi/fullsense/main/docs/articles/assets/evis_mocap/reaction.gif?v=1)

## 8. 正直な限界(どこが本物で、どこが演出か)

honest disclosure がこのシリーズの核なので、境界を明示する:

- **本物(研究層)**: 骨格の動きは実 mocap をリターゲットした関節角。IK も等式拘束の評価も本物。姿勢の直立化も測定に基づく修正。
- **演出(キャラクター層)**: 筋の赤い発火は「速さ」の proxy(筋活性ではない)。目・アホ毛はリグ。箸は**剛体のプロップ**で、**指で握ってはいない**(腕は実 mocap で口へ運ぶが、把持は cosmetic)。
- **キネマティック**: 全体が関節角の直接再生で、物理は回していない。**筋肉で駆動する解剖学的な手の把持**(進化計算で筋を学習させる)は、これとは別の、もっと難しいトラックだ。
- **省略**: 胸郭の**肋軟骨**は骨のみモデルなので省略されている(前面がスカスカに見える正体)。解剖図で正しい形(肋骨 1–7 は胸骨へ、8–10 は肋骨弓を作る)まで裏取りしたが、キャラクター向けの近似としては難しく、今回は既定でオフにした。

## 9. 次回へ

evis は「歩く・踊る・箸で食べる」までできた。だが**箸を"本当に指で握る"**のは、キネマティックの延長ではなく、**筋肉を進化で学習させる**別の山だ。次はそこに登る——81 本の筋肉の手が、剛体プロップではなく本物の箸を摘まみ、食べ物を口へ運べるか。

そしてもう一つ。今回 evis の表情を実況に配線したように、**評価(eval)そのものに顔をつける**——歩行の学習でコケたら ×× になり、うまく進めば ＞＜ で笑う。**シミュレーションが自分で実況する**マスコットへ。

> 測って、正直に負けを残して、次に登る。骨と筋だけの evis の物語は、まだ続く。

### 付録: 使ったもの / 再現

- モデル: MuJoCo Menagerie `MS-Human-700`(Apache-2.0)
- mocap: CMU Graphics Lab Motion Capture Database(商用可・SMPL 非依存)。取得は `scripts/fetch_cmu_dance.py --motion`
- 実装: `numpy` + `mujoco` のみ(scipy も重い依存も無し)。リターゲッタ `src/onocollo/evolve/mocap_retarget.py`、マスコット `scripts/musculo_mascot.py`
- 出力: mp4(h264、Qiita 埋め込み用)/ gif は拡張子で自動切替
