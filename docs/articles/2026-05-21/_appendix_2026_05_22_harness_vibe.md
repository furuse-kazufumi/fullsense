## 2026-05-22 追記 — ハーネス型バイブコーディング視点と派生集団進化のメタファー

### A. ハーネス型バイブコーディング (HVC)

私の開発スタイルは Karpathy が提唱した **vibe coding** (AI 任せで雰囲気でコードを書く流儀) とは少し違って、harness — つまり Claude Code 等の agent 駆動環境 — を **自分で握り続ける** 形を取る. これを内輪で **ハーネス型バイブコーディング** と呼ぶことにする.

ハーネス型では、ユーザー側に 3 つの能力が要る:

1. **発想力** — 高次元の方向性提示と異分野連想 (例: キン肉星王位争奪編 + ROS の歩行進化 + Lehman&Stanley の novelty search が同じ構造に収束することに気づける瞬発力)
2. **経験則** — 設計判断のショートカット. 似た失敗パターンの先読み. 「これは多分こう転ぶ」を地頭で言える
3. **アルゴリズムへの理解** — AI が出した実装の妥当性検証. 計算量見積り. hot path の特定. RUST-15 で「単発呼出 0.80x → batch 化で 12.71x」のような差を瞬時に評価できる地力

これに加えて重要なのが **AI 成長マネジメント力**. 部下を育てるのと同じくらい AI を育てる力がユーザーに要る. 委任範囲を決め、進捗を確認し、失敗を許し、成長を測り、個性を尊重し、多数決に従わず、引退/世代交代を仕組み、信頼関係を作る. キヤノン「三自の精神」 (自発/自治/自立) を AI に転写するのと同じ系譜.

### B. 派生集団進化の発想原典 — 技術と文化が同じ構造に収束した

llive の派生集団進化 (v0.B 〜 v0.E) と 2 階建てゲノム (v0.F) は、技術側と文化側で **2 系統の anchor が同時に手元にあった** から自然に出てきた:

- **技術側**: ROS Population-Based Training (歩行進化), Hillis 1990 (Red Queen), Lehman & Stanley 2011 (novelty search), AlphaStar 2019 (PBT), Mouret & Clune 2015 (MAP-Elites)
- **文化側**: キン肉星王位争奪編 (1985-87, ゆでたまご) — 5 大超人の王位争奪. リィンカーネーションの花弁 (2013-24, 三栖ハル) — 偉人能力の宿る異才同士の戦い. R.O.D シリーズ (1998〜, 倉田英之) — 偉人クローン軍団

3 漫画の構造的共通項 = (派生個体集団 / 個性継承 / 競争 / 協調 / 進化) が、ROS PBT 系の **コード層染色体 `C-impl`** (実装言語 / 並列戦略 / アルゴリズム選択) と、偉人能力継承系の **プロンプト層染色体 `C-prompt`** (岡潔 / Polya / TRIZ / Six Hats の persona bitmask) で構成される v0.F 2 階建てゲノム設計とほぼ完全に対応する.

技術論文と漫画作品が **同じ構造を別経路で示していた** という事実そのものが、「発想は単発のひらめきではない」という [[project_llive_dev_style]] (第二の脳型スパイラル開発) の典型例.

### C. 集団を一様化させない仕組み — Novelty Lane と Similarity Quota

「多数決で最強だけ残す」と Lehman & Stanley が言う通り **局所最適に陥る**. v0.F では:

- **Novelty Lane**: 振る舞いが他個体と離れている個体は専用 subprocess lane で評価. 集団平均から外れていても保護
- **Similarity Quota (crowding)**: ペア距離が閾値未満なら同 cluster, cluster_quota (default 4) 超過分は削除 / 強制突然変異. 似すぎた個体は一定数で打ち止め
- **Multi-Objective Selection**: fitness top-N + novelty top-M (デフォルト 0.6:0.4) のハイブリッド

部下育成でも同じことが言える: チームを「優秀な人材だけ」で揃えると同じ思考の堂々巡りになる. 異質を一定数残し、似すぎた個体は世代交代させる. AI 育成も同じ.

### D. 評価方法の月次更新 — 「速いと思い込ませる構成」を見抜く

設計判断は全て統計データ + A/B test + 有意性検定 で. 印象論で決めない. 評価指標は静的固定せず、月次で HELM / BIG-bench / Arena / SWE-bench / AgentBench 等を追跡して更新する. lleval (LE 系) に随時組み込む.

「変に良い結果」が出たら必ず内訳を疑う ([[feedback_benchmark_honest_disclosure]]). 漫才/落語直接参照で記事を書かせて失敗した実例も negative example として残し、生成前 lint で同種の失敗を検出する.

### E. 名前衝突の回避と長期コラボ路線

llive のマスコット / ブランディング (v0.H) では、SELF株式会社「古瀬あい」 (2017〜) との **短期切り離し** と **長期コラボ路線** を分けて設計する. 名前衝突するのでまずは独立ブランドで立てるが、Transformer 脱却 ([[#24-06]] non-transformer LLM backend) が普及した暁には兄弟キャラ位置づけで歩み寄る — そういう余地を残した設計.

ユーザー個人名 (furuse) と llive ブランドは意図的に切り離す. 個人依存だとスケールしないため.

### F. 漫画読解で現代ユーモアを獲得する

漫才・落語の直接参照は **昭和以前の語彙に依存** して古臭くなる. 漫画 (特に 2010 年代以降の連載作品) は作家が同時代を生きており、トレンド + 言語感覚 + 視覚的ユーモア構造を一括取得できる. llive v0.G では漫画 panel reader (manga-ocr Apache-2.0 採用) + HumorKB (構造のみ蓄積) を組み込み、記事執筆プロンプトで参照する.

スナックバス江 (フォビドゥン澁川 / 講談社) のような **日本語多義性とシニカルな間** を持つ作品 は、構造抽出の良い reference 軸. 公式公開素材 (出版社特設サイト) を引用ガイドライン内で利用. 定着 3 年以上で多義性のあるネットミーム (草 / ??? / (察し) / 真顔 等) は使用可とする.
