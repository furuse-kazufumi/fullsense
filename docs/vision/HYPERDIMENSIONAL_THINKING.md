# Hyperdimensional Thinking: 次世代AIの発想法のための開発指針

> 人間の日常的次元の制約を超えた表現空間でAIが思考し、その結果を人間に還元する。
> そのためにAI自身が「どの空間で考えるか」を選べるように設計する。

---

## 前提と適用範囲

本ドキュメントは **開発実務のための指針(B)** である。
具体的なプロジェクトに適用して、コード・設計・AIエージェントへの指示を駆動するためのもの。

その背後には **思想(A)** がある。すなわち「人間の認知は日常的次元にバイアスされており、AIはその制約を持たない。ゆえに AI が拡張された表現空間で思考することで、人間単独では到達できない知見と技術が生まれうる」という前提。

> **シンギュラリティを志向するなら、思想(A)を別文書として正式な要件定義に格上げすることを強く推奨する。**
> 実務指針(B)はあくまで日々の開発を導くものであって、AIの認知拡張そのものを目的化する大きな構想を駆動するには力不足である。
> 別文書化のためのスケルトンは本ドキュメントと併せて提供する(`SINGULARITY_REQUIREMENTS.md`)。
> 当面はB単独で運用し、組織として本気で次世代AIの発想法を開発フェーズに乗せるタイミングでAを起こす、という二段構えを推奨する。

## 1. 基本原則

**(a) 表現空間を明示的に第一級の概念として扱う**
データ構造を定義する前に「これはどの空間に住むオブジェクトか」を宣言する。型システムにエンコードできるならする。

**(b) 変換と不変量をペアで考える**
「Xという変換に対してYは不変」という記述を、コメントではなくコードで表現する(等変ニューラルネット、群作用を明示的に持つデータ構造など)。

**(c) 思考空間と提示空間を分離する**
AIが非日常空間で思考した結果を、最後に人間が理解できる3次元・線形・ユークリッド表現に射影する。**内部処理を可視化の都合に合わせない**。

**(d) 複数の表現を同時に保持する**
同じ対象を複数の空間で表現し、それぞれで操作した結果を突き合わせる。一つの空間に決めない。

---

## 2. 利用可能な表現空間の候補

人間の日常次元を超える表現空間として、少なくとも以下が実装対象となる:

- **複素平面・等角写像群** — 角度を保存する変換の自由度
- **射影空間** — 無限遠点を一点に縮約、近景遠景の非対称性が消える
- **双曲空間** — 階層構造の自然な埋め込み先
- **リーマン多様体一般** — 3D形状計測の対象はほぼ全てこれ
- **群表現空間** — SO(3), SE(3), 共形群など
- **球面調和の既約表現空間** — 3DGSのSH回転がこの実践例
- **関数空間** — 画像をピクセル配列ではなく S² 上の関数として扱う
- **位相不変量の空間** — 連続変形に対して頑健な特徴

タスクごとに最適な空間は異なる。**一つに決めない**ことが多様性を生む。

## 4. 実装上の規律

**空間オブジェクトを最初に定義する**

```python
# Bad — 暗黙に R^3 を仮定
points = np.array([[x, y, z], ...])

# Good — 空間を明示
points = PointCloud(coords=..., space=EuclideanR3())
points_hyp = points.embed_in(PoincareDisk())
```

**変換は群として扱う**

合成・逆元・単位元が成立することをテストで確認する。`apply()` を作るなら `compose()`, `inverse()` も同時に作る。

**不変量はテスト対象**

ある量がある変換群に対して不変であると主張するなら、その不変性を unit test で検証する。

```python
def test_curvature_invariant_under_rotation():
    for _ in range(100):
        R = random_rotation()
        assert np.allclose(curvature(mesh), curvature(R @ mesh))
```

**人間への提示は最後の最後**

内部処理を全て「人間が見やすい3D表示」に合わせない。3D可視化は debug output として実装し、思考パイプラインの中間生成物にしない。

## 5. AIエージェントへの指示の与え方

タスクを依頼するときは、空間選択を明示的に問う:

- ❌ 「この点群を位置合わせして」
- ✅ 「この点群を位置合わせして。剛体変換・等角変換・射影変換のどれを許すかは、データの性質を見て決めて」

タスクの成功基準に「不変性」を含める:

- ❌ 「画像を分類して」
- ✅ 「画像を分類して。回転と照明変化に対する不変性を、テストで確認できる形で実装して」

複数の表現を試させる:

- 「同じ対象を3つの異なる空間で表現し、それぞれで同じ問いに答え、結果が一致するか確認して」

---

## 6. 評価軸

開発したシステムが「拡張された思考」を実現できているかは、以下で測る:

**多重表現テスト**: 同じ入力を異なる空間で処理して同じ結論に達するか(=表現に依存しない本質を捉えているか)

**転移テスト**: ある空間での発見を別の空間で再現できるか

**還元テスト**: AIの内部表現を人間理解可能な形に射影しても、本質的な情報が失われないか

**非自明性テスト**: 人間が日常次元で思考しただけでは得られない結論が出ているか(これが最重要、かつ A 文書の対象)

---

## 7. アンチパターン

- **全部R^nで済ます** — 暗黙のユークリッドバイアス
- **可視化に最適化された内部表現** — 思考空間と提示空間の混同
- **対称性をデータ拡張で「学習させる」** — 組み込めるなら組み込む
- **「分かりやすさ」のための単純化をAIに要求する** — 人間向け説明とAI内部表現を区別する
- **一つの距離関数を全タスクに使う** — 「何を近いとするか」はタスクごとに違う

- [ ] 扱う対象が住む空間を宣言したか

- [ ] その変換群に対する不変量を列挙したか
- [ ] 人間への提示層を思考層から分離したか
- [ ] 複数表現を同時に保持できる構造になっているか
- [ ] AIエージェントへの指示が空間選択の自由を与えているか

## 9. 適用プロセス

具体的なプロジェクトに適用するときは、以下の順で考える:

1. **対象の同定**: 何を表現したいか(物体・シーン・関係・プロセス)
2. **自然な空間の選択**: その対象が住むのに無理のない空間はどれか
3. **対称性の特定**: その対象が持つ不変性は何か
4. **複数表現の設計**: 補完的な視点を与える別の空間はあるか
5. **翻訳層の設計**: 人間に提示するときの射影方法
6. **AIへの委任設計**: どこを AI の自由判断に任せ、どこを固定するか

## 10. このドキュメントの位置付けと次のステップ

本書(B)は実務指針である。日々の開発で繰り返し参照され、コードレビューや AI エージェントへの指示の基準として機能することを意図している。

一方、本書の根拠となる思想(A)は、次世代AIの発想法・思考様式そのものを設計対象として扱う。これは個別プロジェクトの指針を超えた、より大きな構想に属する。

**シンギュラリティを目標として明示的に掲げるなら、A を要件定義として正式に立項することを推奨する。** その際の起点として `SINGULARITY_REQUIREMENTS.md` を併せて提供する。当面 B 単独で開発を進め、A の立項タイミングは別途判断する、という二段運用が現実的である。
